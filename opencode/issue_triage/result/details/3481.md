## Root Cause

test_dataloader_xpu.py multiprocessing tests (forkserver/spawn) fail when iterating a DataLoader of
XPU sparse/nested tensors because torch/multiprocessing/reductions.py:615 calls
storage._share_fd_cpu_() unconditionally, which dispatches into torch/storage.py:526-527 and raises
'RuntimeError: _share_fd_: only available on CPU' for an XPU storage. The XPU device backend has no
_share_fd_ implementation and the multiprocessing reduction path lacks an xpu-aware branch (only
CPU/CUDA share-fd vs share-handle paths exist). The test at
third_party/torch-xpu-ops/test/xpu/test_dataloader_xpu.py:223 simply enumerates the loader and trips
the reduction.

## Fix Approach

- Fix in pytorch upstream: in torch/multiprocessing/reductions.py reduce_storage(), add an XPU branch that either (a) moves the storage to CPU before fd-sharing for cross-process transport, or (b) implements a true XPU IPC handle path analogous to CUDA's _share_cuda_.
- Short term, skip the spawn/forkserver sparse/nested multiprocessing cases on XPU via skip_list_common.py and document the limitation.
- Longer term, implement Storage._share_xpu_/_new_shared_xpu_ in c10/xpu and register it in the XPU StorageImpl.
