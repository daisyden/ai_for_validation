## Root Cause

DataLoader sparse-tensor multiprocessing tests (spawn/forkserver) fail when the worker pickles an
XPU storage: torch/multiprocessing/reductions.py:615 unconditionally calls storage._share_fd_cpu_(),
which raises 'RuntimeError: _share_fd_: only available on CPU' (torch/storage.py:526-527). XPU
storages have no shared-memory/share_fd reducer registered analogous to CUDA's rebuild_cuda_tensor
path, so any spawn/forkserver DataLoader that sends an XPU-backed (sparse) tensor across processes
fails. The defect is in pytorch/pytorch multiprocessing reductions, not in a SYCL kernel.

## Fix Approach

- Upstream pytorch fix: register an XPU reducer in torch/multiprocessing/reductions.py (analogous to reduce_tensor's CUDA branch using IPC handles) so XPU storages serialize via xpu IPC instead of CPU share_fd.
- gate sparse_tensor multiprocessing tests behind that capability.
- Until landed, keep these cases in the torch-xpu-ops skip list and (optionally) override the test to force the 'fork' context where storages are shared in-memory.
