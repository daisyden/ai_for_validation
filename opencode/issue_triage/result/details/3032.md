## Root Cause

The TorchAO UT test/prototype/safetensors/test_safetensors_support.py is hard-coded for CUDA:
configs 5/6 require the 'mslk' package (CUDA-only), configs 7/9 explicitly assert 'Torch not
compiled with CUDA enabled' / 'Invalid scaling configuration', and configs 10 require SM100+ (NVFP4
DYNAMIC). None of them are gated by skip_if_no_cuda / device checks, so when torchao is built
against a torch+xpu wheel they surface as failures in the XPU CI. There is no corresponding XPU
backend for these quantization configs.

## Fix Approach

- Land the proposed skip PR pytorch/ao#4049 that parametrizes/skips the unsupported configs when CUDA (or required SM/library) is unavailable, e.g. @unittest.skipIf(not torch.cuda.is_available() or get_sm() < 100 or not has_mslk, ...).
- In torch-xpu-ops, pin/skip the test file for XPU CI until the upstream ao change is merged.
- Longer-term, add explicit device gating in ao's test helpers and re-enable once XPU supports the corresponding NVFP4/MXFP8 quant paths.
