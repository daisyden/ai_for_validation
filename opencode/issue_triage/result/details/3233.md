## Root Cause

torch.distributed.split_group in torch/distributed/distributed_c10d.py:5478-5483 rejects any backend
whose parent_backend.supports_splitting is False. ProcessGroupXCCL
(third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.{hpp,cpp}) never overrides supportsSplitting()
/ setSupportsSplitting(true) (no occurrences found in the XCCL sources), so it inherits the base
default (false). Consequently init_device_mesh -> _init_one_process_group -> split_group raises 'No
backend for the parent process group or its backend does not support splitting' for every
XCCL-backed DeviceMesh that uses eager subgroup initialization.

## Fix Approach

- Implement comm-splitting in ProcessGroupXCCL: add a splitGroup()/merged config override, expose supports_splitting=true, and forward comm_split_count().
- Mirror the NCCL implementation (ncclCommSplit) using the oneCCL equivalent (ccl::split / ccl::create_communicator with sub-ranks).
- Until then, skip the eager_init subgroup tests on XPU.
