## Root Cause

`torch.linalg.ldl_solve` segfaults on XPU when given out-of-range pivot values (e.g.,
pivots=[2,3,5,7,11] for N=5). The op is not registered for XPU in torch-xpu-ops, so it falls back to
CPU (UserWarning emitted at RegisterXPU_0.cpp:54882) and the CPU LAPACK kernel `apply_ldl_solve` at
aten/src/ATen/native/BatchLinearAlgebraKernel.cpp:892 dereferences memory using the bogus pivot
indices. The TORCH_META_FUNC at aten/src/ATen/native/BatchLinearAlgebra.cpp:511-549 only checks
shapes/dtypes but never validates `1 <= |pivot| <= N`, which is exactly the gap
pytorch/pytorch#181032 (commit be51081) closes upstream. Pure upstream-pytorch validation bug
surfaced through the XPU CPU-fallback path.

## Fix Approach

- 1) Cherry-pick / track pytorch/pytorch#181032 (adds `1 <= |pivot| <= N` validation in the meta function) and confirm it lands in the torch-xpu-ops-tracked PyTorch base. 2) Once the meta check is in, the XPU CPU-fallback will raise a clean RuntimeError instead of segfaulting.
- add an XPU regression test mirroring the reproducer in `third_party/torch-xpu-ops/test/xpu/test_linalg_xpu.py` to prevent recurrence. 3) Optionally register a native XPU `ldl_solve` via oneMKL LAPACK to remove the CPU fallback (currently flagged as a feature gap).
