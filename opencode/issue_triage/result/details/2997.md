## Root Cause

test_linear_and_cel_max_autotune compiles a Linear+CrossEntropyLoss bf16 graph with max_autotune;
the compiled output produces NaN in the loss and gradients while eager produces finite values. The
NaNs originate in the bf16 matmul autotune path on XPU — likely the oneDNN bf16 GEMM template chosen
by max_autotune accumulates without proper fp32 accumulator, or the inplace-padding optimization
(test_inplace_padding.py) corrupts a bf16 weight/grad buffer on XPU. The selected algorithm in
torch/_inductor/kernel/mm.py max_autotune choice for XPU bf16 returns NaN-producing kernel.

## Fix Approach

- Bisect: disable inplace-padding (config.inplace_padding=False) and disable oneDNN-backed bf16 GEMM autotune candidate to localize.
- If oneDNN: ensure fp32 accumulator in the bf16 brgemm template under torch-xpu-ops oneDNN integration (xpu_aten/native/mkldnn) or in PyTorch's torch/_inductor/kernel/mm.py XPU template.
- If inplace-padding: fix bf16 buffer aliasing in torch/_inductor/fx_passes/post_grad.py inplace-padding pass.
- Add to inductor skip list pending fix.
- cross-link upstream PR #162370.
