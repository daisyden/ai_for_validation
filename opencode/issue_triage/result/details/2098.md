## Root Cause

torch-xpu-ops runs torchgen out-of-tree (third_party/torch-xpu-ops/cmake/Codegen.cmake:6) to
generate XPUFunctions.h / RegisterXPU.cpp, then its install_xpu_headers.py
(tools/codegen/install_xpu_headers.py:92) post-hoc patches the in-tree aten generated headers by
injecting XPU function declarations (also cmake/Codegen.cmake:40-44). This works because XPU ops are
not in the upstream aten/src/ATen/native/native_functions.yaml, so the standard in-tree codegen
never emits xpu:: overloads. The patching is fragile and breaks whenever upstream header layout or
torchgen changes.

## Fix Approach

- Upstream XPU dispatch entries into aten/src/ATen/native/native_functions.yaml (add 'XPU: <kernel>' to each op currently registered out-of-tree in torch-xpu-ops/yaml/native/native_functions.yaml), so in-tree torchgen produces the XPU declarations directly.
- Then delete install_xpu_headers.py and the header-rewrite logic in cmake/Codegen.cmake:40-44 and have torch-xpu-ops consume the upstream-generated headers unchanged.
- Coordinate staged landing op-by-op to keep the tree buildable.
