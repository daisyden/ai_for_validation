## Root Cause

Three FX profiler tests
(test_profiler_stack_trace_augmentation/multiple_modules/nested_graph_modules in
test/xpu/test_fx_xpu.py) compare profiler trace output to upstream baselines that assume one event
per kernel ('cudaLaunchKernel'). On XPU, Kineto/PTI emits many low-level Level-Zero /
Unified-Runtime events per aten op (zeKernelSetArgumentValue, urEnqueueKernelLaunch,
zeCommandListAppendLaunchKernel, zeMemGetAllocProperties, etc.), so assertExpectedInline never
matches regardless of which event substitutes for cudaLaunchKernel. Root cause is profiler trace
shape parity: torch's XPU profiler does not aggregate L0/UR events into a single 'kernel launch'
record like CUDA does.

## Fix Approach

- Preferred: filter the XPU trace in the test helper to keep only the launch-equivalent event (zeCommandListAppendLaunchKernel or urEnqueueKernelLaunch) and provide an XPU-specific expected baseline.
- Longer-term: aggregate L0/UR events at the Kineto/profiler layer in pytorch/pytorch (torch/csrc/profiler/...) so XPU traces have CUDA-parity granularity.
- Until either lands, skip the three tests on XPU.
