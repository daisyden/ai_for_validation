## Root Cause

hf_GPT2_large training under inductor with --amp (bf16/fp16) produces NaN on XPU starting at pytorch
commit 1a270b4aa73b7169a76dd3aa09c52a04d1312c13. Owner weishi-deng root-caused it: the new dtype
guard in that PR blocks unfuse_bias_add_to_pointwise on XPU under AMP, so bf16 addmm forces a
_to_copy of the fp32 bias to bf16 before GEMM, introducing an extra bf16 truncation per linear
layer. The fp32-bias-in-Triton-registers path that previously avoided truncation is no longer taken.
The check lives in torch/_inductor/fx_passes/post_grad.py (should_prefer_unfused_addmm /
unfuse_bias_add_to_pointwise). CUDA hit this earlier via different graph-partition reasons.

## Fix Approach

- Submit an upstream PR (pytorch/pytorch) relaxing the dtype guard in unfuse_bias_add_to_pointwise / should_prefer_unfused_addmm so that under AMP on XPU the unfused bias-add path is preserved (i.e., do not gate it on the dtype-equality check that the offending commit added).
- Alternatively, gate the new check by device type so XPU keeps prior behavior.
- Validate with `python benchmarks/dynamo/torchbench.py --accuracy --training --amp --amp-dtype bfloat16 --only hf_GPT2_large -d xpu` on PVC.
