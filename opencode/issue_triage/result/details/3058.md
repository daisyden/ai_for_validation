## Root Cause

hf_GPT2_large AMP bf16/fp16 training produces NaN (fail_accuracy) after a recent Inductor change.
Root cause identified in comments: the new dtype guard inside unfuse_bias_add_to_pointwise at
torch/_inductor/fx_passes/post_grad.py:1569-1574 (config.keep_addmm_fused_for_half_dtypes for
bf16/fp16) prevents the unfuse of addmm on XPU under AMP, forcing a bf16 addmm which down-casts the
fp32 bias before the GEMM. The extra per-layer bf16 truncation accumulates into NaN, whereas the
previous unfused path let Triton keep the bias-add in fp32 registers. CUDA does not hit the same
regression because its graph (non-pointwise reshape chain) makes should_prefer_unfused_addmm False
even before the PR.

## Fix Approach

- In torch/_inductor/fx_passes/post_grad.py:1570, make the keep_addmm_fused_for_half_dtypes guard device-aware (skip it for XPU) or gate it on a config flag that XPU can opt out of, so XPU continues to unfuse addmm+bias under AMP and the bias-add remains in fp32 inside the Triton kernel.
- Alternatively have XPU Inductor set config.keep_addmm_fused_for_half_dtypes=False.
- Add an E2E regression test for hf_GPT2_large amp_bf16/amp_fp16 training accuracy.
