- **GitHub**: https://github.com/intel/torch-xpu-ops/issues/2436
- **Category**: Torch Operations
- **Priority**: P3
- **Status**: open
- **Assignee**: daisyden
- **owner_transferred**: daisyden
- **Reporter**: daisyden

- **Dependency**: CPU fallback
- **action_Type**: ROOT_CAUSE

## action_TBD

- Owner @daisyden to re-check case design (per @CuiYifeng request)

## action_reason

- No fix PR; behavior matches CUDA per upstream pytorch/pytorch#97395 (CPU issue).

## Root Cause

TestExpandedWeightModuleXPU cases in third_party/torch-xpu-ops/test/xpu/test_expanded_weights_xpu.py
hit 'NoneType' has no attribute 'clone' inside torch.nn.utils._expanded_weights when a parameter's
.grad is None at the time the ExpandedWeight hook tries to clone it. Known upstream CPU bug
pytorch/pytorch#97395: grad may be None depending on whether a prior test populated/zeroed it, so
failures are order-dependent and random (confirmed: passes when run alone in nightly). Not
XPU-specific.

## Fix Approach

- Wait for the upstream fix in pytorch/pytorch#97395 (guard None .grad in _expanded_weights clone path).
- In the meantime redesign the XPU test to explicitly reset/populate grads per sub-test (zero_grad() + forward/backward before the assertion) as requested by CuiYifeng, and keep the skip entry in skip_list_common.py labelled 'random' until upstream lands.
