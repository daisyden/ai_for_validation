## Root Cause

test/xpu/test_unary_ufuncs_xpu.py::test_reference_numerics_{extremal,large,normal}_* produces
mismatches at extreme/large complex inputs for
acos/acosh/asin/asinh/log/log10/log1p/tan/tanh/tanhshrink (complex64/complex128/complex32), plus
bfloat16 round_decimals_3, float16 polygamma_n_{1..4}, and int8 nonzero_large. The complex unary
kernels in third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/UnaryComplexKernels.cpp /
UnaryLogKernels.cpp / UnaryGeometricAcosKernels.cpp etc. rely on SYCL std::complex math which loses
precision at NaN/inf/large magnitudes compared with CUDA's hand-rolled complex implementations in
c10/util/complex_math.h. Secondary: round_decimals bfloat16 uses float32 std::round causing ULP
drift, polygamma float16 upcast path differs, and nonzero_large int8 exceeds an int32 index
accumulator.

## Fix Approach

- Port CUDA's explicit complex formulas (from aten/src/ATen/native/cuda/UnaryComplexKernels.cu and c10/util/complex_math.h) into the XPU sycl complex kernels so extremal branch cuts/NaN handling match, widen bfloat16 round_decimals to double before std::round in UnaryKernels, cast polygamma to float32 for float16 like CUDA, and use int64 indexing in NonzeroKernel.
- Where numerics match CUDA reference but not the libc++ complex model, update xfail/tolerance entries in test/xpu/skip_list_common.py and test_unary_ufuncs_xpu.py.
- Track by tag bug_fix_stage4.
