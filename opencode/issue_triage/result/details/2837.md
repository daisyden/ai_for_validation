## Root Cause

test_state_dict_with_cuda_params_Muon_xpu_float32 fails with only 1/6 element exceeding tolerance
(abs 1.94e-5 vs allowed 1e-5). Muon's update uses a Newton-Schulz iteration that internally casts to
bfloat16 (torch/optim/_muon.py:55 `ortho_grad = grad.bfloat16()`) and runs repeated bf16 matmuls
(torch.addmm at lines 63-66). XPU bf16 matmul rounding differs slightly from CUDA's, so the final
fp32 result drifts just past the default tolerance when cross-checked against a CUDA-loaded
state_dict. This is a tolerance/precision artifact, not a functional bug.

## Fix Approach

- Loosen tolerances for this test on XPU via the toleranceOverride / tf32_on_and_off mechanism in test/test_optim.py (or add an xpu-specific override in the Muon OptimizerInfo sample input), setting atol/rtol to ~5e-5 for float32 to account for bf16 intermediate rounding differences.
- Alternatively, add the test to the xpu skip list (already tagged skipped/bug_fix_stage5) once the tolerance PR lands upstream.
