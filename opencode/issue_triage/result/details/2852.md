## Root Cause

Windows nightly 0206 regresses the entire torch.fft.* suite (all dtypes including bool non-standard
values, fft/ifft/rfft/hfft and their n/2d variants) plus test_mem_get_info. Regression window is
PyTorch 988b354..9b907f1 and torch-xpu-ops 5da2326..0d3b388. The SpectralOps path in torch-xpu-ops
src/ATen/native/xpu/SpectralOps.cpp goes through oneMKL DFT descriptors; on Windows oneMKL 2025.2
the CPU-XPU result comparison now diverges, pointing to a Windows-only oneMKL DFT regression.

## Fix Approach

- Bisect the torch-xpu-ops range (SpectralOps.cpp changes) and the PyTorch range (test_ops extended harness) to locate the breaking commit.
- check oneMKL DFT version pulled on Windows vs Linux.
- If oneMKL is the culprit, pin to the last known-good oneMKL build or file a ticket.
- Re-enable cases in test/xpu/skip_list_win.py once resolved.
- also triage test_mem_get_info separately (likely an XPUCachingAllocator.cpp API change).
