## Root Cause

torch._scaled_mm_v2 on XPU rejects DeepSeek-style block-wise scales early in argument validation
with ValueError('Invalid scaling configuration'), while the upstream test expects
NotImplementedError('DeepSeek-style scaling is only supported in CUDA for SM90'). The XPU port skips
via _get_torch_cuda_version()<(12,9) which trivially returns (0,0) on XPU — a misleading skip. The
validation message lives in pytorch upstream (aten/src/ATen/native/cuda/Blas.cpp / scaled_mm
dispatch) and is shared with XPU via _scaled_mm_v2.

## Fix Approach

- Preferred: in upstream torch._scaled_mm_v2 (or the XPU dispatch path), detect blockwise scales on XPU and raise NotImplementedError with a clear 'DeepSeek-style scaling not supported on XPU' message before the generic ValueError.
- Then generalize the test to assertRaisesRegex(NotImplementedError, ...) device-agnostically.
- Alternative: replace the stale CUDA-version skip in test_scaled_matmul_cuda_xpu.py with an explicit @skipXPU reason.
