## Root Cause

On BMG (Arc B580), Llama-3.2-1B dynamic INT8 next-token latency regressed ~20% starting from
2026-01-22 nightly. Regression was bisected to pytorch/pytorch#166698 which converts SDPA scalar
arguments to host scalars. For the SDPA shapes used in llama3.2-1B on BMG, oneDNN SDPA falls into
the reference (non-ukernel) path; the reference path introduces an extra H2D memcpy/scalar sync per
call since the host-scalar change, while PVC's ukernel path is unaffected. Tracked in MFDNN-14745.

## Fix Approach

- Wait for oneDNN fix of MFDNN-14745 (avoid redundant H2D sync in reference SDPA path).
- In the meantime, in the XPU SDPA wrapper (third_party/torch-xpu-ops/src/ATen/native/xpu/Attention.cpp or similar), route small/BMG shapes away from the reference path, or revert/gate the host-scalar conversion for the XPU SDPA call site so scale/dropout remain device scalars until oneDNN adds a ukernel for the BMG shape.
- After oneDNN version bump, re-benchmark Llama-3.2-1B on BMG to confirm recovery.
