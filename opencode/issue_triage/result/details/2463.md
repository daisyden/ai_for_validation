## Root Cause

test_xpu.TestXpu.test_xpu_pluggable_allocator uses torch.utils.cpp_extension.load_inline to
JIT-build a dummy allocator on Windows. Two failure modes surface: (1) SYCL_HOME is not detected so
_join_sycl_home() at torch/utils/cpp_extension.py:199-212 raises 'SYCL runtime is not dected.' when
called from _prepare_ldflags (line 2524); (2) even when SYCL_HOME is found, the build uses MSVC
cl.exe which cannot compile <sycl/sycl.hpp> (DPCPP static_assert 'DPCPP does not support C++ version
earlier than C++17' and STL4038 <span> warnings from sycl/vector.hpp). The helper SYCL_HOME =
_find_sycl_home() at line 260 returns None in the CI conda env where intel-sycl-rt is pip-installed
under site-packages rather than under the Intel oneAPI install root it searches for. A sibling test
test_set_per_process_memory_fraction also OOMs on Arc B580 (16 GiB) with an 8 GiB alloc.

## Fix Approach

- In torch/utils/cpp_extension.py: extend _find_sycl_home() to also detect the pip-installed intel-sycl-rt layout (site-packages headers/libs) and to honor the CMPLR_ROOT/ONEAPI_ROOT envs set by setvars.bat.
- For the Windows MSVC path, force load_inline to use icx/dpcpp when compiling SYCL sources (detect via TORCH_EXTENSION_SYCL or presence of sycl headers) and pass /EHsc -fsycl /std:c++17.
- otherwise raise a clearer error up front.
- For the OOM case, reduce application_memory in test_set_per_process_memory_fraction to fit on <=12 GiB cards or xfail on BMG.
- Keep the current skipped_windows skip until the Windows JIT SYCL toolchain is wired up.
