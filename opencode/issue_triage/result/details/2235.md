## Root Cause

test_triton_bsr_dense_addmm_meta_xpu in test/test_sparse_csr.py:4308 calls
get_meta_with_checks(32,32,48, warn_count=0) and asserts no 'non-optimal triton kernel parameters'
UserWarning is emitted, but torch/sparse/_triton_ops.py:821 emits the warning because the XPU-tuned
meta table doesn't contain optimal Triton block params for this shape. The bsr_dense_addmm meta DB
is CUDA-tuned; XPU shapes fall back to default (Ms=16,Ks=16) which is flagged non-optimal.

## Fix Approach

- Populate the Triton bsr_dense_addmm tuning table in torch/sparse/_triton_ops_meta.py with XPU-specific optimal meta entries (run the autotuner offline on XPU for the test shapes), or relax the test's warn_count expectation for XPU until the meta DB is tuned.
