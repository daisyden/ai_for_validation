## Root Cause

test_triton_bsr_dense_addmm_meta asserts that for sizes registered in _operation_device_version_data
no 'non-optimal triton kernel parameters' warning is emitted. On XPU get_meta()
(torch/sparse/_triton_ops_meta.py:114) looks up the key with device_name from
torch.cuda.get_device_name() and the table only contains NVIDIA A100/H100 entries, so no meta
matches; bsr_dense_addmm in torch/sparse/_triton_ops.py:821 then issues the 'bsr_dense_addmm uses
non-optimal triton kernel parameters for M=32 K=32 N=48' UserWarning which the test CHECK-NOTs. Same
root cause as issue 2246.

## Fix Approach

- Resolve together with #2246: make get_meta() query the current accelerator and register XPU tuning results in _operation_device_version_data for the sizes exercised by test_triton_bsr_dense_addmm_meta (M/K/N = 32/32/48 etc.).
- Once XPU-specific meta is present for those shapes, the warning disappears and the assertExpected passes.
- Until then, keep the test skipped in the XPU skip list.
