## Root Cause

test_triton_sampled_addmm_* fails with 'sampled_addmm(): all inputs are expected to be on the same
GPU device'. The triton-backed sampled_addmm path checks all tensors share the same device, but the
test/kernel constructs an intermediate (e.g., values/indices) on a default device while inputs are
on xpu:N, triggering the device-mismatch ValueError. The triton XPU kernel dispatch in the sparse
compressed triton kernels module does not propagate the input device when materializing scratch
tensors.

## Fix Approach

- In the sparse-compressed triton sampled_addmm helper (test_sparse_csr.py / torch._inductor sparse triton path) ensure all intermediates are constructed with device=input.device.
- alternatively xfail/skip until the triton-xpu sampled_addmm path is plumbed.
- Confirm via single-device run (XPU_VISIBLE_DEVICES=0).
