## Root Cause

AOT Inductor test `test_custom_op_square_xpu` fails during g++ build of the wrapper with `error:
'aoti_torch_xpu_fn_square' was not declared in this scope; did you mean 'aoti_torch_cpu_fn_square'`.
The generated wrapper.cpp (from pytorch inductor codegen) emits `aoti_torch_xpu_fn_<op>(...)` for
user-registered custom ops on XPU, but no XPU shim is generated/registered on the C++ side. On CPU
the shim `aoti_torch_cpu_fn_square` is auto-generated by
torch/csrc/inductor/aoti_torch/generated/*.cpp, while the XPU codegen in
torch/_inductor/codegen/cpp_wrapper_gpu.py (or aoti_runtime/utils.h shim generation for custom ops)
doesn't emit an equivalent `aoti_torch_xpu_fn_*` shim, so the reference is unresolved at link time.

## Fix Approach

- In pytorch upstream torch/_inductor/codegen (see cpp_wrapper_gpu.py / aoti shim generation for custom ops) and torch/csrc/inductor/aoti_torch/, generate the `aoti_torch_xpu_fn_<op>` shim for user-registered XPU custom ops analogous to the CPU path (see torch/csrc/inductor/aoti_torch/generated/c_shim_cpu.cpp mechanism).
- Alternatively, codegen can fall through to `aoti_torch_cpu_fn_<op>` when the custom op dispatches to CPU/XPU via the ProxyExecutor path.
- File a pytorch PR adding XPU custom-op shim generation.
- in the meantime keep this test skipped.
- No torch-xpu-ops kernel change required.
