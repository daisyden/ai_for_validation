## Root Cause

torch._fft_c2r with last_dim_size=536870912 (and 0) on XPU triggers TORCH_INTERNAL_ASSERT at
third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp:242-247 ('in_size ==
signal_size[i+1] || in_size == (signal_size[i+1]/2)+1') instead of a user-facing RuntimeError. The
XPU oneMKL-backed _fft_c2r path is missing the user-input validation that upstream added in
pytorch/pytorch#181060: there is no early TORCH_CHECK on last_dim_size before _exec_fft is called,
so inconsistent sizes flow into the internal invariant check.

## Fix Approach

- Mirror upstream pytorch/pytorch#181060 in third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp: add an upfront TORCH_CHECK at the _fft_c2r entry that validates last_dim_size > 0 and self.size(dim.back()) == last_dim_size/2 + 1 (emitting the canonical 'Expected size of last transformed dimension of input to be ...' / 'Invalid number of data points' messages) before delegating to _exec_fft, so the TORCH_INTERNAL_ASSERT at line 242 becomes unreachable for user inputs.
