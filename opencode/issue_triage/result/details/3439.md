## Root Cause

torch._fft_c2r on XPU with an inconsistent last_dim_size (e.g. 536870912 or 0) hits an INTERNAL
ASSERT in torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp:242-247 ('in_size ==
signal_size[i+1] || in_size == (signal_size[i+1]/2)+1'). The XPU path enters the oneMKL backend
without first user-validating last_dim_size against the input's signal-dim size, so an invalid
argument trips an internal-assert (suggests a bug to user) rather than producing the upstream
RuntimeError 'Expected size of last transformed dimension of input to be ...'/'Invalid number of
data points'. Upstream PR pytorch/pytorch#181060 (commit e13ac6e) added the user-facing TORCH_CHECK
in the generic _fft_c2r path; XPU's MKL specialization needs to mirror it before reaching the
assert.

## Fix Approach

- In src/ATen/native/xpu/mkl/SpectralOps.cpp, before the loop at lines 238-247, add upstream-aligned TORCH_CHECK validations: verify last_dim_size > 0, verify in_size == signal_size[i+1] or (signal_size[i+1]/2)+1 with a user-message ('Expected size of last transformed dimension of input to be ...'), and reject overflowing last_dim_size early.
- Replace the TORCH_INTERNAL_ASSERTs with TORCH_CHECKs (or keep the assert as a post-validation invariant).
- Add UT in test/xpu covering last_dim_size in {0, 536870912, mismatched} and assert RuntimeError with the expected message substrings.
