## Root Cause

Upgrading oneDNN from v3.10.2 to v3.11 introduces ~245 XPU test regressions, virtually all
concentrated on Convolution forward/backward paths (test/nn/test_convolution.py,
test/functorch/test_ops.py conv* grad/vmap/jvp tests, test/test_modules.py
Conv2d/Conv3d/ConvTranspose*, test/test_ops_gradients.py conv gradgrad). LuFinch narrowed the
failure to the ConvBackwardWeight primitive, but benchdnn cannot reproduce standalone, indicating
the regression is in how torch-xpu-ops invokes the backward-weight primitive in
third_party/torch-xpu-ops/src/ATen/native/xpu/mkldnn/Conv.cpp (primitive descriptor / memory
descriptor setup for depthwise and grouped conv variants). The internal tracker MFDNN-14584 is open
against the oneDNN team.

## Fix Approach

- Block the oneDNN 3.11 uplift until MFDNN-14584 is resolved.
- as mengfei25 requested, retest with oneDNN 3.11.1 to confirm whether the primitive bug is already fixed.
- In parallel, dump the exact primitive descriptors (src_md, weights_md, diff_dst_md, attr) from Conv.cpp for a failing depthwise/grouped case and hand that to the oneDNN team for a reduced benchdnn repro.
- if still not fixed upstream, apply a workaround in Conv.cpp to force plain (NCHW/OIHW) memory format for ConvBackwardWeight of the failing configurations.
