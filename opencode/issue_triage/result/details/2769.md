## Root Cause

Bumping oneDNN from v3.10.2 to v3.11 introduced 245 new failures, all clustered around
Conv2d/Conv3d/ConvTranspose forward/backward and depthwise variants (test_convolution.py,
test_ops.py, functorch). LuFinch's comment in the issue confirms the failures are conv-related. Conv
on XPU is dispatched through oneDNN (see torch-xpu-ops oneDNN integration in
src/ATen/native/onednn/), so the regression is in the oneDNN v3.11 conv primitive
selection/numerical behavior.

## Fix Approach

- File a oneDNN upstream issue with a minimal conv repro.
- pin oneDNN to v3.10.2 in CI until fixed, or adapt torch-xpu-ops oneDNN conv dispatch (src/ATen/native/onednn/Conv.cpp) to work around the regression (e.g., disable specific algorithm selections that changed in 3.11).
