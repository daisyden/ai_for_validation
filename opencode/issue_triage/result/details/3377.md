## Root Cause

test_abort_in_destroy_pg / test_xccl_dist_backend_error fail with 'DistBackendError not raised'
because ProcessGroupXCCL does not surface a DistBackendError when a collective is aborted/torn down
— the abort() override is unimplemented (ProcessGroupXCCL.hpp:107) and the watchdog/exception
propagation seen in ProcessGroupNCCL is missing in XCCL. Likewise test_individual_send_recv
(timing_enabled) fails with state 'started' != 'completed' because XCCL flight-recorder/work state
transitions are not driven to 'completed' on send/recv timing path. Both are XCCL feature parity
gaps with NCCL.

## Fix Approach

- In ProcessGroupXCCL: (1) implement abort() to mark outstanding works as exception/aborted with a c10::DistBackendError so assertRaises in destroy_pg passes.
- (2) wire send/recv work state machine to update entries to 'completed' upon CCL completion when timing_enabled — likely in WorkXCCL::synchronizeInternal / FlightRecorder update sites — analogous to ProcessGroupNCCL's updateWorkflowState.
- Add unit-level coverage in torch-xpu-ops xccl tests.
