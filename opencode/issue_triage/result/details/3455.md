## Root Cause

Huggingface/Timm/Torchbench all fail-to-run with `AssertionError: Torch not compiled with CUDA
enabled` after upstream commit 620ac21ae34 (pytorch/pytorch#181038, "Enable PadMM AutoHeuristics by
default"). That PR changed `autoheuristic_use.pad_mm` default semantics in
`torch/_inductor/config.py` so that `use_autoheuristic('pad_mm')` falls through to the
`deterministic` branch and reaches `AutoHeuristic.__init__` at
`torch/_inductor/autoheuristic/autoheuristic.py:88`, which unconditionally calls
`torch.cuda.get_device_capability()`. On XPU/CPU-only builds that triggers the CUDA-not-enabled
assertion through the pad_mm pattern matcher in `torch/_inductor/fx_passes/pad_mm.py:766`. Pure
upstream-pytorch bug; chuanqi129 already noted the fix has landed.

## Fix Approach

- 1) Confirm the upstream fix landed (guard `torch.cuda.get_device_capability()` with `torch.cuda.is_available()` in `autoheuristic.py:88` and/or early-return non-CUDA in `pad_mm.run_autoheuristic`) is in the next nightly and re-run benchmarks to verify (kaileiyx is on it). 2) For users on affected nightlies, recommend workaround `TORCHINDUCTOR_SHAPE_PADDING=0` or `TORCHINDUCTOR_AUTOHEURISTIC_USE=`. 3) Add a torch-xpu-ops CI smoke check that runs an inductor compile on XPU to catch any future inductor pass that hard-depends on CUDA APIs.
