## Root Cause

Inductor accuracy regression for pyhpc_turbulent_kinetic_energy fp32 inference (RMSE 12.15 vs ref
0.61). Root cause identified by jianyizh: pytorch/pytorch PR #171773 switched Inductor Triton
codegen from libdevice.sqrt to tl.sqrt_rn (round-to-nearest). The IGC compiler mis-optimizes
tl.sqrt_rn on Max 1100: setting -cl-opt-disable in triton makes the test pass, confirming it is an
IGC/driver issue triggered by the sqrt_rn lowering (linked to intel-xpu-backend-for-triton#6261).

## Fix Approach

- Short term: in torch/_inductor/codegen/triton.py (or the xpu override), force libdevice.sqrt (or an fp32-safe fallback) for the XPU backend until the IGC fix ships.
- alternatively pass -cl-opt-disable as a temporary kernel-build flag on the XPU Triton path.
- Long term: wait for the IGC/driver fix tracked in intel-xpu-backend-for-triton#6261, then re-enable tl.sqrt_rn.
- Add the model to the XPU accuracy CI allowlist once unblocked.
