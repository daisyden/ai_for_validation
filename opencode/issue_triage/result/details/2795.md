## Root Cause

src/ATen/native/xpu/SummaryOps.cpp:38-47 _histc_xpu unconditionally calls
globalContext().alertNotDeterministic("_histc_xpu") for every dtype because the underlying
_histc_kernel uses atomicAdd. CUDA guards this alert behind isFloatingType(self.scalar_type()) so
integer inputs remain allowed under torch.use_deterministic_algorithms(True); XPU does not, which
breaks transformers MoE parity between CUDA and XPU when deterministic mode is on.

## Fix Approach

- Mirror the CUDA gating in _histc_xpu: only call alertNotDeterministic when isFloatingType(self.scalar_type()) (or when floating-point weights/atomicAdd are actually used).
- For integer tensors the atomicAdd on integer bins is already bitwise-deterministic, so the guard can simply be removed.
- Add/enable the corresponding deterministic-histc int tests in test/xpu to prevent regression.
