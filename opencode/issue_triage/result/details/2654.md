## Root Cause

T5/T5Small fp16 bs=32 on BMG regressed 292ms -> 435ms because the Inductor Triton heuristics choose
XBLOCK=8, num_warps=8 for a persistent_reduction softmax kernel, which triggers register spill
(25216) and large-GRF mode. Setting XBLOCK=1 drops the same kernel to 7.4ms. The bad heuristic lives
in torch/_inductor/runtime/triton_heuristics.py around the persistent_reduction config selection
(see `_persistent_reduction_configs` / line ~3424 referenced in the bug). No torch-xpu-ops SYCL
kernel is involved.

## Fix Approach

- Tune the XPU branch of persistent_reduction heuristics in torch/_inductor/runtime/triton_heuristics.py so that for reduction kernels where r0_numel is already full-tile (e.g.
- R0_BLOCK == rnumel) XBLOCK is capped at 1 on Intel GPUs, matching the measured optimum.
- Author noted T5 is now masked by SDPA fusion.
- re-measure T5/T5Small in latest OOB and close if gap < 5%, otherwise land the heuristics patch upstream.
