## Root Cause

In test_tp_examples.py the TP transformer forward invokes F.scaled_dot_product_attention; since XPU
only supports SDPBackend.MATH for training, the decomposed math path executes aten.add on the
attention-mask plus q@k^T, where mask is a plain torch.Tensor but q@k^T is a DTensor, producing the
mixed-tensor error (same as CUDA when forced to MATH). Root cause is the absence of a device-side
fused attention kernel on XPU that would keep the op graph as a single DTensor-aware call. Fix
depends on oneDNN fused attention landing (referenced in issues #1480 and pytorch/pytorch#161161).

## Fix Approach

- 1) Integrate the oneDNN fused attention kernel for XPU so SDPA does not fall back to the MATH decomposition under DTensor. 2) Until then, either register a sharding rule that broadcasts the mask to DTensor in the MATH path, or mark these TP examples as expected-fail in skip_list_dist*.py. 3) Track dependency on PT PR #161161 as noted by dayanandav.
