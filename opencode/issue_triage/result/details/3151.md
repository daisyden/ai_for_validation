## Root Cause

fbnetv3_b / rexnet_100 / sebotnet33ts_256 timm training accuracy regresses after updating triton-xpu
from 64bb0de3 to 3.7.1+git21033c4e (BMG hardware, AMP fp16/bf16). Failures are on batchnorm-adjacent
tensors (bn1.weight.grad, running_var, conv weight.grad) with RMSE exceeding the allowed multiplier.
No torch-xpu-ops kernel changed between the passing/failing configs; only triton changed. Root cause
lies in a newly-generated Triton kernel (likely the fused reduction/normalization or backward conv
epilogue) that loses precision under low-precision accumulation on BMG.

## Fix Approach

- Bisect triton-xpu between 64bb0de3 and 21033c4e to identify the offending commit and file a triton-xpu upstream fix.
- Short term: pin triton-xpu commit in .ci/docker/ci_commit_pins/triton-xpu.txt to the last good SHA, and/or force fp32 accumulation for BN reductions via inductor config (e.g., torch._inductor.config.triton.force_higher_precision_acc) for these models.
- Validate by rerunning the three benchmarks/dynamo/timm_models.py commands.
