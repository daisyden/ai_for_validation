Root cause and fix are in the Triton component, not torch-xpu-ops, so this issue is out-of-scope for torch-xpu-ops code changes.

## Root Cause

Timm_models rexnet_100 / fbnetv3_b / sebotnet33ts_256 fail accuracy in training+AMP on BMG with
triton-xpu 3.7.1+git21033c4e, while commit 64bb0de3 passes — a Triton ReduceOpToLLVM lowering change
introduced fp16/bf16 reduction precision drift visible in BN running_var and conv weight grads.
Tracked as intel-xpu-backend-for-triton#6489/#6719; mitigated by partial revert in
intel/intel-xpu-backend-for-triton#6497. Verified PASS on PT 2.12 RC + triton-xpu 3.7.1; issue is
CLOSED.

## Fix Approach

- No torch-xpu-ops change needed — pin/upgrade to triton-xpu 3.7.1 (with #6497 revert) bundled in PT 2.12 RC.
- keep tracking #6719 for proper root-cause fix in ReduceOpToLLVM and re-test if reduction lowering is reintroduced.
