## Root Cause

On Windows the extended op_ut 'test_compare_cpu' reference-comparison tests for std, std_mean, var,
var_mean (bfloat16/float64) and sum (complex128) diverge from CPU beyond tolerance, indicating the
XPU reduction kernels (third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ReduceOpsKernels /
ReduceSumProdKernel / ReduceMomentKernel) produce different intermediate accumulation on Windows vs
Linux — likely due to OS-dependent work-group scheduling / different default sub-group size picked
by the Windows Level-Zero driver, combined with low-precision (bf16) or complex (complex128 sum)
accumulation order sensitivity. Linked to sibling issue #2133.

## Fix Approach

- Reproduce on a Windows DGM box with XPU_VERBOSE.
- compare per-workgroup partial sums vs CPU.
- Enforce higher-precision accumulator (float for bf16 variance, double for complex128 sum) in the reduction kernels, matching the CUDA implementation.
- If accumulation order is inherently non-deterministic, relax tolerances only for bf16 with a dtype-specific atol/rtol bump in test_ops_xpu.py and keep float64/complex128 under investigation.
- Coordinate with the #2133 fix.
