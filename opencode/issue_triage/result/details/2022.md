## Root Cause

On Windows, op_extended tests for std/std_mean/var/var_mean (bfloat16/float64) and sum (complex128)
regress against CPU reference. The variance/std reductions live in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ReduceMomentKernels.cpp and sum/prod in
ReduceSumProdKernels.cpp; Welford-style reduction or accumulator dtype on Windows-IGC produces small
numeric drift exceeding tolerance. Tests were previously skipped on Windows; new PyTorch main
re-enabled them, exposing the regression. Related to issue #2133.

## Fix Approach

- Compare Windows vs Linux numerics.
- verify accumulator promotion (use double/float for bfloat16 accumulation) in ReduceMomentKernels.cpp Welford merge and complex sum reduction in ReduceSumProdKernels.cpp.
- if Windows IGC-specific, add tolerance bump or keep skipped_windows entry until Windows IGC is fixed.
