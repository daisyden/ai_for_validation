## Root Cause

torch.distributed.all_to_all_single corrupts small tensors on B60/XCCL while
torch.distributed._functional_collectives.all_to_all_single works (ft_c flattens + internally
dispatches differently). ProcessGroupXCCL::alltoall_base at
third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp:1977 calls xccl::onecclAllToAll (line 2055)
with per-rank send/recv offsets computed from inputSplitSizes via c10d::computeLengthsAndOffsets.
For the equal-split branch (splitSizes empty, line 1985) it still funnels through the v-style
onecclAllToAll rather than a flat onecclAllToAll, and on B60 with CCL_SYCL_ALLTOALL_ARC_LL=1
oneCCL's LL small-message algorithm appears to mis-handle this offset/length layout, yielding
scrambled output for small tensors (large tensors pass because they fall back to the non-LL
algorithm). ft_c works because it flattens and uses the split-less code path at a different call
site.

## Fix Approach

- (1) In alltoall_base (ProcessGroupXCCL.cpp:1977), when both inputSplitSizes and outputSplitSizes are empty, call a plain onecclAllToAll/onecclAllToAllEqual on contiguous input/output without computing per-rank offsets (mirroring NCCL's ncclAllToAll fast path), so the LL algorithm receives the simple layout it expects. (2) Verify the contiguity/stride path in xccl::onecclAllToAll (src/xccl/xccl.cpp) and add a contiguous() guard for non-contiguous inputs. (3) File an upstream oneCCL bug for the LL all_to_all on Battlemage and, until fixed, add a device-based workaround that unsets CCL_SYCL_ALLTOALL_ARC_LL for B60. (4) Add a distributed UT based on the reporter's test_alltoall_single.py.
