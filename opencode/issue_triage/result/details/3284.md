## Root Cause

torch.nn.functional.one_hot triggers two D2H syncs per call on XPU (~390ms aten::_local_scalar_dense
per 1000 calls in vLLM Gemma4-26B). The CPU reference at aten/src/ATen/native/Onehot.cpp:54 and :62
calls self.min().item() and self.max().item() for boundary checks, but the device-skip whitelist
only excludes kCUDA/kMPS/kPrivateUse1/kXLA, not kXPU, so XPU tensors still take the slow validating
path instead of relying on scatter's on-device asserts. This causes 120 D2H copies per step in
Gemma4 prefill, raising TTFT to 6.0s at concurrency=16.

## Fix Approach

- Land pytorch/pytorch#179831 (Silv3S) which adds at::kXPU to the device-type guards on lines 51-52 and 59-60 of Onehot.cpp so XPU follows the same scatter-asserts path as CUDA.
- Verified by Silv3S microbenchmark (494ms->15ms) and yintong-lu Gemma4-26B run (TTFT 6.0s->1.6s, throughput 1285->4700 tok/s).
- Target PT2.12 per xinyu-intel/Silv3S consensus.
