## Root Cause

PyTorch XPU wheels ship pinned SYCL/UR runtime in torch/lib and rely on RPATH to find
libur_loader.so.0. When the user sources a newer/older system oneAPI via oneapi-vars.sh,
LD_LIBRARY_PATH takes precedence over RPATH, so libsycl.so.8 (built against LIBUR_LOADER_0.11) loads
an older /opt/intel/oneapi/2025.2/lib/libur_loader.so.0 that does not export that symbol version,
causing ImportError on 'import torch'. The packaging scripts for the torch XPU wheel do not use
RUNPATH (or --disable-new-dtags) nor wrap the loader, so sourced environments silently override the
pinned UR loader.

## Fix Approach

- In the XPU wheel build/packaging (pytorch setup.py and torch-xpu-ops CMake install rules for libsycl/libur), convert RPATH to RUNPATH only where needed and bundle libur_loader.so.0 inside torch/lib, then relink libsycl.so.8 with an explicit DT_NEEDED + RPATH ($ORIGIN) pointing to the bundled loader.
- Alternatively, prepend torch/lib to the dynamic lookup via a small ctypes preload of the bundled libur_loader in torch/_C init, as CUDA does for NVRTC.
- Coordinate version pin with intel-sycl-rt / intel-cmplr-lib-ur pip metadata so sourcing an older oneAPI does not regress the UR ABI.
