## Root Cause

Of the originally listed 9 cases, 5 already pass in nightly (pow complex, reductions large half,
matmul_mv, pinv). The still-failing cases (test_depthwise_conv_64bit_indexing_xpu,
test_matmul_mv_xpu_float32, test_index_copy_deterministic_xpu,
test_index_put_non_accumulate_deterministic_xpu, test_state_dict_with_cuda_params_Muon_xpu_float32)
share a common root cause per gplutop7's analysis (2026-03-12/16): oneDNN GPU matmul/conv uses
lower-precision intermediate accumulation in the FP32 addmv/matmul path, producing ~7.6e-4 error on
large reductions vs the ~2.7e-4 CPU reference. Tracked in MFDNN-14761. mkldnn.matmul.fp32_precision
toggles do not change the outcome.

## Fix Approach

- Primary fix lives in oneDNN (MFDNN-14761): enable FP64 or chunked FP32 accumulation for the GPU matmul/conv primitives used by addmv and depthwise conv.
- As a stopgap, either (a) relax tolerances for these specific XPU tests via @toleranceOverride, or (b) route the addmv/matmul float32 path in torch-xpu-ops (src/ATen/native/xpu/Blas.cpp) through an FP64 accumulator when deterministic/high-precision is requested, accepting the perf regression.
- The remaining index_copy/index_put/Muon optimizer failures overlap with issue #2280 and should be triaged together.
