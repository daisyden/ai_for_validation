## Root Cause

The MultiThreadedTestCase infrastructure relies on the 'threaded' process-group backend registered
at torch/testing/_internal/distributed/multi_threaded_pg.py:548 via
dist.Backend.register_backend('threaded', _create_threaded_pg, devices=['cpu','cuda']); 'xpu' is not
in the device list, so FSDP/Replicate MultiThread test classes
(TestFullyShardPostAccGradHookMultiThread, TestFullyShardCollectiveOps, TestFSDPMiscMultiThread,
etc.) cannot dispatch collectives to xpu tensors and are skipped/fail.

## Fix Approach

- Add 'xpu' to the devices tuple at multi_threaded_pg.py:548 (register_backend('threaded', _create_threaded_pg, devices=['cpu','cuda','xpu'])).
- Then audit the collective helpers in that file (_all_gather_base/_reduce_scatter_base/_broadcast around lines 147-284) to ensure they preserve xpu device placement and use torch.xpu streams where appropriate.
- Finally, drop the xpu skip entries for the listed TestFullyShard*MultiThread / TestFSDPMiscMultiThread cases.
- No SYCL kernel work is required.
