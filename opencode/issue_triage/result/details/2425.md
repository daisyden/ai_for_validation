## Root Cause

test_nested_tensor_dense_elementwise calls nt.add(t) where nt is a nested XPU tensor and t is a
dense XPU tensor. In aten/src/ATen/native/nested/NestedTensorBinaryOps.cpp:104 the mixed (nested,
dense) fast-path is gated on 'self.is_cuda() && other.is_cuda()', so XPU tensors fall through to the
generic NestedTensor_elementwise_Tensor path which requires both operands to be nested and raises
'Expected both self and other to be nested'. This is the same underlying gap tracked in feature
issue #2412.

## Fix Approach

- In aten/src/ATen/native/nested/NestedTensorBinaryOps.cpp:104 extend the guard to also accept XPU (e.g. `(self.is_cuda()||self.is_xpu()) && (other.is_cuda()||other.is_xpu())`) and route the [B,*,D]x[B,1,D] / [B,*]x[B,1] broadcasted kernel through a dispatcher so a SYCL implementation can be plugged into torch-xpu-ops (new src/ATen/native/nested/xpu/NestedTensorBinaryOps.cpp).
- Short-term, register a CPU fallback for the nested+dense broadcast case on XPU to unblock the UT.
