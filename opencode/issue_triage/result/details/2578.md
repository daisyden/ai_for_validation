## Root Cause

Int4WeightOnlyConfig(version=1) uses `TensorCoreTiledLayout(inner_k_tiles=8)` +
`Int4ChooseQParamsAlgorithm.TINYGEMM`, which is CUDA's tinygemm packed layout. torchao calls
`_convert_weight_to_int4pack` and `_weight_int4pack_mm`; the XPU implementations are at
third_party/torch-xpu-ops/src/ATen/native/xpu/WeightInt4Pack.cpp:17 and LinearInt4.cpp:21 but they
use a different packed format (uint8 int4x2 / int4x8, see sycl/WeightInt4PackKernel.cpp:52 and
sycl/Dequant_int4.cpp) and ignore `innerKTiles`. When torchao passes the TC-tiled packed buffer, the
XPU dequant interprets bytes in a different order, producing essentially garbage weights and
SQNR=-2.9 dB (random-noise level).

## Fix Approach

- Either (a) implement TensorCoreTiledLayout support in _convert_weight_to_int4pack_xpu (WeightInt4Pack.cpp:17) by honoring `innerKTiles` and packing with the same K-tile interleave as CUDA, and correspondingly update the XPU dequant/gemm kernels (sycl/Dequant_int4.cpp:95, sycl/LinearInt4.cpp:220).
- or (b) register an XPU-specific layout in torchao (like the existing BMG/XPU `Int4XPULayout`) and dispatch Int4WeightOnlyConfig to it on XPU so the packed data matches what _weight_int4pack_mm_xpu expects.
- Add a numerics test mirroring test_workflow_e2e_numerics for xpu checking SQNR>=16.5.
