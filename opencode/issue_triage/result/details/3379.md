## Root Cause

`test/distributed/test_c10d_xccl.py::ProcessGroupXCCLGroupTest::test_get_uid` fails with `Expected 0
but got -1` at test_c10d_xccl.py:1291 (`self.assertEqual(_get_process_group_uid(pg), 0)`). The XCCL
ProcessGroup (`torch/csrc/distributed/c10d/ProcessGroupXCCL.{hpp,cpp}` in the distributed_2.11
branch) does not assign / return a per-PG UID counter the way ProcessGroupNCCL does —
`_get_process_group_uid` falls through to the default `-1`. The XCCLTraceTest failures (`test_long`,
`test_batched_send_recv_op_sizes_per_coalesce*`, `test_trace_while_active_*`) share the same root
cause: FlightRecorderXCCL emits records keyed on `pg_uid`, and with `uid=-1` the trace assertions /
sequencing checks fail. This is an XCCL feature gap, not an accuracy bug despite the issue title.

## Fix Approach

- Implement `getUid()` (or equivalent) on `ProcessGroupXCCL` mirroring `ProcessGroupNCCL::getUid` in pytorch/torch/csrc/distributed/c10d/ProcessGroupNCCL.cpp: maintain a static atomic counter incremented on construction and expose it via the C10d Python binding consumed by `_get_process_group_uid`.
- Wire the same uid into `FlightRecorderXCCL` entries so the XCCLTraceTest assertions on `pg_uid` succeed.
- Add the test list to torch-xpu-ops CI and verify on `distributed_2.11` branch (https://github.com/daisyden/pytorch).
- Track on torch-xpu-ops + upstream pytorch (XCCL code lives in pytorch core).
