## Root Cause

ProcessGroupXCCL.getUid() / FlightRecorder bookkeeping returns -1 instead of a monotonic id starting
from 0 expected by test_get_uid (test_c10d_xccl.py:1291) and several XCCLTraceTest cases (test_long,
test_batched_send_recv_op_sizes_per_coalesce*, test_trace_while_active_*). The XCCL
flight-recorder/trace path in third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp does not
assign per-PG unique ids equivalent to NCCL's getUid(), so values diverge from NCCL reference
behavior expected by the upstream test.

## Fix Approach

- Implement getUid()/setUid() bookkeeping for ProcessGroupXCCL (mirror ProcessGroupNCCL's PG counter) and ensure FlightRecorderXCCL emits seq=0,..N.
- Update ProcessGroupXCCL.cpp to allocate uid_ in constructor and expose pybind getter.
- align trace state transitions ('started'->'completed') so XCCLTraceTest asserts pass.
