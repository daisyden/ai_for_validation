## Root Cause

The XPU topk kernel hard-rejects any selected dimension whose extent exceeds INT_MAX via TORCH_CHECK
at third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/TensorTopKKernel.cpp:62-64, instead of
dispatching to a 64-bit indexing path. The same kernel uses int-typed segment counters and the
segmented_group_select_pairs path (line 77+) is hard-coded for 32-bit indices, so any input with
nelements > INT_MAX is unsupported. CUDA upstream had the inverse silent-overflow bug (issue 173622,
PR 176095) that was fixed by widening to int64_t; XPU never got that treatment.

## Fix Approach

- Port pytorch/pytorch#176095-style 64-bit indexing into TensorTopKKernel.cpp: template the kernel on an IndexType (int vs int64_t) and dispatch based on `nelements > INT_MAX`.
- Mirror the same change in Sorting.cpp:105 since it has the identical guard.
- Keep the int path as the fast path and add the int64_t fallback for the >INT_MAX case.
- Add a regression UT under torch-xpu-ops/test/xpu mirroring the upstream CUDA test.
