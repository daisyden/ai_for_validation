## Root Cause

test_binary_ufuncs.test_ldexp_xpu fails with 'DispatchStub: missing kernel for xpu' at
aten/src/ATen/native/DispatchStub.cpp:275. The new integer-exponent path in
aten/src/ATen/native/BinaryOps.cpp:1574-1582 (_ldexp_int_exponent) builds a TensorIterator and calls
ldexp_stub(iter.device_type(), iter). The XPU ldexp_kernel is registered at
third_party/torch-xpu-ops/src/ATen/native/xpu/BinaryOps.cpp:80 and implemented at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/BinaryMiscOpsKernels.cpp:158, but the
AT_DISPATCH_FLOATING_TYPES_AND2 gate on iter.input_dtype(0) does not cover the integer-exponent
iterator shape used by the new path, and the stub registration ends up not being picked up for the
produced iterator device_type, so the dispatcher reports no XPU entry.

## Fix Approach

- Align third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/BinaryMiscOpsKernels.cpp:158 ldexp_kernel with the CUDA implementation (aten/src/ATen/native/cuda/BinaryMiscOpsKernels.cu ldexp_kernel_cuda): mirror the exact AT_DISPATCH template/dtype list it uses for the int-exponent iterator, and verify REGISTER_XPU_DISPATCH(ldexp_stub, ...) in BinaryOps.cpp:80 is linked into the XPU library (add an explicit reference in Ops.cpp registry if stripped).
- Then remove the WA skip added in pytorch PR #171238.
