## Root Cause

All three failures hit the XPU index_add_ kernel registered at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/Indexing.cpp:368 (index_add_kernel) / :431-433
(AT_DISPATCH_INDEX_TYPES "index_add_xpu"). The bf16 cases show L2 rel error ~0.017-0.42 and the
fp32+alpha case shows ~5e-4 rel vs upstream tolerance 1e-5 (test_indexing.py:2158/2198/2218),
consistent with a parallel-reduction layout that accumulates in the input dtype rather than
promoting to fp32 like the CUDA path. Reduction order non-determinism across work-items further
amplifies the alpha case.

## Fix Approach

- In Indexing.cpp index_add_kernel, force fp32 accumulation for low-precision dtypes (bf16/fp16) using accumulate_type<scalar_t,true>, mirroring CUDA's IndexKernel.cu accumulator promotion.
- Audit atomic add path for ordering stability.
- Re-run test_index_add_correctness_2d_* to confirm L2 rel < 0.01 (bf16) and atol/rtol 1e-5 (fp32).
- In the meantime, add the three Cases: lines to Known_issue.log.
