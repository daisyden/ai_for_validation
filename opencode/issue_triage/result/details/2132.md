## Root Cause

test/nn/test_pooling.py:2059-2068 defines test_pooling_large with the @onlyCUDA decorator and
hardcodes device='cuda' inside the helper (line 2064) plus torch.cuda.synchronize() (line 2068). The
torch-xpu-ops test runner in third_party/torch-xpu-ops/test/xpu re-instantiates the NNDeviceType
test class for XPU, but the test body still routes to CUDA, which raises 'Torch not compiled with
CUDA enabled' on a Windows CUDA-less wheel. The @onlyCUDA guard should have prevented XPU
instantiation; it is being ignored because the xpu test runner explicitly patches/overrides device
filters.

## Fix Approach

- Add TestPoolingNNDeviceTypeXPU.test_pooling_large_xpu to the torch-xpu-ops skip list (test/xpu/skip_list_common.py or the pooling-specific skip dict) so the @onlyCUDA-only large-tensor test is not run on XPU.
- Alternatively, submit an upstream PR to generalize the test to accept the device parameter (replace 'cuda' literal with device, use torch.accelerator.current_accelerator().synchronize()).
