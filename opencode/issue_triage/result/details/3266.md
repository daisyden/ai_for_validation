## Root Cause

RFC / refactor: XPU SYCL kernels under third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ currently
call libm/std math (std::exp, ::expf, sqrtf, std::tanh, ...) inside device functors instead of
sycl:: math, e.g. the STD_FUNCTOR macro at ForeachUnaryKernels.cpp:181 expanding to std::OP_NAME at
lines 189-324, bare ::expf/::logf/sqrtf in Philox4x32.h, and std::abs/std::log/std::exp spread
across ~60+ files. This is not a correctness bug today but misses potential accuracy/perf alignment
with CUDAMathCompat.h and creates divergent behavior with at::exp/log/log1p/tan fast-math paths in
aten/src/ATen/NumericUtils.h (no XPU branch).

## Fix Approach

- Execute the 34-item task list from the RFC as individual PRs.
- Start with infrastructure: (0) update XPUMathCompat.h with sycl:: equivalents mirroring CUDAMathCompat.h.
- (0b) add a __SYCL_DEVICE_ONLY__ branch using sycl::native::{exp,log,log1p,tan} in aten/src/ATen/NumericUtils.h upstream.
- Then migrate function-by-function, preserving std:: for c10::complex overloads (AbsKernel, UnarySignKernels, SharedReduceOps, ForeachUnaryKernels complex paths) and for host code.
- Each PR must include accuracy + perf regression tests as specified in the RFC.
