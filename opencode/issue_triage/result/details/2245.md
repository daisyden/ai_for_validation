## Root Cause

test_addmm_errors feeds invalid CSR inputs expecting the CSR front-end check 'crow_indices[..., 0]
== 0' to fire. On CUDA addmm validates CSR metadata before calling cuSPARSE. The XPU
addmm_out_sparse_csr path in
third_party/torch-xpu-ops/src/ATen/native/sparse/xpu/SparseCsrTensorMath.cpp:93 densifies the sparse
operand via mat1.to_dense() / mat2.to_dense() and then routes through oneDNN matmul
(aten/src/ATen/native/mkldnn/xpu/Blas.cpp), which rejects the resulting shape with 'could not create
a primitive descriptor for the matmul primitive' long before any CSR invariant is checked. The test
therefore fails the regex match.

## Fix Approach

- Invoke the standard CSR validation (at::_validate_sparse_csr_tensor_args or the same crow_indices/col_indices checks used on CUDA) at the top of addmm_out_sparse_csr before any layout conversion, so the same error message surfaces on XPU.
- Alternatively, update the XPU skip/xfail list to keep the test skipped until the validation parity is implemented.
- Long term the XPU path should also fall back cleanly (or raise a clearer error) instead of surfacing a raw oneDNN primitive-descriptor failure.
