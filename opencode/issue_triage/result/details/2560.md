## Root Cause

addcmul does not accept a CPU scalar tensor for the `tensor2` (and `tensor1`) argument on XPU. In
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/PointwiseOpsKernels.cpp:48-59 (addcmul_kernel),
the implementation explicitly TORCH_CHECKs that iter.is_cpu_scalar(1)/iter.is_cpu_scalar(2) is
false, but the test passes `c = rand_tensor([], device='cpu')` which becomes arg 3 in the iterator;
gpu_kernel then asserts `iter.device(arg).is_xpu()` at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/Loops.h:626 because no hoisting of arg 3 to a
scalar is done. Other similar test_cuda_vitals_gpu_only* tests hit the same generic iterator path
(Loops.h:626) when a CPU scalar slips in.

## Fix Approach

- Mirror CUDA's gpu_kernel support for CPU scalars: before calling gpu_kernel in addcmul_kernel, detect iter.is_cpu_scalar(3) (and/or 2), extract the scalar value via iter.scalar_value<scalar_t>(idx), remove that operand with iter.remove_operand(idx), and dispatch a 2-input functor that captures the scalar.
- Replace the TORCH_CHECKs at PointwiseOpsKernels.cpp:49-59 with this hoisting path.
- In parallel, extend Loops.h gpu_kernel to recognize is_cpu_scalar and hoist (matching ATen/native/cuda/Loops.cuh).
- Add coverage for addcmul/addcdiv/where CPU-scalar cases and re-enable the listed test_torch_xpu.py tests.
