## Root Cause

GitHub Actions concurrency group in torch-xpu-ops pull workflow auto-cancels duplicate workflow runs
when a PR is created with several disable_* labels; each label-add event re-triggers the workflow
and the concurrency.cancel-in-progress logic kills the prior run. Resulting UI in PR Conversation
tab only shows the last triggered run, hiding the actually-completed one. CI/infra harness issue, no
kernel/runtime impact.

## Fix Approach

- Inspect .github/workflows/_linux_ut.yml / pull.yml concurrency: definition.
- either widen the concurrency group key to include the label set, or debounce label-triggered runs (e.g., on: pull_request with types restricted, or use 'labeled' event with cancel-in-progress: false for label adds at PR creation).
