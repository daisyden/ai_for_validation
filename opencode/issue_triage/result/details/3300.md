## Root Cause

On PR creation, multiple instances of the `_linux_ut.yml` / pull workflow are dispatched (likely via
both `pull_request` and `pull_request_target` triggers or by each `disable_*` label add firing a
labeled event), and the concurrency group cancels all but one. The surviving run is sometimes not
the last so it is hidden from the Conversation tab. This is a CI workflow trigger/concurrency-group
misconfiguration in .github/workflows, not a runtime bug.

## Fix Approach

- Audit triggers in .github/workflows/_linux_ut.yml and pull.yml: ensure only one trigger event (pull_request) fires, and tighten the `concurrency: group:` key so labeled events on PR creation are deduped.
- Consider gating on `if: github.event.action != 'labeled' || ...` or moving disable_* labels to PR body markers.
