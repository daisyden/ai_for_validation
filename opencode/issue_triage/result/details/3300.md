## Root Cause

Pure CI/GitHub Actions workflow infra issue: when a PR is opened with disable_* labels applied at
creation time, the labeled and pull_request triggers fire near-simultaneously
(.github/workflows/_linux_ut.yml / pull.yml use concurrency groups keyed on PR number plus
label-driven re-dispatch). Concurrency.cancel-in-progress=true then cancels all but the
last-launched run, and because the surviving run is not the most-recently-rendered one, it is hidden
from the Conversation tab while still visible under Checks. No code path in torch-xpu-ops src/ is
involved.

## Fix Approach

- Refine the pull-workflow concurrency group to include the label set (or debounce via on: pull_request types: [opened, labeled, synchronize] with a single dispatch job) so duplicate runs are coalesced before launch instead of cancelled mid-flight.
- Alternatively, gate the labeled trigger to skip when action == 'opened' since pull_request already covers that.
