## Root Cause

PT2E INT8 throughput on resnext50_32x4d regresses ~12% (Asymm) / ~13% (Symm) on torch 2.12.0+xpu vs
release/2.11 while FP32 is unchanged. The shared INT8 lowering path is
torch/_inductor/mkldnn_ir.py::QConvPointWisePT2E.create -> _prepare_convolution_fusion_create (gated
by SUPPORTED_MKLDNN_DEVICES = ('cpu','xpu') in torch/_inductor/utils.py) which dispatches to
oneDNN-graph aoti_torch_xpu__qconv_pointwise_tensor. Because resnext50 is grouped-conv
(cardinality=32) and the regression is independent of zero-point asymmetry, the slowdown is in the
oneDNN-graph grouped-INT8-conv primitive selection / channels-last propagation, not the asymmetric
w_zp branch at torch/_inductor/mkldnn_lowerings.py:763.

## Fix Approach

- Capture ONEDNN_VERBOSE on 2.11 vs 2.12 builds for resnext50_32x4d INT8 to confirm which qconv primitive degraded.
- bisect the inductor mkldnn_ir.py / mkldnn_lowerings.py changes between 2.11 and 2.12 for grouped-conv channels-last propagation.
- If the regression is in oneDNN itself, file upstream and pin/patch the version.
