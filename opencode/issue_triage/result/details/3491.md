## Root Cause

PT2E INT8 throughput on resnext50_32x4d (grouped conv, cardinality=32) regresses ~12-13% on torch
2.12.0+xpu vs release/2.11 under both ASYMM and SYMM, while FP32 is unchanged. The shared CPU/XPU
mkldnn lowering path in torch/_inductor/mkldnn_ir.py (QConvPointWisePT2E.create ->
_prepare_convolution_fusion_create, gated by SUPPORTED_MKLDNN_DEVICES=('cpu','xpu') in
torch/_inductor/utils.py) dispatches to oneDNN graph via aoti_torch_xpu__qconv_pointwise_tensor; a
2.11->2.12 change in channels-last propagation, grouped-conv post-op fusion, or w_zp materialization
(torch/_inductor/mkldnn_lowerings.py:763) selects a slower oneDNN primitive. Additionally,
torch/_inductor/select_algorithm.py:3254 silently swallows TensorMeta.from_irnodes failures for the
extern qconv choice, so the autotuner picks a non-optimal kernel. Localized to INT8 conv lowering /
oneDNN kernel selection on XPU, not Dynamo or post-grad passes.

## Fix Approach

- Bisect 2.11..2.12 commits in torch/_inductor/mkldnn_ir.py and torch/_inductor/mkldnn_lowerings.py touching _prepare_convolution_fusion_create / w_zp / channels-last to identify the regression commit.
- Verify oneDNN graph primitive chosen for grouped INT8 conv (compare verbose=oneDNN_VERBOSE=1 logs between 2.11 and 2.12).
- Fix TensorMeta.from_irnodes failure for XPU extern qconv in select_algorithm.py:3254 (don't swallow the exception silently.
- populate proper meta for XPU layouts).
- File companion fix in pytorch/pytorch and uprev oneDNN if grouped-conv fusion eligibility regressed.
