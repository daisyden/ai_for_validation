## Root Cause

test/inductor/test_triton_heuristics.py::test_prune_configs_over_shared_memory_limit_do_pruning_True
expects 1 surviving config but gets 2 on XPU. In torch/_inductor/choices.py
(_get_exceeding_shared_memory_checker, shown in the issue) the checker only queries
torch.cuda.current_device() / get_device_properties().shared_memory_per_block_optin; on XPU the
try/except returns None, so no config pruning happens and all configs pass through. PR
pytorch/pytorch#172314 added the SLM-size query on the XPU device-properties side, but the heuristic
and the test were never updated to consume it on XPU.

## Fix Approach

- After pytorch/pytorch#172314 lands, update torch/_inductor/choices.py::_get_exceeding_shared_memory_checker to be device-agnostic: use torch.accelerator.current_accelerator() (or dispatch on current_device().type) to query torch.xpu.get_device_properties(dev).shared_memory_per_block_optin when the device is XPU, and fall back to CUDA path otherwise (ideally factor a BaseDeviceConfigHeuristics as the TODO suggests).
- Mirror the change in test_triton_heuristics.py so the expected count is produced on XPU, and remove the skip/xfail once green.
