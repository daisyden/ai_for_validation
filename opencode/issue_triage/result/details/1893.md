## Root Cause

test_out_mv / test_out_addmv (float32) and a few test_neg_view_* (float64) in
test/xpu/test_ops_xpu.py fail assertEqual by ~2e-7 absolute / ~3e-7 relative (default tol is 1e-7) —
e.g. test/test_ops.py:2104 _test_math_view. Root cause is accumulation-order differences between the
oneDNN XPU matmul/gemv kernels and the eager CPU reference; differences are within normal fp32/fp64
float rounding but exceed the strict default tolerance. Most sub-cases have since passed in nightly
CI, leaving only test_out_mv/test_out_addmv_xpu_float32 persistently failing.

## Fix Approach

- Bump tolerance for the remaining mv/addmv fp32 ops in XPU OpInfo (torch-xpu-ops test/xpu/xpu_test_utils.py or a toleranceOverride for mv/addmv on XPU) rather than modifying kernels, since the gap is within expected fp32 accumulation error for oneDNN gemv.
- Remove now-passing cases from the skip list and add a comment linking this issue.
