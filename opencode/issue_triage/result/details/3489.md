## Root Cause

PT2E INT8 accuracy on shufflenet_v2_x1_0 (depthwise/grouped-conv + channel-shuffle heavy) collapses
from Acc@1 66.5 to 0.09 between release/2.11 and 2.12.0+xpu, while FP32 on the same build remains
healthy — a hard correctness regression in the XPU PT2E INT8 path. Two diagnostic warnings appear
only on the INT8 run: (1) torch/_inductor/mkldnn_lowerings.py:763 re-wraps w_zp via
torch.tensor(<existing tensor>, dtype=int32), indicating a w_zp dtype/layout change is exercised
every run; (2) torch/_inductor/select_algorithm.py:3254 logs 'Constructing input/output tensor meta
failed for Extern Choice' and silently proceeds with empty meta, allowing wrong-layout extern
dispatch for the XPU qconv_pointwise. Both paths sit in shared CPU/XPU lowerings (mkldnn_ir.py
QConvPointWisePT2E -> oneDNN graph aoti_torch_xpu__qconv_pointwise_tensor, gated by
SUPPORTED_MKLDNN_DEVICES in torch/_inductor/utils.py), so a regression intended for CPU is
corrupting XPU INT8 weight quantization math.

## Fix Approach

- Reproduce on release/2.11 vs 2.12 and bisect commits to mkldnn_lowerings.py / mkldnn_ir.py / select_algorithm.py between the two tags.
- the w_zp re-wrap at L763 and the swallowed TensorMeta exception at L3254 are the prime suspects.
- Audit w_zp dtype/stride round-trip for XPU (ensure int32 contiguous, no spurious copy that drops layout).
- Make TensorMeta.from_irnodes succeed on XPU qconv extern choice (don't except-and-warn — populate proper IR layout) so the autotuner picks the correct kernel.
- Add a PT2E INT8 shufflenet correctness test to nightly XPU CI to gate future regressions.
