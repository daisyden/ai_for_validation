## Root Cause

aten::mm, addmm, linear and convolution produce non-bitwise-deterministic outputs across two runs on
XPU PVC, while CUDA A100 is bitwise-equal for the same inputs (max abs diff up to 1.45e-4 for conv,
1.56e-2 for bf16 linear). These ops are dispatched through oneDNN matmul/conv primitives in
torch-xpu-ops (src/ATen/native/xpu/Blas.cpp / mkldnn matmul/conv paths); oneDNN may select different
parallel reduction kernels or work-group splits run-to-run when use_deterministic_algorithms is not
enforced, and the XPU backend does not honor torch.use_deterministic_algorithms(True) for these
primitives.

## Fix Approach

- Audit XPU dispatchers for mm/addmm/linear/convolution to honor at::globalContext().deterministicAlgorithms().
- when set, force oneDNN primitive descriptors to a deterministic implementation (or fall back to a fixed reduction order).
- Add tests under test_ops_xpu.py mirroring CUDA's deterministic checks.
- Coordinate with oneDNN team if a deterministic flag/attr is missing for matmul/conv on PVC.
