## Root Cause

aten::mm, addmm, linear and convolution on XPU are dispatched to oneDNN matmul/convolution
primitives (third_party/torch-xpu-ops/src/ATen/native/xpu/mkldnn/Matmul.cpp, Conv.cpp, Linear.cpp).
These primitives select algorithms/partitioning heuristically per invocation (different reduction
orders, split-K, post-op fusion) so bit-exact results are not guaranteed across identical runs,
producing max_diff ~1e-6 for fp32 matmul and ~1.5e-2 for bf16 linear. CUDA's cublasLt/cuDNN paths,
with PyTorch's deterministic defaults, produce bitwise-equal results. XPU currently ignores
torch.use_deterministic_algorithms(True) for these ops.

## Fix Approach

- Wire torch.use_deterministic_algorithms through XPU matmul/conv: pin oneDNN primitive attributes (dnnl::fpmath_mode, scratchpad mode, deterministic flag) and select a stable algorithm (e.g. avoid split-K and atomic accumulation) in the XPU oneDNN op wrappers.
- Where oneDNN cannot guarantee determinism, raise the same NotImplementedError that CUDA does.
- Also document that default XPU execution is non-deterministic, matching CUDA behavior only when the deterministic flag is set.
