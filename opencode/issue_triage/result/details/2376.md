## action_reason

- OpInfo BinaryUfuncInfo('logaddexp') at torch/testing/_internal/common_methods_invocations.py:14626-14630 declares dtypesIfXPU=floating_types_and(bfloat16, float16) with no complex types, so complex128 variant is not generated.

## Root Cause

CUDA added complex dtype support for logaddexp/logaddexp2 but the XPU kernel historically only
dispatched floating types, producing 'NotImplementedError: logaddexp_xpu not implemented for
Complex' across ~60 op_ut cases (binary_ufuncs, decomp, meta, linalg, gradients). Kernel has since
been implemented at third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/LogAddExpKernels.cpp (see
`_logaddexp_minmax` and the complex branch around line 73-90 and dispatch macro at line 140-146),
but the skip workarounds in stock PyTorch test suite (PR pytorch/pytorch#171238) and xpu_test skip
lists still exclude these cases.

## Fix Approach

- Remove the XPU-specific skips/WAs added in pytorch/pytorch#171238 (test/test_binary_ufuncs.py, test_decomp.py, test_meta.py OpInfo entries) and update torch-xpu-ops skip lists (test/xpu/skip_list_common.py / test_binary_ufuncs_xpu.py) to re-enable logaddexp complex64/complex128/complex32 cases.
- Re-run the 60+ listed op_ut cases to confirm.
- if complex32 remains numerically unstable, keep only that dtype skipped and file a narrower follow-up.
- Coordinate with CuiYifeng (kernel owner) to land the test-enable PR.
