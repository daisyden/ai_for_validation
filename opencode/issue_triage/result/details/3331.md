## Root Cause

torch.compile(inductor) on `slice_scatter(x, y, dim=1, start=0, end=6).sum(dim=1)` produces an
x.grad that differs from CPU eager by ~3.67 (essentially garbage), while the forward matches. The
backward of slice_scatter w.r.t. the destination `x` should zero out the [0:6) slice along dim=1 and
pass the rest of grad through; the inductor-generated triton-xpu kernel for this masked-fill
backward is emitting wrong values (likely an incorrect mask or stride in the scatter-add/zero
kernel). This is the XPU mirror of upstream pytorch/pytorch#180164 and is reproducible on Max 1550
with triton-xpu 3.7.1+git21033c4e, so the defect lives in inductor codegen / triton-xpu lowering of
the slice_scatter backward rather than in a hand-written torch-xpu-ops SYCL kernel (no
aten::slice_scatter_backward SYCL op exists; aten/native/TensorShape.cpp slice_scatter is
decomposed).

## Fix Approach

- Reproduce with TORCH_LOGS=output_code,inductor and capture the generated triton kernel for the backward.
- diff against the CUDA-generated kernel for the same graph.
- File/track upstream pytorch/pytorch#180164 for fix in torch/_inductor/lowering.py / decomposition of slice_scatter backward.
- if XPU-specific (e.g. triton-xpu mask handling), open a triton-xpu issue.
- Add a regression test in test/inductor/test_torchinductor.py once fixed.
