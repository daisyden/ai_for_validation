## Root Cause

torch.compile(backend='inductor') on XPU produces an incorrect x.grad for torch.slice_scatter (max
abs diff ~3.67 vs CPU eager). Forward is fine but the backward lowering of slice_scatter through
Inductor (the autograd backward decomposes to slice + zero-fill of the complement region) is
generating wrong indexing/masking on the Triton-XPU code path. No XPU-specific slice_scatter kernel
exists in third_party/torch-xpu-ops/src/ATen/native/xpu/ (grep returned nothing), so the bug is in
the inductor lowering / triton-xpu codegen for the backward, mirroring upstream
pytorch/pytorch#180164 referenced by the reporter.

## Fix Approach

- 1) Reproduce with TORCHINDUCTOR_TRACE / output_code dump on XPU and CUDA.
- diff the generated Triton kernel for the backward to localize the bad index/mask expression. 2) Cross-check upstream issue pytorch/pytorch#180164 and any candidate fixes there.
- if the upstream fix already lands a corrected backward decomposition / lowering for slice_scatter, port/pin it. 3) Add a regression test mirroring the reporter's reproducer under test/xpu (CPU vs XPU compiled grad allclose) to prevent regression. 4) If root cause is triton-xpu codegen rather than the lowering, file a triton-xpu issue with the minimal generated kernel.
