## Root Cause

Follow-up perf investigation task from PR #2763 fixing test_large_max_pool_contig (issue #2366). The
MaxPool2d XPU dispatch in third_party/torch-xpu-ops/src/ATen/native/xpu/DilatedMaxPool2d.cpp and
src/ATen/native/xpu/sycl/DilatedMaxPool2d.cpp was implicitly converting Contiguous input to
ChannelsLast, producing wrong indices. Benchmark report (in comment) shows the implicit
Contig->ChannelsLast conversion is >2x slower than just running the native Contiguous kernel for
tensors >10M elements, so layout-aware dispatch is needed.

## Fix Approach

- Make the dispatcher in DilatedMaxPool2d.cpp choose kernel by suggested memory format of the input: ChannelsLast input -> ChannelsLast kernel, Contiguous input -> Contiguous kernel.
- never silently call .contiguous(MemoryFormat::ChannelsLast).
- Add a regression perf test for both layouts at large sizes (>500M elems).
