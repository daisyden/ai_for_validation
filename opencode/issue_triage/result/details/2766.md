## Root Cause

Follow-up investigation task (not a crash) after PR #2763 fixed #2366 by removing an implicit
`Contiguous -> ChannelsLast` layout conversion inside the MaxPool2d XPU implementation at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/DilatedMaxPool2d.cpp. BBBela's benchmark shows
that while the ChannelsLast kernel is 1.07x-1.68x faster than the Contiguous kernel on its native
layout, implicitly converting a Contiguous input to ChannelsLast before pooling is ~2x slower than
running the Contiguous kernel directly once tensors exceed ~10M elements, because the
memory-bandwidth cost of the layout conversion dominates. So the current layout-aware dispatch in
DilatedMaxPool2d.cpp is the right choice for large contiguous inputs.

## Fix Approach

- Keep the layout-aware dispatch introduced by PR #2763 as the default (no implicit Contiguous->ChannelsLast conversion).
- Optionally add a size-threshold heuristic in DilatedMaxPool2d.cpp: for small contiguous inputs (roughly <=4M elements) where the ChannelsLast kernel plus conversion is break-even or slightly faster, switch to the ChannelsLast path.
- otherwise use the native Contiguous kernel.
- Improve the ChannelsLast Contiguous-kernel backward performance separately, since backward is the main gap.
- Document the decision in the kernel file and close the issue once the threshold heuristic (if adopted) is merged.
