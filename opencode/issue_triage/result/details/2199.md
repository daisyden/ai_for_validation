## Root Cause

ReduceKernel<vt0, ReduceOp<...>> (third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/Reduce.h:397
and ReduceOp at line 477) is heavily templated and, when vt0=4 with ArgMin/ArgMax/Min/Max/Norm ops,
instantiates with 256 allocated registers and spills 30-150 registers under SIMD32. Root cause is
(a) oversized thread-private storage (input_vec_t/out_scalar_t/index pairs accumulated across vt0
unroll factor) plus inlined reduce_fragment/accumulate paths, and (b) lack of a register-count hint
(intel::reqd_sub_group_size / max_work_group_size) to guide IGC. This causes perf regressions on
affected kernels without functional failure.

## Fix Approach

- (1) Reduce vt0 to 2 for wide types (long/int64) and composite types (pair<index,value> in ArgMin/ArgMax) where spill is highest.
- (2) split the combined load+accumulate loop in ReduceOp::run into smaller loops so the compiler can reuse registers.
- (3) move large per-thread scratch arrays to SLM for the argmin/argmax case via group_reduce utils in GroupReduceUtils.h.
- (4) add sycl::reqd_sub_group_size(16) or add a specialized low-register path for long/Half types.
- Verify no remaining 'spilled around' warnings in the compile log.
