## Root Cause

torchbench squeezenet1_1 bf16 inference and functorch_dp_cifar10 training fail the dynamo accuracy
comparison on BMG/PVC with RMSE exceeding the tolerance multiplier (reported at
torch/_dynamo/utils.py:2998). The comment from 2025-11-21 confirms squeezenet1_1 now passes on 2.10,
leaving only functorch_dp_cifar10 (bn1.bias.grad mismatch) — pointing to a batch-norm backward
numeric discrepancy in the inductor/triton-xpu code path rather than an eager-mode ATen op. No
reproducer on CUDA, so likely triton-xpu codegen rounding in bn reductions.

## Fix Approach

- Bisect functorch_dp_cifar10 bf16 training failure to isolate the differing kernel (suspected BatchNorm backward) by comparing eager vs inductor outputs for bn1.bias.grad.
- test with TORCHINDUCTOR_MAX_AUTOTUNE=0 and disabling fused reductions to identify whether intel-xpu-backend-for-triton lowering of welford/reduction is at fault.
- If so, file upstream triton issue.
- otherwise fix the corresponding decomposition/lowering in pytorch/_inductor.
- Also close the squeezenet portion (already resolved on 2.10).
