## Root Cause

With the XPU Triton backend and configs `triton.use_block_ptr=True` +
`triton.codegen_upcast_to_fp32=False`, the argmax Inductor codegen at
torch/_inductor/codegen/triton.py:4446-4471 emits a `tmp3 = tmp3_idx[:, None]` followed by an
unexpected `tmp3 = tmp3.to(tl.float16)` before the final `.to(tl.int64)` store. The extra fp16 cast
appears because XPU's dtype-propagation pass infers the index variable's dtype from the surrounding
reduction value's dtype (fp16) rather than from `select_index_dtype()` (int64) when generating the
post-loop block-ptr store; the cast is injected by the generic output-to-src-dtype rewrite when
`codegen_upcast_to_fp32=False`. CUDA skips the cast because its block-ptr store path special-cases
integer indices. The test `test_float16_reduction_with_int_output` at
test/inductor/test_cuda_repro.py:2893 is already `@skipIfXpu`'d pointing to this issue.

## Fix Approach

- In the Triton reduction codegen for argmax/argmin (triton.py around line 4469 `final_argreduce` and the block-ptr store generation), ensure the index result variable's dtype is tracked as int (from `select_index_dtype()`) so the dtype-propagation pass does not insert a cast to the value dtype.
- Concretely, when `codegen_upcast_to_fp32=False` and the reduction is argmax/argmin, the post-loop index store should bypass the fp16 src-dtype coercion.
- add a guard in `TritonKernel.store`/`codegen_indirect_indexing` that skips `.to(tl.float16)` when the logical dtype is integer.
- Validate by re-enabling the test on XPU and confirming generated code contains no `.to(tl.float16)` and numerical parity vs eager argmax.
