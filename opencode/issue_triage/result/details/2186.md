## Root Cause

nn.MultiheadAttention fast-path gating in torch/nn/modules/activation.py:1056-1063
(_check_arg_device) only allows device types 'cpu', 'cuda', and the registered privateuse1 backend
name. 'xpu' is a first-class native backend (not privateuse1), so the check fails and the assertion
at activation.py:1449 fires for NestedTensor inputs on XPU. The issue is purely in the eligibility
predicate; the underlying XPU SDPA / nested path itself is not actually invoked.

## Fix Approach

- Extend _check_arg_device in torch/nn/modules/activation.py to accept 'xpu' (e.g. `x.device.type in {'cpu', 'cuda', 'xpu', torch.utils.backend_registration._privateuse1_backend_name}`).
- Also update the error-message string at line 1415 accordingly.
- Add a corresponding XPU entry in the TransformerEncoder 'why_not_sparsity_fast_path' list in torch/nn/modules/transformer.py if present.
- Verify test_with_nested_tensor_input_xpu and test_transformerencoderlayer_xpu_{float16,float32} pass after the change.
