## Root Cause

test/inductor/test_cuda_repro.py::test_effn_attn_bias_padding uses FileCheck to assert the generated
wrapper contains the stride tuple '(262144, 0, 512, 1' — i.e. the attention-bias tensor reaches
_scaled_dot_product_efficient_attention still expanded with a 0 stride. On XPU the actual generated
code reinterprets arg0_1 as stride (512, 0, 1, 0), so Inductor's padding/expand handling for SDPA
bias produces a different (but functionally valid) stride layout than on CUDA. The test is a
CUDA-specific codegen assertion that was auto-collected into the XPU Inductor suite without being
adapted.

## Fix Approach

- Either (a) skip/xfail this test for XPU in third_party/torch-xpu-ops/test/inductor skip lists because it is a CUDA codegen-text assertion, or (b) relax the FileCheck to accept the XPU stride pattern '(512, 0, 1, 0)'.
- Preferred: add it to the upstream skip list in torch-xpu-ops with a comment noting the stride layout divergence, and open a follow-up to align the XPU SDPA bias expand lowering with CUDA if desired.
