## Root Cause

55+ TestCompositeComplianceXPU.test_cow_input_* tests for linalg/matmul/conv/attention ops
unintentionally materialize COW (copy-on-write) tensors. The XPU kernels (and particularly the
oneDNN-backed matmul/conv/linear paths in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/*Kernels.cpp and the Blas/Linear oneDNN wrappers
under src/ATen/native/xpu/mkldnn/) obtain mutable data pointers on read-only inputs, which triggers
COW materialization. Prior PRs (#2857, #2865, #3226, pytorch#178305) introduced RO/RW wrappers for
many kernels, but matmul/conv still materialize because oneDNN has no const-pointer-aware memory
API. Completion is blocked on oneDNN providing a const-pointer interface (JIRA filed by SH team).

## Fix Approach

- For each remaining op family, audit the XPU kernel path and replace mutable .data_ptr()/.mutable_data_ptr() access on input tensors with const RO wrappers (c10::impl::cow::check_tensor_read_only pattern, as done in PR #2857/#2865).
- For the matmul/conv/linear ops dispatched through oneDNN (src/ATen/native/xpu/mkldnn/{Matmul,Conv,Linear}.cpp), wait for oneDNN const-pointer API and then rewire make_onednn_memory to use read-only buffers.
- Until oneDNN is updated, xfail or skip the dependent tests via decorators in third_party/torch-xpu-ops/test/xpu/skip_list_common.py while keeping the non-oneDNN ones (topk, einsum, tensordot, etc.) fixed in parallel.
