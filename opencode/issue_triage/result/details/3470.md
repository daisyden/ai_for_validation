## Root Cause

The CI summary job in torch-xpu-ops re-queries GitHub for issues with the 'skipped' label at summary
time rather than snapshotting the allowlist at build start. When a multi-hour build+UT run begins
with an issue open, the underlying fix lands on main, and the issue is closed before summary
executes, the still-pre-fix UT failure is no longer matched against the (now-closed) skipped-issue
list and is reported as a false regression. This is a CI/infra ordering bug internal to
torch-xpu-ops workflow scripts (PR #3429 already proposes a fix).

## Fix Approach

- Snapshot the skipped-issue allowlist at build-start time (write Known_issue.log as a workflow artifact at the start of the build job) and have the summary job consume that artifact instead of re-fetching live GitHub state.
- Review and merge PR #3429 which implements this.
- Optionally add a fallback that also checks recently-closed (within run duration) skipped issues to reduce flakes during the transition.
