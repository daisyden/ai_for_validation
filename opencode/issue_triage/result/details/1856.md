## Root Cause

For channels-last tensor (e.g. shape 512,16,112,112 with stride 200704,1,1792,16), in-place
aten::hardswish_ launches an extra VectorizedElementwiseKernel<CopyScalarFunc> before the
HardswishFunctor kernel. The Loops/TensorIterator path used by hardswish_kernel in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ActivationHardswishKernels.cpp:58-64 does not
preserve channels-last memory layout and falls back to materializing a contiguous copy, doubling
memory traffic.

## Fix Approach

- Make hardswish_kernel (and related pointwise activations) channels-last aware in TensorIterator setup so vectorized kernel runs directly on the original strided memory.
- alternatively register a foreach/structured variant similar to relu that keeps suggested memory format.
