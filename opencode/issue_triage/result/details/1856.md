## Root Cause

hardswish_kernel at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ActivationHardswishKernels.cpp:58 dispatches
through the generic Loops-based gpu_kernel which materializes a contiguous copy when the
TensorIterator input is channels-last (stride {200704,1,1792,16}). The extra
VectorizedElementwiseKernel<CopyScalarFunc> seen in the profile is the TensorIterator trying to fold
non-standard strides — the XPU Loops path does not preserve channel-last memory format for inplace
pointwise ops the way CUDA's jiterator/foreach does. relu_ avoids this because it takes the in-place
TensorIteratorBase fast path.

## Fix Approach

- Teach the hardswish (and other Activation*Kernels.cpp files using gpu_kernel) to detect NHWC/channels-last input and dispatch on a flattened contiguous view via TensorIterator::Builder::check_all_same_dtype(false).build(), or move inplace pointwise activations onto the same foreach-style launcher used by relu_.
- Validate with a microbenchmark on input (512,16,112,112) channels-last half to confirm the Copy kernel is gone and inspect remaining NHWC pointwise activations (hardsigmoid, mish, silu, hardtanh).
