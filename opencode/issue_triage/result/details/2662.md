## Root Cause

Release/2.10 Windows BMG run shows a mixed bag: (1) Most failures in test_content_store.py,
test_multihead_attention.py, etc. are `InductorError: Failed to find C compiler ... cl is not found`
— an environmental/triton-knobs issue on Windows where MSVC `cl.exe` isn't on PATH for the
Triton/inductor C-shim build (torch/_inductor/runtime/triton_heuristics & triton.knobs.build.impl).
(2) Real functional bugs: test_GroupNorm_general_xpu reports scalar mismatch 1.02e-5 vs tol 1e-5
(tiny numerical drift in XPU GroupNorm SYCL kernel `GroupNormKernels.cpp`), and
test_cross_entropy_loss_2d_out_of_bounds_class_index_xpu fails to surface the device-side `Assertion
cur_target >= 0 && cur_target < n_classes` message because SYCL kernel asserts are not piped to
stderr on Windows (Loss kernels in src/ATen/native/xpu/sycl/LossNLLKernel.cpp). These only reproduce
on Windows+BMG; PVC Linux passes.

## Fix Approach

- Split into sub-tasks: (a) CI side — install MSVC Build Tools and set `CC=cl`/add cl.exe to PATH for Windows XPU runners so inductor's C-shim builds.
- or skip the inductor-compile-path tests on Windows. (b) GroupNorm — relax tolerance in the test for half/bf16 on BMG, or investigate welford accumulator precision in `GroupNormKernels.cpp`. (c) cross_entropy OOB — ensure SYCL `SYCL_KERNEL_ASSERT` / printf messages flush to stderr on Windows.
- alternatively mark these assertion-text tests skipped on Windows.
- Coordinate with tadkrawiec/kdrozd-dev for the Windows-specific pieces.
