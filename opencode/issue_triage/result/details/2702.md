## Root Cause

test_monitored_barrier_allreduce_hang relies on ProcessGroup::monitoredBarrier(), which is not
implemented in the XCCL backend. ProcessGroupXCCL.cpp only provides a standard allreduce-based
barrier (third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp:1927) and no monitoredBarrier
override, so when a rank deliberately delays its allreduce, the barrier in _barrier()/Barrier.sync
(distributed_test.py:571) times out after 0 ms because the default Work timeout has already expired
before the collective is launched. The XCCL Work timeout plumbing does not honor the per-op timeout
passed via BarrierOptions either.

## Fix Approach

- Implement ProcessGroupXCCL::monitoredBarrier (mirroring ProcessGroupNCCL::monitoredBarrier: per-rank send/recv based handshake with explicit timeout and rank reporting) and wire it into Register.cpp.
- until then, mark test_monitored_barrier_* XFAIL/skip in the XCCL test allow-list.
- Also fix BarrierOptions::timeout propagation so Work->wait(timeout) uses opts.timeout instead of defaulting to 0 ms.
