## Root Cause

On B60 + XCCL backend, dist.batch_isend_irecv(P2POp(...)) silently returns zeroed/garbage receive
buffers (causing all-black VAE output in sglang-diffusion USP mode), while individual
dist.isend/dist.irecv work. The bug lives in the XCCL P2P batching path —
third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp pointToPoint / coalesced send-recv handling
on top of oneCCL 2021.16.x. Reporter and Intel confirm the failure reproduces on oneCCL 2021.16.1 /
driver libze1 1.24.3 but is fixed on DLE 2026.0 with level-zero 1.27.0 + intel-compute-runtime
26.05.37020.3, indicating an oneCCL/level-zero regression in the batched P2P code path.

## Fix Approach

- Track on DLE 2026.0 / oneCCL upgrade: validate the user’s reproduce script (test_batch_isend_irecv.py) against the new stack and close once green in CI.
- In parallel, audit ProcessGroupXCCL.cpp batch_isend_irecv coalescing (group start/end, send/recv ordering) and add a torch-xpu-ops distributed UT mirroring the halo-exchange pattern so the regression cannot return.
