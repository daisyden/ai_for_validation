## Root Cause

On Arc Pro B60 with XCCL, batch_isend_irecv silently produces zero/garbage receive buffers while
individual isend/irecv work. PyTorch's batch_isend_irecv wraps the P2P list in a coalescing group,
which in third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp:613-665 calls
startCoalescing()->groupStart() (xccl::oneccl_group_start at line 603) and
endCoalescing()->groupEnd() at line 655, issuing the per-op onecclSend/onecclRecv inside
pointToPoint (lines 981-1080) without binding each op to the coalesced comm/stream. Combined with
the ARC_LL fast path enabled by CCL_SYCL_ALLTOALL_ARC_LL=1, oneCCL on Battlemage (B60) appears to
mis-order or drop the coalesced P2P kernels, so the recv buffer is never actually written before the
completion event is recorded on line 657 and req.wait() returns success.

## Fix Approach

- Two parts: (1) audit ProcessGroupXCCL::send/recv (src/xccl/ProcessGroupXCCL.cpp:981 and :1025) and pointToPoint so that when coalescing_state_ & CoalP2P is set they enqueue onto coalescedComm_/coalescedStream with a real stream-sync at endCoalescing before xcclEndEvent_->record (line 657) - i.e. make sure each coalesced send/recv runs inside the same oneccl_group_start/end pair on the same comm like NCCL does. (2) In parallel, file an oneCCL bug against B60 group-P2P + CCL_SYCL_ALLTOALL_ARC_LL path and add a runtime workaround: detect Xe2/ARC device (via XPUDeviceProp) and either disable coalescing for P2P or force the non-LL path until oneCCL ships a fix.
- Add a distributed UT mirroring the reporter's test_batch_isend_irecv.py.
