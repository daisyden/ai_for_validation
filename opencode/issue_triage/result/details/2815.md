## Root Cause

addmv with complex dtype and empty mat (shape [2,0]) / vec (shape [0]) takes the addmm complex
fallback path. In addmm_complex_out_xpu
(third_party/torch-xpu-ops/src/ATen/native/xpu/Blas.cpp:329-362) the mat1.numel()==0 short-circuit
at lines 344-354 calls out.copy_(self) then out.mul_(beta), but for the addmv route out has been
pre-resized to broadcast shape [2,2] while self is the 1D [2] input vector, producing 'output with
shape [2] doesn't match the broadcast shape [2, 2]'. The fallback does not account for the
addmv-vs-addmm shape difference in the empty path even though line 124 acknowledges the mixed-shape
contract.

## Fix Approach

- In the empty short-circuit (Blas.cpp:344-354) reshape/broadcast self to out's shape before copy_, e.g., self.expand_as(out) (or use at::native::expand_inplace), mirroring the reshape_as logic already used in the non-empty branch at lines 181-182.
- Add a regression test mirroring the issue snippet for complex64/complex128 addmv with empty mat/vec.
