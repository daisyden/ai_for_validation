## Root Cause

`torch.full((tgt_len, tgt_len), torch.finfo(dtype).min, device='xpu')` with dtype=bfloat16 (and in
some paths float16) raises `RuntimeError: value cannot be converted to type float without overflow`
on PyTorch 2.11 XPU (WSL + native BMG Windows), whereas 2.10 passed. `torch.finfo(bfloat16).min` ≈
-3.39e38 which overflows fp32; the regression comes from the fill-scalar type-check path used by the
XPU `full` implementation in third_party/torch-xpu-ops/src/ATen/native/xpu/Fill.cpp
(scalar_to<float> check inside checked_convert/checked_from_scalar). CUDA handles this via direct
bfloat16/fp16 scalar conversion so the same user code works. Reproduces with hf TrOCRForCausalLM and
XGLMForCausalLM via benchmarks/dynamo/huggingface.py through transformers
modeling_attn_mask_utils.py:164.

## Fix Approach

- In the XPU `full`/`fill_` scalar dispatch, convert the Scalar directly to the target dtype (bfloat16/half) instead of first narrowing to float.
- Mirror the pattern used on CUDA in aten/src/ATen/native/cuda/FillKernel.cu / Fill.cpp: when `self.scalar_type()` is bfloat16/half, use `value.to<at::BFloat16>()` / `value.to<at::Half>()` rather than `value.to<float>()` before kernel launch.
- The fix lives in third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/FillKernel.cpp (or whichever fill kernel file applies `checked_convert<float>`), and the dispatch in Fill.cpp should route through an AT_DISPATCH_ALL_TYPES_AND2(kHalf, kBFloat16, ...).
- Add a regression unit test filling a bf16/fp16 tensor with finfo.min/finfo.max.
