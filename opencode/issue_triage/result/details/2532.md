## Root Cause

torch.compile lowering of torch.ops.aten._convert_weight_to_int4pack on XPU produces an output with
a different rank/stride than what Inductor's generated assert_size_stride expects. Generated code at
torchinductor cache: assert_size_stride(buf1, (s54 // 8, 1, 32, 1), (32, 32, 1, 1)) - a 4-D shape -
but the XPU kernel registered in third_party/torch-xpu-ops/src/ATen/native/xpu/WeightInt4Pack.cpp
returns a different-ranked tensor (XPU expects weight dtype kInt, CUDA expects kByte per skiplist
note), so the Inductor shape-inference/meta func disagrees with the actual runtime tensor. Triggered
only from test/test_linalg.py:7922 int4_mm compiled path in test_compile_int4_mm_m_*. Passes
eagerly; passes locally per yucai-intel, fails in CI env.

## Fix Approach

- Align the XPU _convert_weight_to_int4pack op with the CUDA contract: (1) accept kByte weight and produce the same 4-D packed layout (N//8, 1, K, 1) as CUDA in aten/src/ATen/native/cuda/int4mm.cu - modify the XPU implementation in third_party/torch-xpu-ops/src/ATen/native/xpu/WeightInt4Pack.cpp/.h and its meta function registration in torch/_meta_registrations.py (or xpu-specific meta) so the inductor-generated assert_size_stride matches. (2) Until then, keep tests skipped via skip_list_common.py entry 'TODO: align input data type for convert_weight_to_int4pack with CUDA'.
- Owner yucai-intel is already tracking.
