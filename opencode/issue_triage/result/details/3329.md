## Root Cause

launch_cumprod_kernel dispatches with scan<INCLUSIVE_TYPE, scalar_t, scalar_t> using scalar_t
(float) as the accumulator
(third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/CumprodKernel.cpp:23-32). Over 1M elements,
repeated float multiplication accumulates rounding error; CPU cumprod on float32 has the same
algorithmic bound, but reduction order and SIMD pairing differ enough on XPU to exceed atol=1e-4
(max_diff ~5.6e-3).

## Fix Approach

- Either accept the precision divergence as inherent float32 numerics (loosen test tolerance / document) or promote the scan accumulator to acc_type<float, kXPU>=float (no-op) — true fix is to use a double accumulator on XPU for float32 cumprod, similar to what reduction kernels do, then cast back.
- alternatively add a tree-reduction path to lower error growth.
