## Root Cause

torch.cumprod on a length-1,000,000 fp32 tensor near 1.0 diverges from CPU by max abs 5.6e-3.
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/CumprodKernel.cpp:23-32 dispatches via
AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND2 and calls scan<INCLUSIVE_TYPE, const scalar_t, scalar_t>(...)
with `std::multiplies<scalar_t>()` — i.e. the partial-product accumulator stays at fp32 (scalar_t)
inside ScanUtils.h block/grid scans rather than using an accscalar_t (fp64 for fp32 input). Across
1e6 successive multiplications fp32 rounding error compounds, producing the observed drift; CPU uses
a serial double-precision accumulator. Mirrors upstream pytorch/pytorch#180154.

## Fix Approach

- Switch the cumprod scan to use accscalar_t (double for fp32, float for half/bf16) for the internal accumulator, similar to CUDA's cumprod.
- Either change scan<> to take an explicit acc-type template parameter or perform the multiply in higher precision and cast back.
- Add a precision regression test for long fp32 cumprod in test/xpu and align with the upstream fix from #180154.
