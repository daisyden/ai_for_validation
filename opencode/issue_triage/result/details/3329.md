## Root Cause

torch.cumprod over 1M float32 elements diverges from CPU by ~5.6e-3 (above 1e-4 tol). Implementation
in third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/CumprodKernel.cpp uses parallel scan
(ScanUtils.h) which has different rounding-error accumulation order than CPU's serial scan.
Accumulated multiplicative round-off over 10^6 elements naturally exceeds 1e-4 in fp32; this is
inherent to parallel scan, not a kernel bug per se.

## Fix Approach

- Either (a) document/relax tolerance (parallel scan has O(log N) different rounding paths so 1e-3 accuracy at N=1M is expected), or (b) promote scan accumulator to float64 in CumprodKernel.cpp / ScanUtils.h when has_fp64 is available, mirroring CUDA's behavior.
- Confirm CUDA shows similar divergence at this N before treating as defect.
