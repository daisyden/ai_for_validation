## Root Cause

XPU matmul/conv family (matmul, mm, mv, bmm, addmm, addmv, addbmm, baddbmm, einsum, tensordot,
nn.functional.{conv*,conv_transpose*,bilinear,linear}) dispatches to oneDNN via
third_party/torch-xpu-ops/src/ATen/native/xpu/Blas.cpp and the oneDNN conv/mm path; oneDNN has no
int64 GEMM/conv primitive, so int64 inputs throw 'Long is not supported in oneDNN'. CUDA provides
native int64 matmul/conv, so the upstream OpInfos in
torch/testing/_internal/common_methods_invocations.py declare int64 as supported and the generic
OpInfo tests (test_noncontiguous_samples, test_multiple_devices, test_numpy_ref) exercise int64 on
XPU. No XPU-side integer kernel and no fallback decomposition is currently wired into the
Blas/Convolution registrations.

## Fix Approach

- Either (a) land upstream PyTorch PR #169353 (referenced in the issue) which aligns OpInfo dtypes to exclude int64 for XPU matmul/conv, or (b) add an integer-dtype fallback in torch-xpu-ops Blas.cpp / Convolution dispatch: before calling oneDNN, detect isIntegralType(self.scalar_type()) and decompose via CPU fallback or a simple SYCL GEMM kernel (analogous to bmm_complex_fallback at Blas.cpp:96-120).
- Preferred path is (a) to mirror CUDA-less claims consistent with oneDNN limitations.
