## Root Cause

TestNNDeviceTypeXPU.test_transformerencoderlayer_xpu_{float16,float32,float64} produces NaN/Inf
outputs because the XPU SDPA path reached from nn.TransformerEncoderLayer goes through
third_party/torch-xpu-ops/src/ATen/native/transformers/Attention.cpp
(_scaled_dot_product_efficient_attention_xpu at line 341 / math fallback at line 363) which did not
correctly propagate attention-mask dtypes/NaN-handling and did not dispatch for some
activation/batch_first combinations. Fixes landed in torch-xpu-ops PR #2336 (yucai-intel,
2025-11-11) plus the upstream device-enable change pytorch/pytorch#168234, but remaining
float16/float32/float64 variants still need verification.

## Fix Approach

- Rebase onto PR #2336 + pytorch/pytorch#168234 and re-run pytest -v test/test_nn.py::TestNNDeviceTypeXPU -k test_transformerencoderlayer.
- If any variant still produces NaN, instrument the SDPA math fallback to locate the first Inf (likely softmax over masked rows with -inf scores) and mirror the CUDA guard (row-wise all-masked -> zero output).
- Once green, remove the entries from skip_list_common.py and close.
