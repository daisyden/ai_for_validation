## Root Cause

In src/ATen/native/xpu/sycl/TriangularOpsKernels.cpp:46, linear_idx is computed as
item.get_global_id(0) * elements_per_thread under the default -fsycl-id-queries-fit-in-int compile
flag, which silently truncates the global id to uint32. For ranges > UINT32_MAX no exception is
raised (unlike a minimal repro) and results are wrong. The kernel uses a 1D parallel_for where the
total range can exceed 2^32 for large square matrices, producing wrap-around indexing.

## Fix Approach

- Either (a) cast get_global_id(0) to int64_t before the multiplication and use 64-bit IndexType when the total element count exceeds INT32_MAX, or (b) add -fno-sycl-id-queries-fit-in-int locally for this TU, or (c) use a 2D nd_range to keep each dimension within int32.
- Investigate why no diagnostic is emitted (possibly compiler bug — escalate to oneAPI).
- Mirror the int64 indexing pattern already used in other large-tensor kernels.
