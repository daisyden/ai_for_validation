## Root Cause

test_triton_bsr_softmax at third_party/torch-xpu-ops/test/xpu/test_sparse_csr_xpu.py:4832 calls
torch.sparse._triton_ops.bsr_softmax (defined at torch/sparse/_triton_ops.py:1915, kernel
_bsr_softmax_kernel at line 1827). The Triton kernel JITs to SPIR-V via
intel-xpu-backend-for-triton, but uses intrinsics / layout patterns that the Intel Triton backend
currently fails to lower; Level Zero then rejects the generated binary with
ZE_RESULT_ERROR_INVALID_KERNEL_NAME ("Error during Intel loadBinary") when the kernel symbol cannot
be found in the module. This is a triton-xpu compile-time codegen gap, not a bug in the sparse op
itself.

## Fix Approach

- File/track a pytorch-triton-xpu issue with the failing kernel reproducer and, once fixed upstream, bump the triton pin.
- In the meantime, gate bsr_softmax on XPU behind a device check that either (a) uses the non-Triton dense-softmax fallback path similar to the CUDA fallback in _triton_ops.py, or (b) skips the test with a clear xfail pointing to the triton issue.
- The Triton kernel itself (_bsr_softmax_kernel) may also need to be simplified (avoid unsupported tl primitives) to make it lower-able on XPU.
