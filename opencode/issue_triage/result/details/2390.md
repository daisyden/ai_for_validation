## Root Cause

PyTorch SDPA on XPU falls back to the math backend for NF4-quantized LoRA finetuning while IPEX uses
an XeTLA-optimized SDPA, causing ~10% extra GPU memory and slower training (1.16 vs 0.95 s/it).
PyTorch's training-path SDPA needs an oneDNN training kernel that has not yet landed (tracked in
MFDNN-14834, retargeted to PT 2.13). The dispatch logic lives in
third_party/torch-xpu-ops/src/ATen/native/transformers/Attention.cpp / sdp_utils.cpp where backend
selection chooses MATH because oneDNN training SDPA is unavailable.

## Fix Approach

- Wait for oneDNN v3.11+/MFDNN-14834 oneDNN SDPA training kernel.
- once available, register the oneDNN backward path in torch-xpu-ops Attention dispatch and update sdp_utils to prefer it over math.
- Validate memory/perf parity vs IPEX XeTLA on the olora_finetuning Llama-3.1-8B NF4 workload.
