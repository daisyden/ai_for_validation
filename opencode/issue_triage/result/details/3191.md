## Root Cause

At torch/_inductor/lowering.py:2514 make_fallback() asserts the op is not also in the decompositions
table unless override_decomp=True. 'aten::index_add' is in FALLBACK_ALLOW_LIST (lowering.py:110) and
is also registered via @register_decomposition([aten.index_add]) in
torch/_inductor/decomposition.py:267-282, which is only conditionally excluded from the main table
(decomps_to_exclude at decomposition.py:122). When XLNetLMHeadModel bf16 training triggers
make_fallback(aten.index_add.default) through graph.py:1314 with a get_decomp_fn that still contains
aten.index_add, the assertion fires. This is an upstream PyTorch Inductor fallback/decomp
bookkeeping regression exposed by HF/XLNet on XPU (bfloat16 code path that intentionally returns
NotImplemented to defer to fallback).

## Fix Approach

- In torch/_inductor/lowering.py, pass override_decomp=True when registering the fallback for aten.index_add (or remove the conditional decomp from the decomps table consumed by make_fallback when bfloat16 is the target).
- Alternatively, in decomposition.py ensure aten.index_add is removed from the decomp table the graph-level get_decomp_fn returns, since the decomposition returns NotImplemented for bf16 and relies on the fallback.
- File upstream PyTorch PR.
- torch-xpu-ops side only needs to bump pinned pytorch commit once fixed.
