## Root Cause

The xpu _weight_int4pack_mm implementation (torch-xpu-ops, via oneDNN matmul primitive) only
supports 2D activations and integer zero-points, while HuggingFace/TorchAO-saved CUDA int4
checkpoints use 4D packed weights with float zero-points. Per liangan1 (2025-08-07), oneDNN's int4
matmul kernel does not yet support float zp, which blocks dequantizing CUDA int4 layout on XPU.
Current status (Stonepia, 2026-04-01): waiting for oneDNN 3.11 to add float-zp support before
implementation can proceed.

## Fix Approach

- Track oneDNN 3.11 release for float zero-point support in int4 matmul.
- Once available, extend the xpu int4 op (src/ATen/native/xpu/Int4PackMM.cpp and associated oneDNN matmul wrapper) to: (1) accept 4D packed weight tensors matching CUDA's layout, (2) plumb float zp through the oneDNN attr post-ops, (3) add a layout-conversion path from CUDA int4 packing to the XPU/oneDNN expected packing.
- Add targeted tests loading a torchao int4-quantized HF model on xpu (see reporter's repro script).
