## Root Cause

_weight_int4pack_mm_xpu in third_party/torch-xpu-ops/src/ATen/native/xpu/LinearInt4.cpp:34
hard-checks A.dim()==2 and only supports plain int32 packed layout, not the CUDA tile_packed_to_4d
layout. The XPU oneDNN matmul kernel only supports integer zero-points, while the CUDA int4 layout
uses float zero-points, blocking direct dequant of CUDA-quantized HF checkpoints.

## Fix Approach

- Wait for oneDNN 3.11 float-zp support (per chuanqi129/Stonepia comments), then extend LinearInt4.cpp to accept 4D B tensor and float qScaleAndZeros.
- add a layout-conversion path or a dedicated dequant kernel.
- Short-term workaround: instruct users to quantize with int4_packing_format='plain_int32' for XPU.
