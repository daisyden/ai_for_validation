## Root Cause

In third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp the out-variants
_fft_c2c_mkl_out (L386-394), _fft_c2r_mkl_out (L466-474) and _fft_r2c_mkl_out (L555-564) call the
non-out _fft_*_mkl which allocates a fresh tensor, then perform out.copy_(result). This causes an
avoidable extra device allocation + copy per FFT call.

## Fix Approach

- Refactor _fft_c2c_mkl/_fft_c2r_mkl/_fft_r2c_mkl (and _exec_fft) to optionally accept a caller-provided output tensor and have oneMKL DFT compute_forward/compute_backward write directly into it when layout/contiguity permit.
- fall back to copy only when stride/dtype constraints are violated.
- Validate via FFT benchmarks and existing fft tests.
