## Root Cause

third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp:386-396 (_fft_c2c_mkl_out), :466
(_fft_c2r_mkl_out) and :555 (_fft_r2c_mkl_out) each call the non-out variant which allocates a fresh
result tensor via at::empty, then do at::native::resize_output(out, result.sizes()) +
out.copy_(result). oneMKL DFT compute_forward/compute_backward can write directly into a
caller-provided contiguous buffer, so this extra device allocation and full-tensor D2D copy is pure
overhead for every FFT out= call (fft/ifft/rfft/irfft/hfft/ihfft paths in
src/ATen/native/xpu/SpectralOps.cpp:47,82,118).

## Fix Approach

- Refactor _fft_{c2c,c2r,r2c}_mkl and their callee _exec_fft / _mkl_dft to accept an optional output Tensor& and, when the caller's 'out' is contiguous and has the right dtype/shape, pass it as the oneMKL output buffer instead of allocating a new one.
- Fall back to the current allocate+copy path only when out is non-contiguous, mismatched-dtype (e.g.
- ComplexHalf promotion at :381-382), or strided incompatibly with MKL's DFT layout.
- Benchmark on large multi-dim transforms to quantify the copy elimination and add numerics regression tests via the existing fft OpInfo sweep.
