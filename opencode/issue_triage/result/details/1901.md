## Root Cause

torch.linalg.svd falls back from XPU to CPU (RegisterXPU_0.cpp:54525) because no XPU kernel is
registered in torch-xpu-ops. Benchmarking in the issue shows the oneMKL GPU SVD is generally slower
than CPU fallback (speedup <1 for most shapes per CuiYifeng's table); oneMKL docs explicitly note
'even for a GPU-optimized SVD, some computations may be performed on the CPU.' The reported 'hang'
was actually NUMA contention when CPU fallback ran on both cards simultaneously.

## Fix Approach

- Option A: implement linalg_svd XPU kernel via oneMKL gesvd/gesvdj and gate it behind a size/shape heuristic to avoid regressions vs CPU path.
- Option B: keep CPU fallback but add NUMA-aware threading guidance and document the limitation.
- File a oneMKL performance request with the shape/speedup data already collected.
- Decide milestone (currently PT2.13) after MKL team response.
