## Root Cause

test_pdist_norm_large (torch-xpu-ops/test/xpu/test_torch_xpu.py:5186) invokes pdist on a 50000x1
tensor, producing ~1.25e9 output pairs (~5GB). The 'AssertionError: False is not true' is the
`self.assertTrue(torch.allclose(expected_cpu, actual_cpu))` at line 5195 firing due to a numerical
mismatch between CPU reference and the XPU PdistKernelFunctor at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/DistanceKernels.cpp:737. The kernel accumulates
in `scalar_t` (fp32) rather than `accscalar_t` (double) (line 753: `scalar_t agg = 0.0f`) and uses
`device_sqrt<double>` at line 746 for index math; accumulation round-off over stride-1 reductions of
>46k elements exceeds the default allclose tolerance.

## Fix Approach

- Switch pdist forward accumulation to `accscalar_t` (double) like CUDA's reference does, or widen tolerances via `assertTrue(torch.allclose(..., rtol=..., atol=...))`.
- Concretely, in PdistKernelFunctor change `agg` to `accscalar_t`, perform the group reduce in acc type, then cast to scalar_t before `out_ptr[k] = F::finish(...)`.
- Re-run with PYTORCH_TEST_WITH_SLOW=1 on a >=6GB XPU to confirm.
