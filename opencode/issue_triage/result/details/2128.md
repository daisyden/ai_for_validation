## Root Cause

The 0xC0000005 access violation top frame is inside a JIT-compiled Triton kernel .pyd under
%TEMP%\tmp...\rg\crgm2qh5ndl2zbeae36cqmzemw7snxesg2clkx3le633lxfmxdnh.main.pyd+0x161d, invoked by
the autograd engine (torch_cpu!torch::autograd::Engine::evaluate_function) during bf16 training of
speech_transformer. The crash happens right after 'cudagraph partition into 2 partitions', meaning
Inductor split the graph and the Triton-XPU backend emitted a kernel that memory-faults on
BMG-Windows. Root cause is in intel-xpu-backend-for-triton codegen (wrong address/bounds in a
partitioned bf16 kernel on Windows); the torchbench-2.8 pin just exposes an input shape combination
that triggers it.

## Fix Approach

- Reduce to a minimal AOTI/Inductor repro by capturing the offending partitioned subgraph via TORCH_COMPILE_DEBUG=1 and TORCHINDUCTOR_CACHE_DIR, then inspect the generated Triton source (.main.py) at the crash RVA.
- File against intel-xpu-backend-for-triton with the kernel and inputs.
- Workaround: disable cudagraph partitioning (torch._inductor.config.triton.cudagraphs=False) or force eager for this model until the Triton backend fix lands.
