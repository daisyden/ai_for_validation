## Root Cause

test_comprehensive_stft_xpu_float16 in test/inductor/test_torchinductor_opinfo.py shows ~11% of
elements exceeding the fp16 tolerance (max abs 2.4e-4 vs 1e-5 allowed, max rel 4.6e-3 vs 1e-3
allowed) for stft with n_fft=10. The XPU stft path performs the FFT/twiddle accumulation in fp16
instead of promoting to fp32 like CPU/CUDA, accumulating roundoff error that exceeds OpInfo
tolerances. Implementation lives in torch-xpu-ops SYCL spectral ops.

## Fix Approach

- Either (a) promote the accumulator dtype to float32 inside the XPU stft kernel (cast input to fp32, run rfft/twiddle multiply, cast output back) matching CPU/CUDA behavior, or (b) loosen the OpInfo tolerance for stft fp16 in common_methods_invocations / test xpu allowlist.
- Option (a) preferred for numerical parity.
