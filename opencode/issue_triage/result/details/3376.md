## Root Cause

Newly enabled NCCL-derived tests in test/distributed/test_c10d_xccl.py exercise APIs that
ProcessGroupXCCL does not expose: _is_initialized, comm_split_count, _verify_work_timeout,
_set_enable_nan_check, _set_default_timeout, split_group support, _dump_xccl_trace_json,
torch.xpu.xccl.version, _wait_for_pending_works, and a dump-pipe path. None of these are bound on
the XCCL backend (third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.hpp/cpp lacks the
corresponding members; only an unimplemented abort() exists at hpp:107) and there is no
torch.xpu.xccl Python module mirroring torch.cuda.nccl. These are pure feature gaps tracking parity
with ProcessGroupNCCL (related: issue #2659).

## Fix Approach

- Add the missing XCCL backend methods and pybind registrations: _is_initialized, comm_split_count, _verify_work_timeout, _set_enable_nan_check, _set_default_timeout, _dump_xccl_trace_json, _wait_for_pending_works in ProcessGroupXCCL (cpp/hpp) plus its python bindings under torch/csrc/distributed/c10d.
- implement comm split (eager + lazy) so c10d.split_group works for the xccl backend.
- create torch.xpu.xccl Python module exposing version().
- implement dump-pipe handler.
- Until landed, mark these tests as expected feature gaps (skip with reason) in torch-xpu-ops skip list.
