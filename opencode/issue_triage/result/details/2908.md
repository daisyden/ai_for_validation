## Root Cause

Five E2E accuracy regressions (DistillGPT2, AllenaiLongformerBase, whisper-tiny bf16 inference;
pit_b_224 fp16 inference; timm_regnet bf16 training) versus PT2.10. Debug by xuhancn on 2026-02-27
with driver 8531 showed 3 of 5 now pass (DistillGPT2, AllenaiLongformerBase, whisper-tiny) – those
regressions were due to driver 8331; pit_b_224 still shows RMSE 0.111 vs ref 0.00047 (fp16 amp
inference) on BMG Windows/Linux and PVC, and timm_regnet still fails on Windows only (bfloat16
training, RMSE 0.0156 vs 0.0039). Failures come from accumulated fp16/bf16 numeric drift in the
Inductor-compiled graphs, likely in a conv+attention fusion (pit_b_224 uses ViT-style attention) and
grouped-conv/batchnorm kernels (timm_regnet).

## Fix Approach

- Close the 3 driver-fixed models once driver >=8531 is the minimum for 2.11 release notes.
- For pit_b_224: bisect torch-xpu-ops main between PT2.10 and PT2.11 tags to locate the regressing kernel (suspect SDPA/attention fp16 path in third_party/torch-xpu-ops/src/ATen/native/xpu/mkldnn/SDP.cpp or the conv2d+ReLU oneDNN fusion), or increase Inductor accuracy tolerance for amp-fp16 via use_larger_multiplier_for_smaller_tensor.
- For timm_regnet Windows-only: check BN/GroupNorm bf16 reduction kernels under src/ATen/native/xpu/sycl/Norm*.cpp for Windows compiler codegen differences.
- consider disabling fusion or using fp32 reductions on Windows.
- Add both to E2E xfail list until fixed and raise driver-team ticket for the shared accuracy baseline.
