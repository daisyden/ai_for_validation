## action_reason

- test/test_binary_ufuncs.py:2746 has @dtypesIfXPU(*set(integral_types()) - {torch.int64}), explicitly excluding int64 from XPU variants.

## Root Cause

test_fmod_remainder_by_zero_integral_xpu_int64 fails because the XPU integer fmod/remainder kernels
in third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/BinaryRemainderKernel.cpp perform raw `a % b`
(FmodIntegralFunctor::operator() line 47, RemainderIntegralFunctor line 24) with no guard for b==0;
on Intel GPU this yields implementation-defined values instead of the UINT64_MAX sentinel
(4294967295 in the test) that CUDA produces and that test_binary_ufuncs.py:2763 asserts.
test_cudnn_weight_format is a separate miscompare driven by cuDNN-specific weight layout assumptions
not applicable to XPU.

## Fix Approach

- In BinaryRemainderKernel.cpp, update FmodIntegralFunctor and RemainderIntegralFunctor to match the CUDA reference: when `b == 0` return the same sentinel used by CUDA (static_cast<scalar_t>(-1) for signed / ~scalar_t(0) for unsigned) so the `== 4294967295` / `x >= 0` assertion passes for all integral dtypes.
- keep the float path `__ubsan_ignore_float_divide_by_zero__` unchanged.
- For test_cudnn_weight_format, either skip on XPU (no cuDNN semantics) or adapt the XPU RNN weight-format check in torch-xpu-ops to expose the `contiguous()` diagnostic the test expects.
- Then remove the WAs introduced by pytorch PR #171238.
