## Root Cause

torch.fft.rfft on an XPU bfloat16 tensor raises 'Unsupported dtype BFloat16' instead of promoting to
float32. The XPU FFT path is implemented via oneMKL-DFT and the helper promote_fft_input in
third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp:323-329 only promotes Half→Float
and ComplexHalf→ComplexFloat; BFloat16 is not handled, so it falls through to oneMKL DFT which
rejects bf16. Upstream pytorch/pytorch#180766 documents the expected promotion behavior.

## Fix Approach

- Extend promote_fft_input in src/ATen/native/xpu/mkl/SpectralOps.cpp to also promote BFloat16→Float (mirror the Half branch), and apply it consistently in _fft_c2c_mkl, _fft_c2r_mkl, _fft_r2c_mkl entry points (already calling promote_fft_input).
- Add an XPU-enabled test for torch.fft.rfft/torch.fft.fft with dtype=bfloat16 in test/xpu/test_spectral_ops_xpu.py asserting result dtype is complex64.
- Coordinate with upstream PR #180766.
