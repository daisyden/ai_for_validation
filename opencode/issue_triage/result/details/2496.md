## Root Cause

When the full test_torch_xpu.py suite runs serially, execution reaches the test_index_add* tests and
the GPU aborts with a NEO page fault ('Segmentation fault from GPU ... access: Write, banned: 1' at
drm_neo.cpp:269). Each test passes in isolation, and test classes pass in isolation, which points to
state leakage across tests: the index_add_kernel in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/Indexing.cpp:368 (launched via
AT_DISPATCH_INDEX_TYPES at line 433) likely retains or dereferences stale device pointers /
out-of-range offsets when prior tests left the XPU allocator/caching state in a particular
configuration, producing an invalid write on a later launch.

## Fix Approach

- Reproduce locally with pytest --forked and compute-runtime debug (NEO_ENABLE_DEVICE_ENQUEUE / ZE_DEBUG) to obtain a kernel-level fault address, then audit index bounds and source/destination stride computation in index_add_kernel (Indexing.cpp:368-460).
- Add explicit TORCH_CHECKs on index.max() < self.size(dim) and on slice-size computation.
- ensure the kernel does not read past src when index has zero numel.
- Short-term, bisect by running index_add tests with a minimized preceding test subset to identify the polluting test and add an empty_cache/synchronize guard.
