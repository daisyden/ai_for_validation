## Root Cause

test_cow_input_* (TestCompositeComplianceXPU) verifies that ops honoring
supports_cow_input_no_materialize_{forward,backward}=True do not materialize copy-on-write inputs.
On XPU almost every linear-algebra / GEMM op materializes because the oneDNN/oneMKL wrappers in
third_party/torch-xpu-ops/src/ATen/native/xpu/Blas.cpp and
src/ATen/native/xpu/mkl/BatchLinearAlgebra.cpp call .contiguous() / copy tensors into oneDNN memory
format, which triggers COW materialization. Additionally histogramdd has no XPU kernel
('aten::_histogramdd_bin_edges not implemented') and jiterator tests are CUDA-only — those are
environmental. The bulk (matmul / addmm / bmm / cholesky / linalg_* / conv*) stems from
oneDNN/oneMKL layout conversion materializing inputs.

## Fix Approach

- Two-part fix: (1) For ops where materialization is fundamentally required by oneDNN/oneMKL layout conversion (conv*, GEMM-family, cholesky*, linalg_*), upstream-set allow_cow_input_materialize_{forward,backward} on the corresponding OpInfos in torch/testing/_internal/common_methods_invocations.py (and linalg opinfo definitions) to match reality on XPU.
- the CUDA OpInfos already carry these flags for cuBLAS/cuSOLVER materialization and XPU should inherit via dtypesIfXPU/supports flags. (2) Implement aten::_histogramdd_bin_edges natively on XPU or add to fallback list.
- skip jiterator tests on XPU (device-specific).
- No SYCL kernel correctness changes needed.
