## Root Cause

test/profiler/test_profiler.py::TestProfiler::test_basic_chrome_trace asserts every traceEvent's
ts+dur <= captureWindowEndTime_, but the synthetic '__xpu_profiler__' Span entry violates this. In
libkineto's CuptiActivityProfiler::stopTraceInternal, captureWindowEndTime_ is set from `now` BEFORE
session->stop() is called on XpuptiActivityProfilerSession, so the XPU span's stop timestamp is
always slightly later than the recorded window end (analysis in issue comment by PawelSwider2000).
The bug is therefore in upstream pytorch/kineto's stop ordering, not in a torch-xpu-ops SYCL kernel.

## Fix Approach

- Land upstream PR pytorch/pytorch#172219 (currently awaiting Meta review) which patches CuptiActivityProfiler / kineto so the XPU profiler session's stop() accepts captureWindowEndTime_ and adds float-tolerance in the test assertion.
- Alternative interim: as done in torch-xpu-ops issue #3479, port the test locally and special-case the '__xpu_profiler__' Span entry in _validate_basic_json — but the upstream kineto fix is the durable solution.
- No torch-xpu-ops src changes required.
