## Root Cause

test/inductor/test_compiled_optimizers.py test_bitwise_*_capturable_foreach_xpu requires
bitwise-exact parity between eager and Inductor/Triton. Upstream PR 174058 introduced
torch._inductor.config.eager_numerics.use_pytorch_libdevice (see torch/_inductor/config.py:2719 and
torch/_inductor/runtime/compile_tasks.py:58-62) as the CUDA alignment knob that forces Triton to use
the CUDA-toolkit libdevice.bc so Triton pow matches eager sycl::pow. XPU has no analogous hook:
knobs.intel.libdevice_path is a no-op because Intel Triton already bundles its own .bc with
__imf_powf, which differs from sycl::pow by ~1 ULP on pow(0.9,1.0), so AdamW's pow(beta,step)
diverges bitwise. The KeyError surfaces because older torch installs lack the config attribute
entirely.

## Fix Approach

- Confirmed not_target label: the assignee has decided to permanently skip these tests on XPU.
- Add the four test_bitwise_adam*_capturable_foreach_xpu tests to skip_list_common.py (inductor section) with a comment pointing to the 1-ULP pow mismatch between Intel Triton libdevice and sycl::pow.
- Alternatively, request an upstream option in Intel Triton that overrides the bundled libdevice with a PyTorch-supplied .bc (analogous to use_pytorch_libdevice on CUDA) and wire it through knobs.intel.libdevice_path.
- without that, bitwise parity is impossible.
