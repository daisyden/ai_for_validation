## Root Cause

After upgrading Triton 3.6.0 -> 3.7.0 the Inductor flex_attention triton template switched kernel
grid ordering from {8;4;1} to {1;8;4} and the new Triton backend increased per-thread register spill
from 3072 to 11392 bytes on BMG/B580 (reported in intel/intel-xpu-backend-for-triton#6625). This
raises per-call flex_attention decode time from ~15.8us to ~24.2us (+53%), causing the ~5%/~10%
end-to-end regression for INT4-RTN WoQ Qwen/Llama runs. Pressure is hardware-specific (B580 GRF=256)
which is why PVC/B60 do not regress. Template source: torch/_inductor/kernel/flex_attention.py
(flex_attention_template / autotune configs).

## Fix Approach

- Two-pronged: (1) file/track triton XPU backend register-spill regression (already filed as intel-xpu-backend-for-triton#6625) so codegen restores spill levels. (2) In torch/_inductor/kernel/flex_attention.py add an XPU/BMG-aware autotune config that pins grid/block ordering back to (num_warps=4, BLOCK_M=8, BLOCK_N=4, ...) equivalent of the old {8;4;1} layout, or gate the new config behind a property check (device GRF/EU count).
- Validate via the flex_attention_benchmark_causal_mask.py script and the Qwen3-4B flex_attention run.
