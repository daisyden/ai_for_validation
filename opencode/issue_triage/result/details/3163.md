## Root Cause

test_sparse_add at third_party/torch-xpu-ops/test/xpu/test_sparse_csr_xpu.py:3277 asserts that the
output's crow_indices/col_indices dtype matches the input index_dtype (int32 and int64 both tested).
The XPU kernel add_out_sparse_compressed_xpu in
third_party/torch-xpu-ops/src/ATen/native/sparse/xpu/SparseCsrTensorMath.cpp:388-427 takes a
to_dense -> add -> to_sparse_csr() path (line 423-424). to_sparse_csr() defaults to int64 indices
regardless of the input tensors' index dtype, so outputs always come back as int64, failing the
int32 assertion (torch.int64 != torch.int32) for float32/float64/complex64/complex128.

## Fix Approach

- In add_out_sparse_compressed_xpu, after constructing out_dense.to_sparse_csr(), cast crow_indices and col_indices to the index dtype of the inputs (self.crow_indices().scalar_type()) before assigning to out.
- Better: implement a native CSR+CSR add that preserves index dtype (mirroring the CPU impl in aten/src/ATen/native/sparse/SparseCsrTensorMath.cpp) instead of the dense round-trip, which is also inefficient.
- Keep the dense fallback only for CSR+strided mixed inputs.
