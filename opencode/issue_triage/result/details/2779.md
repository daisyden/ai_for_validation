## Root Cause

src/ATen/native/xpu/sycl/RangeFactoriesKernel.cpp:297-309 implements logspace_kernel for integer
dtypes by computing scalar_base, step and pow() in float32 (see line 299-303: 'float scalar_base =
static_cast<float>(base); ... float step = static_cast<float>(end-start)/(steps-1);'). For
int16/int32/int64 operands CUDA/CPU promote to double; the float32 intermediate loses precision and
after truncation to integer a single element rounds off by 1 (one-off mismatch), which is exactly
the 'one-off error ... for some inputs' failure reported in
test_comprehensive_logspace_*_xpu_int{16,32,64} in test_decomp_xpu.

## Fix Approach

- Change the step_type used for integer dispatch in logspace_kernel from float to double (or long double): in RangeFactoriesKernel.cpp line 299-306 instantiate LogspaceFunctor<scalar_t, double> and compute scalar_base/step in double, matching CUDA's reference decomposition.
- If fp64 is unavailable on the target device, fall back via a host-side computation of the pow table.
- Re-enable the listed test_comprehensive_logspace / test_quick_logspace int cases once precision matches.
