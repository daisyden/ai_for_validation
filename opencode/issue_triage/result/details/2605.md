## Root Cause

Feature request: PyTorch PR #170341 added an inductor freezing pattern-matcher that fuses 3 separate
int4 matmuls (Q/K/V projections) into one horizontal int4 mm (and similarly for gate/up in MLP) on
CUDA. XPU's int4 op (aten::_weight_int4pack_mm_xpu in
third_party/torch-xpu-ops/src/ATen/native/xpu/LinearInt4.cpp:21) has a different packed-weight
layout and signature than CUDA tinygemm, so CUDA's fusion pattern (registered in
torch/_inductor/fx_passes/*) does not match XPU graphs; horizontal QKV fusion is therefore missing
for XPU int4 WOQ, leaving per-projection overhead.

## Fix Approach

- Mirror PR #170341 for XPU: register an analogous freezing pattern in torch/_inductor/fx_passes (e.g. a new xpu variant alongside the cuda one) that matches three _weight_int4pack_mm calls sharing input A and concatenable weights/scales, rewriting to a single fused int4 mm followed by a split.
- Ensure XPU's weight-packed layout (int4x2/int4x8 per WeightInt4PackKernel.cpp) supports dim-0 concatenation of packed weights + qScaleAndZeros.
- if not, add a helper to concat pre-pack or support post-pack concat.
- Gate the pattern behind `is_xpu` and add a unit test in test/inductor/test_xpu_basic.py.
