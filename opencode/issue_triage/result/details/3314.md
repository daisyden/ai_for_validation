## Root Cause

On Windows, test_xpu.py::TestMemPool.test_mempool_id (test/test_xpu.py:3051) aborts with
std::terminate originating in sycl8.dll (CxxThrowException -> sycl::queue) during MemPool
destruction. The Windows stack shows c10_xpu.dll!MemPool::~MemPool ->
XPUCachingAllocator::releasePool -> syncStreamsOnDevice rethrowing a SYCL exception
(c10/xpu/XPUCachingAllocator.cpp:2148-2151), which is uncaught at the dtor boundary and aborts the
process. The underlying SYCL queue sync failure is a Windows-only SYCL runtime/driver-layer behavior
(sycl8.dll: ext_codeplay_supports_fusion frame) when the second MemPool is destroyed while streams
are still live.

## Fix Approach

- Wrap MemPool::~MemPool body (releasePool + emptyCache) in a try/catch that logs and swallows c10::Error/sycl::exception so destruction never throws.
- mirror the noexcept guard CUDA uses.
- In parallel, file a SYCL/L0 driver bug for the queue-sync exception on Windows (sycl8.dll path), and add the test to Windows skip list (run_test_win_with_skip_mtl.py) as a temporary mitigation.
