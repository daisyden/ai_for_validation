## Root Cause

Build system does not treat compiler warnings as errors, so newly introduced warnings (e.g., those
surfacing in PR intel/torch-xpu-ops#2561 after a warning clean-up) can regress silently.
third_party/torch-xpu-ops/cmake/BuildFlags.cmake (the central place compiler flags are set for
SYCL/host code) lacks -Werror, and CMakeLists.txt does not propagate it to the xpu_ops target. Only
narrow cases in cmake/SYCLTLA.cmake:12-15 strip specific -Werror=... flags rather than enabling the
full set.

## Fix Approach

- In third_party/torch-xpu-ops/cmake/BuildFlags.cmake add `-Werror` (or at minimum `-Werror=return-type -Werror=unused-result -Werror=reorder -Werror=sign-compare`) to XPU_HOST_CXX_FLAGS and the SYCL device flag list, and ensure the flag reaches both nvcc-style device compile and host compile via target_compile_options on torch_xpu_ops.
- Gate behind a CMake option USE_XPU_WERROR=ON by default in CI so local builds can still opt out.
- Validate by intentionally introducing a warning and confirming CI fails.
