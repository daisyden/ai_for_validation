## Root Cause

bsr_dense_addmm Triton kernel emits SPIR-V that references the extension
'SPV_INTEL_subgroup_matrix_multiply_accumulate' (subgroup DPAS/MMA intrinsic), but the installed
pytorch-triton-xpu / intel-graphics-compiler pair in CI rejects it with 'InvalidModule: Invalid
SPIR-V module: input SPIR-V module uses unknown extension'. The Triton XPU backend generates this
extension when lowering tl.dot for block sizes 16/32/16x32 on bf16/fp16/int8, and the runtime's IGC
does not advertise support. This aborts pytest for all remaining bsr_dense_addmm tests.

## Fix Approach

- Dependency bug: bump/pin intel-graphics-compiler or upgrade triton-xpu to a build whose SPIR-V consumer accepts SPV_INTEL_subgroup_matrix_multiply_accumulate.
- Meanwhile add the listed bsr_dense_addmm_blocksize_{16,32,16x32} cases to test/xpu/skip_list_common.py with a 'Triton SPIR-V ext' reason, and gate the tl.dot codepath in torch/sparse/_triton_ops_meta.py on a runtime extension-capability probe to avoid interrupting the pytest process.
