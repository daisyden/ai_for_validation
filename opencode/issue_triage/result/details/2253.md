## Root Cause

test_dtypes_* compares the OpInfo's declared dtypesIfXPU against the set of dtypes the op actually
dispatches successfully. For addmm/addmv/baddbmm/mv/conv*, XPU dispatches through oneDNN which lacks
some dtypes CUDA supports (e.g. int64 for GEMM, certain low-precision/complex combos for conv).
histogram/histogramdd have no XPU kernel and fall back to CPU with a different dtype set. The
declaration site is torch/testing/_internal/opinfo/definitions (and common_methods_invocations.py)
dtypesIfXPU; XPU registration is in third_party/torch-xpu-ops/yaml/native/native_functions.yaml and
src/ATen/native/xpu/Blas.cpp. Duplicate of #2289; mostly fixed by pytorch PR #161246 (commit
6874f0a).

## Fix Approach

- Update dtypesIfXPU in upstream opinfo definitions for each failing op to match the dtype set actually supported by the XPU/oneDNN backend (drop int64 for GEMMs, drop complex/low-precision for conv where unsupported, set histogram/histogramdd to the CPU-fallback dtype set).
- This is the remaining portion of PR #161246.
- No new kernel work required.
- validate via test_ops_xpu.py::TestCommonXPU::test_dtypes_* after merge.
