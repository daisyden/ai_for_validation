## Root Cause

When submitting many kernels while recording timing events, the host stalls. Level-Zero event-pool
default size is 256; torch.xpu.Event(enable_timing=True) allocates a new ze_event from the pool on
each .record(), and once the pool is exhausted the driver blocks waiting for events to be released.
This is a Level-Zero/driver-level limitation (tracked as GSD-12059 by @jianyizh) reproduced by
creating 20000 timing events and submitting compiled kernels between record() pairs; PTI collection
amplifies the pressure. Relevant code is XPUEvent::record in pytorch/c10/xpu/XPUEvent.h and the
event construction path used by torch.xpu.Event.

## Fix Approach

- Primary fix is on the Level-Zero driver (GSD-12059) to enlarge/grow the event pool or recycle completed events.
- In the torch-xpu-ops/PyTorch layer, mitigate by lazily growing the internal ze_event_pool (allocate additional pools of 256 when exhausted) in XPUEvent creation, and add a documented environment knob (e.g.
- PYTORCH_XPU_EVENT_POOL_SIZE) plus a fall-back that flushes/recycles completed events.
- Until the driver fix, add a user-facing warning when many timing events are outstanding.
