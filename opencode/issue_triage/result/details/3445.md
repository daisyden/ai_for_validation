## action_reason

* Upstream pytorch/pytorch#77881 already added the CPU-side bounds/empty-source checks long ago, but the XPU index_select kernel in torch-xpu-ops still lacks the equivalent input.size(dim)>0 / index<self.size(dim) validation. No xpu-side PR exists yet (V0/VA empty, no VC hit), so the issue needs an owner to author the kernel-side fix in src/ATen/native/xpu/sycl/Indexing.h and IndexingKernels.cpp.

## Root Cause

torch.index_select on an empty source tensor diverges from CPU: with self of shape (0,3) and dim=0,
XPU raises 'Indexing kernel backbone does not support null src ...' from
torch-xpu-ops/src/ATen/native/xpu/sycl/Indexing.h:136-138 instead of CPU's 'index axis dim should be
positive'; with dim=1, XPU silently accepts an out-of-bounds index (5 vs axis_dim=3) and returns
shape (0,2) because the empty-output fast path skips bounds validation. CPU performs the upstream
bounds/empty-source checks (added by pytorch/pytorch#77881) before dispatch, but the XPU
index_select kernel's pre-validation gate is missing the equivalent input.size(dim)>0 /
index<self.size(dim) checks.

## Fix Approach

- In torch-xpu-ops Indexing kernels (src/ATen/native/xpu/sycl/Indexing.h around line 136 and the index_select dispatch in IndexingKernels.cpp), add upstream-aligned validation prior to the null-src check: TORCH_CHECK(self.numel() != 0 || index.numel() == 0, 'index_select(): self indexing axis dim should be positive') and bounds-check indices against self.size(dim) even when output is empty.
- Mirror the logic from aten/src/ATen/native/TensorAdvancedIndexing.cpp's index_select_out_cpu_.
- Add tests for (0,3)+dim=0 and (0,3)+dim=1 OOB cases under test/xpu/test_indexing_xpu.py.
