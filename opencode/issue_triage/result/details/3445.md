## Root Cause

torch.index_select on XPU diverges from CPU for empty source tensors: (a) shape (0,3) dim=0 raises
'Indexing kernel backbone does not support null src ...'
(third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/Indexing.h:138) instead of CPU's 'self indexing
axis dim should be positive', and (b) shape (0,3) dim=1 with out-of-bounds indices [0,5] silently
returns (0,2) because the early `dst_num_elem==0` return at sycl/Indexing.cpp:197 skips bounds
checking entirely. The XPU entry index_select_out_xpu (xpu/Indexing.cpp:31) lacks the CPU-style
validation that upstream PR pytorch/pytorch#77881 added.

## Fix Approach

- In src/ATen/native/xpu/sycl/Indexing.cpp index_select kernel: add explicit TORCH_CHECK that self.size(dim) > 0 with the CPU error message (matching upstream pytorch/pytorch#77881), and validate index bounds against src.size(dim) BEFORE the `dst_num_elem == 0` short-circuit at line 197 so empty-output cases still error on out-of-range indices.
- Replace the 'null src' message in Indexing.h:138 with the canonical CPU error string.
