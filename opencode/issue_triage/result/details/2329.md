## Root Cause

test/inductor/test_inductor_scheduler.py::test_flop_counter_op_options0_xpu_float16 originally hit
an Invalid SPIR-V extension (SPV_INTEL_subgroup_matrix_multiply_accumulate) from triton; that was
fixed in triton 3.6 / PR #166436. The residual failure is a feature gap: torch._inductor utilities
get_device_tflops() and get_dram_gbps() (torch/_inductor/utils.py, used by the flop-counter
scheduler test) are implemented only for CUDA via torch._utils/_internal estimates, with no XPU
branch, so the test cannot compute peak TFLOPS on PVC. Tracking dependency is
intel/intel-xpu-backend-for-triton#5792 which needs to expose the device peak-compute/bandwidth info
from the XPU runtime/triton backend.

## Fix Approach

- Add XPU support in torch/_inductor/utils.py get_device_tflops() and get_dram_gbps(): detect device.type=='xpu' and compute peak TFLOPS from torch.xpu.get_device_properties (EU count * SIMD width * frequency * ops/cycle per dtype) and DRAM BW from hbm_bandwidth / memory_bus_width once exposed by intel-xpu-backend-for-triton#5792 or a static PVC/BMG table as an interim.
- Route the flop-counter scheduler test through torch.accelerator instead of torch.cuda hardcoding, then re-enable test_flop_counter_op_options0_xpu_float16.
