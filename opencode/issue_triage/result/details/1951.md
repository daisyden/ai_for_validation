## Root Cause

Heterogeneous failures under TestCommonXPU.test_out: (1) _native_batch_norm_legit/native_batch_norm
hit TORCH_INTERNAL_ASSERT at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/BatchNormKernels.cpp:585 (out_invstd
contiguity/size check) because the test_out harness passes empty/mismatched out tensors and the
kernel doesn't resize/re-check properly (duplicated to #2233). (2) addmv/mv report stride mismatch
(expected stride (2,), got (1,)) — the xpu gemv out= path (oneMKL BLAS) doesn't preserve the
provided non-contiguous out strides. (3) triangular_solve/cholesky_inverse/geqrf/ormqr/narrow_copy
fail as "Unexpected success" or similar out= semantics issues; histc duplicates #2234. The common
thread is that xpu op implementations don't fully honor the out= resize/stride/dtype contract
exercised by test_out.

## Fix Approach

- Split per-op: (a) In BatchNormKernels.cpp:~585 call at::native::resize_output on out_invstd/out_mean before asserting contiguity, or reshape/clone as CUDA does (see CUDA Normalization.cu path). (b) For addmv/mv at src/ATen/native/xpu/Blas.cpp, mirror CUDA's handling that preserves output strides when out= is provided (use resize_output_check + copy_ if strides differ). (c) For triangular_solve/cholesky_inverse/geqrf/ormqr, check the xpu linalg wrappers (or CPU fallback registrations) for out= compliance.
- many can be fixed by aligning with ATen's structured-kernel out= contract.
- Track histc/batch_norm_legit via the existing dup issues #2233/#2234.
