## Root Cause

On release/2.10 (Max 1100) a broad set of HF / timm / TB training workloads (hf_T5*, hf_Bert*,
Roberta, Megatron, LayoutLM, BertForMaskedLM, nfnet_l0, cait_m36_384, nvidia_deeprecommender,
cm3leon_generate ...) regressed 10-30% vs 2.9. Triage already identified multiple upstream Inductor
changes as the causes: bf16 atomic_add fallback (fixed by pytorch PR #169257), reduction-heuristics
change, online-softmax enablement, and fusion change in pytorch commit 6a2bd1f4 that split a single
`triton_red_fused__softmax_...` kernel into a red+poi pair for cait_m36_384 (weishi-deng analysis:
'fusion failed with memory no shared data'). cait_m36_384 is additionally tracked by
intel-xpu-backend-for-triton#5750. No torch-xpu-ops SYCL kernel is implicated.

## Fix Approach

- Cherry-pick pytorch PR #169257 (bf16 atomic_add fallback) and the reduction-heuristics fix onto release/2.10.
- for the cait_m36_384 fusion regression either revert the behaviour of pytorch@6a2bd1f4 on XPU or teach Inductor's fusion scheduler to keep the red+poi softmax pair fused when the consumer has no shared memory with the producer.
- Gate TORCHINDUCTOR_ONLINE_SOFTMAX off on XPU until the Triton XPU backend catches up (track intel-xpu-backend-for-triton#5750).
- Re-run OOB 2.11 vs 2.9 (as chuanqi129 requested) to confirm the remaining per-model gaps and close sub-issues individually (cm3leon tracked in #2223).
