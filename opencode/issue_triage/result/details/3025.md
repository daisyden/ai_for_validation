## Root Cause

HasDecompTest.test_has_decomposition compares the sorted set of ATen ops expected to have
decompositions to an expectedInline string stored under third_party/torch-xpu-ops/test/xpu/expect/.
After an upstream PyTorch bump new ops (e.g. aten::_flash_attention_forward_no_dropout_inplace, new
aten::_foreach_* overloads) are reported by
torch._decomp.core_aten_decompositions()/has_decomposition table but the checked-in expected file
still has the older placeholder (aten::_foobar...), so assertExpected in test_decomp_xpu.py:1342
raises AssertionError. No real decomposition regression, just a stale golden.

## Fix Approach

- Regenerate the expected list with EXPECTTEST_ACCEPT=1 python test/xpu/test_decomp_xpu.py HasDecompTest.test_has_decomposition (and test_aten_core_operators) to refresh the expect file under third_party/torch-xpu-ops/test/xpu/expect/.
- Commit the updated golden and add the two cases to the nightly allowlist until the refresh lands, or make the test robust by diffing against the live torch._decomp table with a known-missing allowlist instead of a frozen string.
