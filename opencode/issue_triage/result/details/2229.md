## Root Cause

test_invalid_input_Sparse{CSR,CSC,BSR,BSC}_target_* expects the regex 'Expected all tensors to be on
the same device...' but XPU produces 'device of crow_indices (=xpu:0) must match device of
col_indices (=xpu:1)'. Per @jenniew's analysis, aten::_validate_compressed_sparse_indices is not
implemented on XPU and falls back to CPU via xpu_fallback_impl, where check_device_consistency
raises a different message than the native CPU/CUDA path used to produce. The XPU rows have already
been auto-marked passed; only the CPU-host variants remain, hitting the same fallback path because
tensors get split across xpu:0 and cpu.

## Fix Approach

- Implement aten::_validate_compressed_sparse_indices for the XPU backend in torch-xpu-ops so the canonical error message is generated, or relax the test's assertRaisesRegex to accept the fallback message.
