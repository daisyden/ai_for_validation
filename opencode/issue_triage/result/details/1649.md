## Root Cause

When the oneAPI used to build an XPU cpp extension (sourced via setvars.sh) differs from the oneAPI
pinned in the installed torch wheel, users hit opaque loader errors like `libur_loader.so.0: version
LIBUR_LOADER_0.10 not found` or missing libpti_view.so.0.10 originating from
torch/utils/cpp_extension.py's SYCL build path. dvrogozh itemised the scenarios (match / B>A
backward / A<B forward compat), and the root library mismatch check is absent in
torch/utils/cpp_extension.py (SyclExtension) and torch/__init__.py's XPU import sequence.

## Fix Approach

- In torch/utils/cpp_extension.py SyclExtension helpers, record the oneAPI/ICX version used at wheel build time (e.g. in torch/version.py or a generated header) and compare against $ONEAPI_ROOT or `icpx --version` at extension build time, emitting an actionable error naming conflicting components (compiler, pti, umf, ur_loader).
- Also wrap the torch._C import to catch LIBUR_LOADER/libpti symbol errors and rewrite the message pointing to the matching oneAPI version.
- Escalated to oneAPI via PREQS-2946 for backward-compat.
- PyTorch side should still gate with a clear diagnostic.
