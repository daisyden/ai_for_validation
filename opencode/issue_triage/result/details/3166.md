## Root Cause

test_consistency at third_party/torch-xpu-ops/test/xpu/test_sparse_csr_xpu.py:833 iterates OpInfo
samples over all compressed layouts (including SparseCSC at line 242/254). Failing ops are mul,
randn_like, zeros_like on SparseCSC and mul/randn_like/zeros_like on SparseCSR.
torch-xpu-ops/yaml/native/native_functions.yaml:255-277 registers mul_sparse_csr /
mul_out_sparse_csr / mul__scalar_sparse_csr for SparseCsrXPU only; there is no SparseCsc-specific
path and no XPU registration for randn_like/zeros_like on sparse compressed layouts, so either
dispatch falls through or the internal transpose->CSR path used by mul is not implemented for XPU.

## Fix Approach

- In torch-xpu-ops/src/ATen/native/sparse/xpu/SparseCsrTensorMath.cpp add CSC handling for mul (convert via transpose to CSR, call mul, convert back) mirroring the SparseCsrCUDA path.
- Add SparseCsrXPU registrations for zeros_like and randn_like (can dispatch to the generic compressed *_like implementation in aten/src/ATen/native/sparse/SparseCsrTensor.cpp.
- a simple yaml registration with the existing CPU-style implementation often suffices).
- Re-enable tests once kernels land.
