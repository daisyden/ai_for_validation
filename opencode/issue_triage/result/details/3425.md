## Root Cause

XPU clamp_scalar converts the scalar bound through opmath_t (float for fp16) at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/TensorCompareKernels.cpp:136-138
(`lim0.to<opmath_t>()`), which never triggers an overflow check against the storage dtype c10::Half.
CPU instead casts the scalar directly to scalar_t, where Scalar::to<c10::Half> raises 'value cannot
be converted to type c10::Half without overflow'. The XPU path therefore silently accepts values
like 65507 that exceed Half range. Upstream PR pytorch#173776 was closed without merge so a parallel
fix is needed for XPU.

## Fix Approach

- In launch_clamp_scalar (TensorCompareKernels.cpp:129-142), perform an overflow-checked cast to scalar_t before downcasting to opmath_t, e.g. call `lim0.to<scalar_t>()` first to trigger checked_convert (which throws on Half overflow) and then convert that value to opmath_t for the kernel functor.
- Apply same pattern for both lim0 and lim1.
- Add an XPU unit test covering fp16/bf16 overflow scalars to torch-xpu-ops test_ops_xpu.
