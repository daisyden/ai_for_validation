## Root Cause

torch.clip/clamp uses a TensorIterator-based pointwise kernel where the Python scalar bound is
converted to the tensor dtype (c10::Half) without an overflow check on XPU; CPU performs the
overflow check inside Scalar->Half conversion (checked_convert) but the XPU clamp kernel in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/TensorCompare.cpp loads the Scalar via
.to<scalar_t>() which silently saturates/truncates. The missing check is the same gap upstream PR
pytorch/pytorch#173776 attempted to fix in a device-agnostic place; that PR was closed unmerged, so
behavior remains divergent across devices.

## Fix Approach

- Best fix is upstream in aten/src/ATen/native/TensorCompare.cpp clamp_scalar_out helpers: validate scalar bounds against scalar_type using checked_convert before dispatching to any backend (revive pytorch#173776).
- As a stop-gap, add c10::checked_convert<scalar_t>(min/max, scalar_type) checks in the XPU clamp_scalar entry points in TensorCompare.cpp before launching the SYCL kernel.
