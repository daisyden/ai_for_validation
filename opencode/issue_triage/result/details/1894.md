## Root Cause

Two distinct PT2E failures: (1) demucs ASYMM/SYMM hits 'RuntimeError: quantized pointwise conv1d
doesn\u2019t support unary_post_op fusion. Got unary_post_op:relu' from
torch.ops.onednn.qconv_pointwise.tensor — the XPU qconv1d path in
torch-xpu-ops/src/ATen/native/xpu/QConv.cpp (or equivalent) only registers post-op fusion for
conv2d/3d; conv1d is missing the relu unary post-op branch. (2) hf_DistilBert fails in convert_pt2e
with 'prepare only works with cpu or single-device CUDA modules, but got devices {xpu:0, cpu}' —
torchao/AO's _assert_and_get_unique_device (torchao/utils.py:52) doesn't whitelist 'xpu'.

## Fix Approach

- For demucs: extend oneDNN-backed qconv XPU lowering to accept conv1d + unary post-ops (relu/hardtanh/etc.), mirroring the conv2d fusion table.
- add a PT2E unit test with conv1d+relu.
- For hf_DistilBert: either upstream a patch to torchao allowing 'xpu' in _assert_and_get_unique_device or ensure all prepared modules/observers are on xpu (no stray CPU tensors) in the benchmark harness and PT2E quantizer.
- Both must be validated against the PT2E performance CI run.
