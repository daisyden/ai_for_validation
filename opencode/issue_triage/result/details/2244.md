## Root Cause

test/test_sparse_csr.py:1602 run_test_block_addmm_addmv calls torch.addmm with a SparseBsr operand;
on XPU the dispatcher falls into a path that calls empty_sparse_compressed without allowing block
layouts, producing 'empty_sparse_compressed expected sparse compressed (non-block) tensor layout but
got SparseBsr'. CUDA has a dedicated BSR addmm path; XPU lacks it and the generic factory rejects
SparseBsr. Most dtype variants (fp32/fp64/complex) were already fixed upstream; remaining failures
are bfloat16/float16 where the BSR addmm path isn't wired for those dtypes.

## Fix Approach

- Extend the upstream sparse compressed addmm dispatch (or torch-xpu-ops registration) so XPU SparseBsr addmm routes to the BSR-aware kernel for bf16/fp16, or registers a CPU fallback.
- alternatively widen empty_sparse_compressed to accept block layouts when invoked from BSR addmm result allocation.
