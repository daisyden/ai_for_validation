## Root Cause

test_block_addmm constructs a BSR mat1 and a BSR result tensor. addmm_out_sparse_csr in
third_party/torch-xpu-ops/src/ATen/native/sparse/xpu/SparseCsrTensorMath.cpp:111-170 only handles
(BSR mat1, strided mat2, strided result) and (strided, BSC, strided); no branch matches BSR->BSR, so
control falls through to the generic sparse-compressed constructor which calls
empty_sparse_compressed in aten/src/ATen/native/sparse/SparseCsrTensor.cpp:697-712 where
AT_DISPATCH_SPARSE_COMPRESSED_NONBLOCK_LAYOUTS rejects SparseBsr with 'expected sparse compressed
(non-block) tensor layout but got SparseBsr'. Only bf16/f16 test variants fail because
f32/f64/complex already hit separate oneDNN issues.

## Fix Approach

- Add the missing BSR-result branch in addmm_out_sparse_csr: compute the dense product via addmm_calculation, convert back with result = result_dense.to_sparse_bsr(block_size) (or use torch::sparse::to_sparse_bsr).
- Mirror the same for (CSC, CSC), (BSR, BSR) and (BSC, BSC) combinations that the CUDA path supports.
- Add/extend GTest or bring the op back in the skip list only as an interim measure.
- No oneDNN change required.
- this is pure XPU dispatch plumbing.
