## Root Cause

torch.std on a 1000-element fp32 tensor with magnitudes ~1e20 returns inf on XPU while CPU returns
~1e19. The XPU std/var path uses WelfordOps in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ReduceMomentKernels.cpp:33
(std_var_kernel_impl_xpu uses `WelfordOps<scalar_t, accscalar_t, int32_t, ...>`); for fp32 input
accscalar_t resolves to float, so the running M2 (sum of squared deviations) overflows when
individual deviations are ~1e20 and squared values exceed FLT_MAX (~3.4e38). CPU uses a
higher-precision accumulation path so it stays finite. Mirrors upstream pytorch/pytorch#180156.

## Fix Approach

- Promote the Welford accumulator type to double (or use Kahan/compensated accumulation) for fp32 inputs in ReduceMomentKernels.cpp, mirroring CUDA's std implementation in aten/src/ATen/native/cuda/ReduceMomentKernel.cu.
- Add a numerical regression test for large-magnitude fp32 std in test/xpu/test_reductions_xpu.py.
- coordinate with upstream PR for #180156 if it generalizes the accumulator type.
