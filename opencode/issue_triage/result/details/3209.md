## Root Cause

In third_party/torch-xpu-ops/src/BuildOnWindows.cmake, the BUILD_SEPARATE_OPS branch (lines 27-42)
calls setup_common_libraries() which defines torch_xpu_ops as STATIC and links it PUBLIC to
torch_xpu (line 22). PyTorch's top-level build then links torch_xpu_ops into the torch_xpu SHARED
library, closing a torch_xpu_ops (STATIC) <-> torch_xpu (SHARED) cycle. CMake permits cycles only
among STATIC libraries, so the Debug/RelWithDebInfo configuration that forces
BUILD_SEPARATE_OPS=TRUE at CMakeLists.txt:99-101 fails with 'Cyclic dependencies are allowed only
among static libraries'. Non-SEPARATE path avoids this because extra per-kernel SYCL SHARED libs are
not introduced in the dependency chain the same way.

## Fix Approach

- In setup_common_libraries(), drop or weaken the PUBLIC link from torch_xpu_ops to torch_xpu (e.g., replace with target_include_directories or PRIVATE link against c10/torch_cpu headers only, as torch_xpu will link torch_xpu_ops at the outer level).
- Alternatively, build torch_xpu_ops as an OBJECT library when BUILD_SEPARATE_OPS=TRUE so the cycle becomes allowed, mirroring the Linux BuildOnLinux.cmake pattern.
