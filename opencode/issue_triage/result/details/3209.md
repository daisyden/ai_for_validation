## Root Cause

When BUILD_SEPARATE_OPS=TRUE (forced in Debug/RelWithDebInfo builds,
third_party/torch-xpu-ops/CMakeLists.txt:98-101), torch_xpu_ops is built as a STATIC_LIBRARY but a
cyclic target dependency exists between torch_xpu (SHARED) and torch_xpu_ops (STATIC). CMake only
allows cycles among STATIC libraries; because torch_xpu is SHARED the generate step fails on
Windows. The cycle stems from torch_xpu_ops linking back to torch_xpu symbols that are exported only
from the shared lib.

## Fix Approach

- Break the cycle by either (a) making torch_xpu_ops OBJECT library and linking objects directly into torch_xpu, or (b) refactoring symbol ownership so torch_xpu_ops only exports symbols and torch_xpu consumes them (one-way dep).
- Alternatively gate torch_xpu_ops to SHARED on Windows when BUILD_SEPARATE_OPS=TRUE.
- Update third_party/torch-xpu-ops/CMakeLists.txt and src/ATen/CMakeLists.txt accordingly.
