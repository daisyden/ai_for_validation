## Root Cause

torch.native_channel_shuffle on XPU diverges from CPU on invalid inputs: groups=0 hits SIGFPE
(integer divide-by-zero) inside the channel_shuffle implementation, and channels-not-divisible /
rank<3 cases silently succeed. native_channel_shuffle has no torch-xpu-ops SYCL kernel (no match for
ChannelShuffle in src/), so XPU dispatches to the upstream native implementation in pytorch/pytorch
(aten/src/ATen/native/ChannelShuffle.cpp) which lacks the input-validation guard added in upstream
PR pytorch/pytorch#181029 (commit 317a3d9). The missing TORCH_CHECK for groups>0,
channels%groups==0, and input.dim()>=3 lets divide-by-zero and undefined behavior reach the SYCL
launch path.

## Fix Approach

- Sync upstream PR pytorch/pytorch#181029 (TORCH_CHECK guards on groups, channels divisibility, and input rank in native_channel_shuffle / channel_shuffle).
- Validate locally with the issue's three subprocess cases.
- No torch-xpu-ops src change needed once the upstream pytorch checks are in place.
- add an XPU parametrized test under test/xpu to lock in the validation behavior.
