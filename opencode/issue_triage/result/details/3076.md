## Root Cause

Llama-3.2-1B-Instruct dynamic INT8 inference on BMG (Arc B580) regresses ~10% on next-token latency
after the oneDNN bump to 3.11.1. Per reporter's investigation (MFDNN-14792), the regression is
isolated to a specific int8 matmul shape used by the decode step; oneDNN 3.11.1 selects a worse GEMM
implementation for that shape than 3.11.0. PyTorch/torch-xpu-ops AO path
(third_party/torch-xpu-ops/src/ATen/native/xpu/QuantizedLinear.cpp + oneDNN brgemm dispatch) is
unchanged, so this is a pure dependency perf regression.

## Fix Approach

- Primary fix is in oneDNN (MFDNN-14792) — restore or improve the int8 matmul kernel selection for the decode shape.
- Once oneDNN has a fix, bump the pinned oneDNN version in pytorch/third_party/ideep / XPU build.
- As a workaround inside torch-xpu-ops, consider adding a shape-aware path in QuantizedLinear/Matmul to force a specific primitive attribute / post-op or fall back to a BLAS path for the regressed shape.
- collect verbose oneDNN primitive cache logs (ONEDNN_VERBOSE=1) to confirm which kernel is picked and add a regression perf test on BMG.
