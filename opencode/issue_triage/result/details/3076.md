## Root Cause

Llama-3.2-1B-Instruct dynamic INT8 inference on BMG (Arc B580) shows ~10% next-token latency
regression after upgrading to oneDNN 3.11.1. Maintainer LifengWang confirmed this is a oneDNN
performance degradation for a specific INT8 matmul shape (tracked internally as MFDNN-14792). The
slowdown originates in oneDNN's INT8 matmul kernel selection/heuristics for the dynamic-quant gemm
shapes used by Llama-3.2-1B, not in torch-xpu-ops glue code at
third_party/torch-xpu-ops/src/ATen/native/xpu/Linear.cpp.

## Fix Approach

- Wait for oneDNN fix from MFDNN-14792 and bump the oneDNN pin in torch-xpu-ops/cmake (third_party/torch-xpu-ops/cmake/Modules/FindONEDNN.cmake) once a fixed release is available.
- As an interim mitigation, pin oneDNN back to the prior known-good version, or add a shape-specific workaround in the dynamic-int8 matmul dispatch in src/ATen/native/xpu/Linear.cpp.
- Validate with the run_generation.py reproducer (bs=32, in=1024, out=1024) on BMG.
