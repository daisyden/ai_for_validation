## Root Cause

Casting a tensor to torch.float8_e4m3fn on XPU goes through float8_copy_kernel_xpu in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/CopyKernel.cpp:49-66, which calls
CastScalarFunc<T, Float8_e4m3fn> -> Float8DataType(src_val). The SYCL/DPC++ compiler fuses the
fp16/bf16->fp32->fp8 conversion and violates the saturation semantics of Float8_e4m3fn (max normal
448), turning values near-max into NaN instead of saturating (same compiler-fusion class of bug
already worked-around for Half->Float8 via the -0.0 check at lines 41-47). CUDA produces 288, XPU
produces NaN.

## Fix Approach

- Extend the CastScalarFunc specialization in CopyKernel.cpp to cover all source dtypes (float, BFloat16) for Float8_e4m3fn/e5m2 and perform an explicit saturating clamp to c10::Float8_e4m3fn::max() (448.0f) in software before invoking the Float8 constructor, mirroring cuda/Copy.cu's sat path.
- Alternatively mark the cast function with sycl::no_fast_math or a volatile intermediate to defeat the fp16->fp32->fp8 fusion, then rely on Float8_e4m3fn::from_fp32_value saturation.
