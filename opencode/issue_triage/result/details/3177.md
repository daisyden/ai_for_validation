## Root Cause

Sparse CSR/BSR addmm XPU reference implementation computes result via dense fallback at
third_party/torch-xpu-ops/src/ATen/native/sparse/xpu/SparseCsrTensorMath.cpp:82-90, where the
beta*input accumulation uses Tensor::add_(input_dense, beta) (i.e. alpha-form add). For fp16/bf16
inputs this low-precision fused scale+add diverges from the test's reference, which computes beta*c
first then adds, producing accuracy gaps for test_block_addmm block_size={2,3} fp16/bf16
(test_sparse_csr_xpu.py:2336). fp32/fp64 paths are unaffected. jenniew confirmed in the comment
thread and has an open fix in PR #3273.

## Fix Approach

- Land torch-xpu-ops PR #3273: replace result_dense.add_(input_dense, beta) at SparseCsrTensorMath.cpp:88 with explicit `result_dense.add_(input_dense * beta)` (or upcast to float32 before accumulation) to match the reference semantics for fp16/bf16, then remove the skip entries from test/xpu/skip_list_common.py.
