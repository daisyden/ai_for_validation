## Root Cause

test/test_cuda_nvml_based_avail.py exercises the CUDA NVML-based availability path by mocking
torch.cuda._device_count_nvml / _raw_device_count_nvml (torch/cuda/__init__.py:906,1052) and
requires the PYTORCH_NVML_BASED_CUDA_CHECK env var. XPU has no analogous NVML/xpu-smi device-count
plumbing in torch/xpu/__init__.py (is_available at line 77 calls _C._xpu_getDeviceCount directly),
so these CUDA-only tests cannot be generalized. They are currently blocking because they are being
picked up by the XPU UT sweep with no skip.

## Fix Approach

- These tests are CUDA-specific and should be excluded on XPU (add to the skip list in third_party/torch-xpu-ops/test/xpu/skip_list_common.py or keep them guarded by TEST_CUDA).
- A longer-term port would require implementing torch.xpu._device_count_sysman via Level-Zero sysman or xpu-smi plus a PYTORCH_XPU_BASED_CHECK env var mirroring the CUDA path.
- given low user impact, preferred fix is to skip.
