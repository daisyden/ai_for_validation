## Root Cause

test/distributed/_tools/test_sac_ilp.py exercises RuntimeEstimator's roofline path. In
torch/distributed/_tools/runtime_estimator.py the _roofline_estimate method asserts
torch.cuda.is_available() at line ~296 and uses torch.cuda.get_device_properties / get_device_tflops
without any XPU branch, so on an XPU machine the assertion trips. The author has confirmed this is a
duplicate of intel/torch-xpu-ops#2163, which tracks the broader work to device-generalize
RuntimeEstimator (it also needs get_device_tflops/get_drams_gbps on XPU, see issue 2329).

## Fix Approach

- Close as duplicate of #2163 and track fix there.
- The real fix is to replace the hard torch.cuda.is_available() gate in torch/distributed/_tools/runtime_estimator.py with torch.accelerator.is_available() (or an explicit XPU branch), route device-property lookups through torch.accelerator.current_accelerator(), and implement XPU equivalents of torch._utils.get_device_tflops / get_dram_gbps (see issue 2329).
- Until XPU peak-flops/bandwidth helpers exist the test should be skipped on XPU with @skipIfXpu or gated behind a roofline-supported predicate.
