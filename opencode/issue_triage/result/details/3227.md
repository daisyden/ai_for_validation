## Root Cause

torch.xpu.Event.elapsed_time exhibits ~120us per-event overhead vs ~6us on CUDA. The SYCL
event-based timing path in c10/xpu/XPUEvent.h (record at line 72; query/elapsed via
sycl::info::event_profiling) relies on Level-Zero/SYCL sycl::event profiling counters whose
resolution and host-side query cost dominate small-kernel measurement. There is no analogue to
CUDA's lightweight event timestamping (cuEventRecord), so each event creation/record/query incurs
significant host-driver round-trip latency.

## Fix Approach

- Profile the XPUEvent.record/elapsed_time path under L0 to confirm whether the cost is in sycl::event creation, ze_event_pool host-flush, or the sycl::event::get_profiling_info() call.
- Consider (a) using a pre-allocated ze_event_pool with ZE_EVENT_POOL_FLAG_KERNEL_TIMESTAMP and direct L0 calls (bypassing SYCL wrapper) inside c10/xpu/XPUEvent.h::record/elapsed_time.
- (b) batching event queries.
- (c) reporting the regression to compute-runtime/level-zero if the overhead is in the driver.
- Coordinate with the driver team since native SYCL profiling is the bottleneck.
