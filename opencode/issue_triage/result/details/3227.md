## Root Cause

c10/xpu/XPUEvent.h:157-163 records events via
sycl::ext::oneapi::experimental::submit_profiling_tag(queue) when enable_timing=true, and
elapsed_time (lines 125-143) calls get_profiling_info<command_end>() on each event. Each
submit_profiling_tag enqueues a host-round-trip profiling barrier and each record/elapsed_time pair
blocks waiting for profiling info, yielding ~120 us host overhead per event on PVC/B580 (vs ~6 us on
CUDA which uses lightweight cudaEventRecord). The overhead comes from the SYCL/level-zero runtime
profiling path, not kernel work.

## Fix Approach

- Replace submit_profiling_tag with a lighter mechanism: either use an in-order queue's built-in timestamp (ze_event_pool with TIMESTAMP flag) or batch-queue a kernel timestamp event via level-zero directly.
- alternatively cache one profiling event per stream and reuse it.
- Needs coordination with SYCL/level-zero driver team (compute-runtime) since the overhead is in the SYCL runtime.
