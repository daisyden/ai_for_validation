## Root Cause

torch.xpu.Event.record/elapsed_time exhibits ~120us per-event overhead on B580 vs ~6us on CUDA when
measured outside the profiler (issue body). The XPU event implementation in
c10/xpu/XPUEvent.h:72-103 (record) and :125 (elapsed_time) relies on SYCL/Level-Zero queue
submission and host-side sycl::event timing rather than a lightweight GPU timestamp; the heavy host
synchronization plus L0 event creation cost dominates kernel time for fast pointwise ops like neg_.
Profiler path uses a separate timing mechanism that's much cheaper (0.0009ms).

## Fix Approach

- Investigate using a cached/pooled sycl::event or Level-Zero timestamp queries instead of constructing/submitting marker events on every record().
- Compare with CUDA cudaEventRecord behavior.
- consider a fast-path that defers event creation and uses ze_event_pool timestamps.
- File a driver/SYCL runtime ticket if overhead is in L0 ze_event submission.
- Update c10/xpu/XPUEvent.h record() to avoid full queue barrier where possible.
