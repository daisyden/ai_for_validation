## Root Cause

test/distributed/_tools/test_runtime_estimator.py, test_sac_ilp.py, and test_sac_estimator.py are
skipped on XPU because torch/distributed/_tools/sac_estimator.py -> runtime_estimator.py hard-codes
CUDA: (1) torch/_inductor/analysis/device_info.py has no XPU DeviceInfo entry, (2)
torch/_inductor/utils.py and torch/distributed/_tools/runtime_estimator.py call torch.cuda.* APIs
unconditionally, and (3) triton.testing.do_bench and its GPU clock helpers at
intel-xpu-backend-for-triton/python/triton/testing.py:421 lack an XPU branch. Even after
@githubsgi's hack patches, test_sac_ilp_case1 fails at test_sac_ilp.py:158 with peak_mem/2583888896
== 2.52 vs expected 1, showing the XPU memory-estimation model is uncalibrated.

## Fix Approach

- Land the four enumerated upstream enablement changes: add an XPU entry in torch/_inductor/analysis/device_info.py, replace torch.cuda.* with torch.accelerator in torch/_inductor/utils.py and torch/distributed/_tools/runtime_estimator.py, and add an XPU path in intel-xpu-backend-for-triton python/triton/testing.py do_bench.
- Then calibrate the activation-checkpoint memory model in sac_estimator.py for XPU (likely adjust allocator alignment/peak-tracking).
- gate ratio tolerances per-device or compute a dynamic baseline instead of the hard-coded 2583888896 constant in test_sac_ilp_case1.
