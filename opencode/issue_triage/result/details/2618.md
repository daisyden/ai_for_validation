## Root Cause

All failing linalg tests (det/solve/tensorinv/inverse/cond/linalg_matrix_exp) compute their CPU
reference via numpy==1.23.2 which ships OpenBLAS 0.3.20. OpenBLAS 0.3.20 has a numerical bug on
Sapphire Rapids / Granite Rapids CPUs (Cooperlake core type) producing wrong determinant/solve
results; the XPU results are actually correct. Root cause is not in torch-xpu-ops kernels but in the
CI reference library at pytorch/.ci/docker/requirements-ci.txt:138 pinning numpy==1.23.2 for python
3.10. Fix was tracked via OpenBLAS commits dc49edd4e/dac14a5f7 available starting OpenBLAS 0.3.21
(numpy>=1.24.0).

## Fix Approach

- Since upstream pytorch is unlikely to bump numpy on python 3.10 soon, apply the workaround in torch-xpu-ops CI workflow: detect via `python -m threadpoolctl -i numpy` whether numpy 1.23.* with OpenBLAS Cooperlake arch is in use, and set `OPENBLAS_CORETYPE=SkylakeX` before running linalg UTs (see PR intel/torch-xpu-ops#3219).
- Alternatively add a `skipIfNumpyOpenBLASBug` decorator in the test list for these specific linalg tests with a clear message referencing OpenBLAS 0.3.20 Cooperlake bug.
- Long-term: submit a PR to bump pytorch requirements-ci.txt numpy to >=1.24.0 for py3.10, or wait for CI hardware upgrade past SPR.
