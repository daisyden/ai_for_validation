## Root Cause

Linalg cases (det/logdet/slogdet, solve, tensorinv, inverse_many_batches, cond, matrix_exp_analytic
on float32/float64) report 'Scalars/Tensor-likes are not close'. Per @jmamzax investigation in the
issue thread, these failures are caused by an OpenBLAS bug in numpy==1.23.2 reference computation on
Sapphire Rapids / Granite Rapids CPUs: the numpy reference is wrong while the XPU oneMKL-backed
linalg results are correct. test_max_pool2d_cudnn_xpu fails because the test guards on torch.cuda
which is unavailable; view_as_complex Sparse cases were dispatch-registration gaps already resolved
in nightly.

## Fix Approach

- For the linalg block: skip these cases on Sapphire/Granite Rapids CI or bump numpy>=1.24 in the CI image to avoid the OpenBLAS reference bug.
- do not change XPU oneMKL kernels.
- For test_max_pool2d_cudnn_xpu: add a skip-if-no-cuDNN guard in the xpu test override.
- The view_as_complex SparseXPU items are already passing per github-actions bot.
