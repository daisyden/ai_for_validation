## Root Cause

test_vjp_linalg_tensorsolve_xpu_float32 fails on Windows with a marginal numerical mismatch
(greatest rel diff 2.096e-4 vs 2.0e-4 tolerance; abs diff 0.666 on float32). tensorsolve is a
composite op that routes through linalg.solve backed by oneMKL on XPU
(third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/BatchLinearAlgebra.cpp), and its VJP uses
solve+matmul. The backward path is numerically sensitive at float32 and oneMKL's getrf/getrs
accumulation order differs slightly from CUDA cuSOLVER, producing reference-drift just over the
OpInfo tolerance. Windows-only label indicates DPC++/oneMKL build on Windows exacerbates the drift.

## Fix Approach

- Add an XPU-specific tolerance override (toleranceOverride/Windows skip) for linalg_tensorsolve float32 VJP in third_party/torch-xpu-ops/test/xpu/xpu_test_utils.py or an OpInfo skip list, consistent with other linalg float32 tolerance bumps, until a deterministic oneMKL backward path is implemented.
- Verify by rerunning test_vjp_linalg_tensorsolve_xpu_float32 with relaxed rtol (~3e-4).
