## Root Cause

On Windows LNL (16.5GB shared host memory), once allocation triggers OOM the SYCL/UR context is left
in a bad state: subsequent access to previously-valid tensors returns
UR_RESULT_ERROR_OUT_OF_RESOURCES and then opaque 'UR error'. Tracked as JIRA
GSD-10738/10905/11849/11670; driver reports incorrect free memory and compute-runtime's Windows flag
(commit d667614) does not actually recover the context. Problem is in the Intel compute-runtime/UR
layer, not torch-xpu-ops.

## Fix Approach

- Wait for the compute-runtime / UR fix tied to GSD-11670 and oneAPI 2026.1 (per chuanqi129 2026-04-01).
- In the meantime, document the limitation in XPUCachingAllocator error text and consider catching UR_RESULT_ERROR_OUT_OF_RESOURCES on the first access after OOM to convert to a clearer torch.OutOfMemoryError.
- Re-verify once agama build with GSD-11670 fix lands.
