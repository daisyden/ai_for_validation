## Root Cause

TorchAO int4 weight-only quantization (int4wo) produces wrong outputs on XPU for bfloat16 tensors
after Tensor.copy_() or slicing on the packed weight. In
third_party/torch-xpu-ops/src/ATen/native/xpu/WeightInt4Pack.cpp:17
(_convert_weight_to_int4pack_xpu) and the associated kernels LinearInt4.cpp:21
(_weight_int4pack_mm_xpu) / sycl/WeightInt4PackKernel.cpp:52 (weight_to_int4pack_kernel) /
sycl/LinearInt4.cpp:220 (linear_int4_kernel), the packed weight layout assumes a specific contiguous
storage. When copy_ is invoked, the generic copy path does not re-pack/re-layout (or bypasses the
XPU-specific packing), so ql2's weight storage differs from ql's, yielding >99% mismatches. The
issue only surfaces after copy/slice because the packed tensor metadata/stride is not preserved
end-to-end in TorchAO's XPU path.

## Fix Approach

- Align XPU int4pack packing and slice/copy semantics with CUDA: (1) in _convert_weight_to_int4pack_xpu ensure the packed layout is identical to CUDA's, and (2) in TorchAO AffineQuantizedTensor XPU dispatch, override copy_/slice to operate on the packed tensor correctly, or force re-packing when source storage layout changes.
- Add a unit test mirroring test_slice_int4wo_xpu_bfloat16.
- Cross-check with CUDA reference and, if the packed layout differs, add a converter.
- As a short-term mitigation, force `aten.copy_` on int4pack tensors to go through a re-pack path in torchao's TensorImpl.
