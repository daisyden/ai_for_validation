## Root Cause

Inductor tests `test_argmax_argmin2/3_xpu`, `test_argmax_argmin_with_duplicates_xpu`,
`test_argmax_argmin_with_nan_xpu` (test/inductor/test_torchinductor.py:10319) mismatch by 3/6
elements, greatest absolute diff 1. XPU argmax/argmin results disagree with eager on tie-break (must
return first index among equal values) and on NaN propagation. The implementation lives in
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/ReduceArgMaxKernel.cpp / ReduceArgMinKernel.cpp
where the combine operator in the parallel reduction does not strictly prefer the smaller index on
equality, and does not treat NaN as greater/lesser consistently with CUDA semantics.

## Fix Approach

- Fix the argmax/argmin reduction combine functor in ReduceArgMaxKernel.cpp / ReduceArgMinKernel.cpp to: (1) on equal values pick the pair with the smaller index (matching `at::native::argmax_out` CUDA semantics).
- (2) propagate NaN so that any NaN operand yields NaN ordering consistent with eager (comparing with `a != a` style checks).
- Mirror the CUDA `ArgMaxOps`/`ArgMinOps` template used in aten/src/ATen/native/cuda/ReduceOps.cpp.
- Add XPU-specific unit tests with duplicate values and NaN inputs to lock in behavior.
- The driver-dependency label is misleading: this is a kernel bug, not a driver bug.
