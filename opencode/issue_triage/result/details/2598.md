## Root Cause

Performance regression: on BMG (Arc B580), Qwen2.5-1.5B-Instruct first-token latency drops >10% when
`--max-new-tokens` changes from 2 -> 1 with `--inductor --woq --woq-type rtn --quant-dtype uint4`.
With tokens=1 only the prefill graph runs once and warmup cycles do not amortize compile/autotune
and oneDNN primitive-cache lookup costs for the int4 GEMMs dispatched via _weight_int4pack_mm_xpu
(LinearInt4.cpp:21) and its M>1 two-kernel path (dequant_int4_kernel + A.matmul). The first call
re-compiles/rebuilds oneDNN matmul primitives (1024x3072 etc.), so with fewer iterations the
first-token metric captures a larger share of this one-time overhead.

## Fix Approach

- Profile first-token with `SYCL_PI_LEVEL_ZERO_USE_IMMEDIATE_COMMANDLISTS=2` vs default and with onednn-verbose=1 to confirm primitive cache miss on the prefill shape.
- Options: (1) warm oneDNN matmul primitive cache for the expected prefill shape during `torch.compile` warmup.
- (2) skip the dequant-then-matmul path in LinearInt4.cpp:63 for prefill shapes where a fused int4 GEMM would avoid the K*N bf16 materialization.
- (3) ensure the first warmup iter in run_generation is counted as warmup so the reported first-token matches steady state.
- Validate: measure first-token latency with `--num-warmup 5 --num-iter 5` for tokens=1 and tokens=2.
