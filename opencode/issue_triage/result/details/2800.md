## Root Cause

test/test_scaled_matmul_cuda.py:920 reads
torch.get_device_module(device).get_device_properties(0).major to gate an FP8 scaled_mm check, but
torch._C._XpuDeviceProperties has no 'major' attribute. The XPU property binding in
torch/csrc/xpu/Module.cpp (THXP_FORALL_DEVICE_PROPERTIES at lines 327-349) only exposes SYCL fields
(name, device_id, driver_version, architecture, etc.); there is no DPC++/Level-Zero API today that
reports CUDA-style (major, minor) compute-capability numbers. Per the comment thread, the missing
API is tracked as CMPLRLLVM-72166 and is targeted for DPC++ 2026.2 / oneAPI 2026.1.

## Fix Approach

- Short-term: add a test-side guard that only queries '.major' on CUDA (e.g. skip/branch when device.type=='xpu' in test_scaled_matmul_cuda.py::test_scaled_mm_vs_emulated) and/or expose a dummy 'major'/'minor' property_readonly on _XpuDeviceProperties returning -1 so that inequality checks still skip the Blackwell-only path.
- Long-term: once DPC++ 2026.2 lands CMPLRLLVM-72166, map the new arch-version API to .major/.minor in torch/csrc/xpu/Module.cpp registerXpuDeviceProperties and update python-side torch/xpu/__init__.py to surface them.
