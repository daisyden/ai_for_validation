## Root Cause

torch/sparse/_triton_ops_meta.py and torch/sparse/_triton_ops.py are CUDA-hardcoded: get_meta()
calls torch.cuda.get_device_name() (torch/sparse/_triton_ops_meta.py:138, also lines 486, 579, 741,
940) and _triton_ops.py:532, 788 does the same. On an XPU-only build torch.cuda._lazy_init() fails
with 'Torch not compiled with CUDA enabled', so every bsr_dense_addmm / bsr_dense_mm / scatter_mm
test path aborts before it can dispatch to the XPU triton kernel. _operation_device_version_data
also only contains NVIDIA device keys, so even with the guard fixed there is no XPU meta to return.

## Fix Approach

- Port the two modules to be device-agnostic: replace torch.cuda.get_device_name() with a helper that picks the active accelerator (torch.accelerator.current_accelerator() / torch.xpu.get_device_name() when xpu is available), and add XPU entries to _operation_device_version_data (either by running the tune_bsr_dense_addmm/tune_scatter_mm scripts on an Intel GPU and committing the meta, or by providing a safe default meta for XPU).
- Update tests or OpInfo decorators in test_sparse_csr.py only if the triton kernels themselves are not supported.
- otherwise remove the skip after meta is landed.
- Coordinate with the existing PyTorch upstream PR for triton-xpu sparse support.
