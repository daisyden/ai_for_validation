## Root Cause

test_out_addmv/test_out_mv (and the vmap variant) on XPU float32 fail because the out tensor's
strides differ after the op: original strides were (2,) but become (1,) (issue body). The XPU
addmv/mv implementation (oneDNN gemv path under src/ATen/native/xpu/Blas.cpp) re-allocates or
resizes the provided out tensor instead of preserving the user-provided non-contiguous strides,
while the CPU/CUDA reference preserves them. This violates the out-variant contract checked by
TestCommonXPU.test_out.

## Fix Approach

- In the XPU addmv/mv out-variant, when the supplied out tensor has compatible shape but non-default strides, write into it in-place without resize_as_/copy_ that drops strides.
- Mirror CUDA's at::native::addmv_out logic.
- Add a unit test reproducing the strided-out scenario, then remove the case from the skip list.
- Cross-check with linked issue #2669 for the vmap exhaustive variant.
