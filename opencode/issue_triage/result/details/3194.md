## Root Cause

addmv.out is registered as a structured op (yaml/native/native_functions.yaml:9585-9588) and for
plain XPU delegates to the upstream structured kernel; torch-xpu-ops only provides SparseCsrXPU
dispatch. test_out_addmv_xpu_float32 / test_out_mv_xpu_float32 / test_vmap_exhaustive_addmv verify
that when the user supplies an `out=` tensor with non-contiguous strides (e.g. (2,)), the op
preserves those strides. The XPU path instead resizes/overwrites the output to contiguous strides
(1,), producing 'Strides are not the same! Original strides were ((2,),) and strides are now
((1,),)'. The violation is in the XPU structured set_output_strided glue (upstream at::meta/addmv
structured impl + XPU's resize_output_check) rather than numerics.

## Fix Approach

- Trace the structured addmv.out impl on XPU: confirm whether meta::addmv's set_output_strided propagates through to_plain_output_strides in torch-xpu-ops' structured registration.
- Fix by honoring `out.strides()` when is_non_overlapping_and_dense, mirroring CUDA's addmv_out path in aten/src/ATen/native/cuda/Blas.cpp.
- Linked/duplicates #2669 (test_vmap_exhaustive_addmv already tracked there).
