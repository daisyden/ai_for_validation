## Root Cause

On Windows nightly 0406 the addcmul int8/uint8/int16/int32/int64 cases of test_compare_cpu
(third_party/torch-xpu-ops/test/xpu/extended/test_ops_xpu.py via TestCommonXPU) regressed. The XPU
addcmul kernel in third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/PointwiseOpsKernels.cpp:48-78
dispatches integer types via AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND2 using accscalar_t and
gpu_kernel; the Linux runs of the same kernel are clean, indicating the failure is
Windows-driver-specific (ocloc/IGC integer-promotion or saturation behaviour on Windows
compute-runtime), not algorithmic. No code change in torch-xpu-ops introduced it; it's a
'skipped_windows' regression.

## Fix Approach

- Add the five addcmul int cases to test/xpu/extended/skip_list_win.py to unblock Windows CI, then file the failure against Windows compute-runtime/IGC with a minimal SYCL reproducer of integer fma semantics.
- Compare the failed-vs-CPU diff in the linked CI log (intel/torch-xpu-ops actions/runs/24034124139) to determine if this is wraparound/saturation or wrong-result.
- if confirmed driver bug, attach driver version and open an internal JIRA against the Windows GPU driver team.
