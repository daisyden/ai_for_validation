## Root Cause

functorch conv_transpose2d/3d sample inputs include corner cases (e.g. stride=(3,2), padding=(1,2),
output_padding=(2,3), groups=2, dilation=(4,4)) where output_padding >= max(stride, dilation) or
dilation>1 with non-trivial output_padding. In aten/src/ATen/native/mkldnn/xpu/detail/Deconv.cpp the
dst dims are computed in deconv_dst_size (~line 35-48) and the primitive descriptor is created at
line 194 (dnnl::deconvolution_forward::primitive_desc). oneDNN enforces output_padding < max(stride,
dilation*(kernel-1)+1), which these samples violate, so 'could not create a primitive descriptor for
the deconvolution forward propagation primitive' is thrown. The XPU path does not validate or fall
back before hitting oneDNN, while CUDA uses ATen's native slow_conv_transpose implementation that
accepts these parameters. Partial fix landed in torch-xpu-ops PR #2210 but several jvp/vjp-composite
variants still fail.

## Fix Approach

- In Deconv.cpp add pre-dispatch parameter validation mirroring oneDNN's requirements.
- when output_padding or dilation combinations are unsupported, fall back to the generic ATen slow_conv_transpose{2,3}d_forward kernel (or densified at::_convolution path) rather than calling oneDNN.
- Match CUDA dispatch logic in aten/src/ATen/native/Convolution.cpp (select_conv_backend) so XPU reports 'SlowNd' for those parameter combos.
- Continue tracking the xfail list from PR #2210 until the fallback is merged.
