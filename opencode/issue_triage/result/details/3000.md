## Root Cause

DataLoader with multiprocessing context forkserver/spawn pickles XPU storages via reduce_storage in
torch/multiprocessing/reductions.py:615, which falls through to storage._share_fd_cpu_()
(storage.py:527). The XPU storage class inherits the CPU code path but the underlying _share_fd_
TypedStorage method raises 'only available on CPU' because XPU does not implement file-descriptor
based shared-memory IPC for nested-tensor DataLoader workers. This is a Torch runtime/IPC feature
gap for XPU storages.

## Fix Approach

- Either (a) implement XPU-side reduce_storage / _share_fd_ via Level Zero IPC handles in torch/multiprocessing/reductions.py and the XPU storage backend, or (b) keep the test skipped via test/xpu/skip_list_common.py for test_nested_tensor_multiprocessing_context_{forkserver,spawn}_xpu (already labeled 'skipped').
- Long-term: file upstream feature request to add XPU IPC support paralleling CUDA's cudaIpcGetMemHandle path.
