## Root Cause

test_quick_addmv_xpu_float64 cross-references the oneDNN GEMV result against the decomp reference
via assertEqual with the default tight tol (1e-7). With the oneDNN v3.10 uplift, addmv's fp64 matmul
kernel changed FMA/accumulation order yielding ~9e-7 abs diff on 1 of 5 elements, exceeding the
decomp test's default tolerance at test/xpu/test_decomp.py:310 (op_assert_equal). Passes with oneDNN
v3.9, so it is a library-upgrade precision regression within fp64 numerical noise, not a correctness
bug.

## Fix Approach

- Adopt the upstream tolerance fix proposed for addmv decomp cross-reference in pytorch/pytorch#174590 (add addmv to the per-op tolerance override map in test_decomp.py).
- Mirror the same override in third_party/torch-xpu-ops/test/xpu/test_decomp.py op_assert_equal overrides, then unskip in skip_list_common.py.
- Track duplicate issue #2269 for closure.
