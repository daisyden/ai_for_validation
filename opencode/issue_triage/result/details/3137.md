## Root Cause

test_transformerencoder_fastpath_*_use_autocast_True_* exercises
`torch._transformer_encoder_layer_fwd` under xpu autocast. The dispatcher still routes to the native
transformer encoder fastpath, which on XPU goes through the oneDNN SDPA path that does not apply the
autocast Half promotion to all internal parameters, so the kernel sees a mixture of Half activations
and Float weights/bias
(aten/src/ATen/native/transformers/transformer.cpp:_transformer_encoder_layer_fwd, invoked from
torch/nn/modules/transformer.py:917), yielding 'expected scalar type Half but found Float'. A
functional fix is already in pytorch/pytorch PR #179701 which disables the fastpath / aligns dtypes
for XPU autocast; a residual numerical gap remains because XPU SDPA backward still falls back (will
close when oneDNN supports SDPA training).

## Fix Approach

- Merge/forward-port pytorch/pytorch#179701 so `_transformer_encoder_layer_fwd` takes the slow path (or casts parameters) under xpu autocast, then re-enable the four skipped cases.
- Track the accuracy delta under a separate oneDNN-SDPA-training follow-up.
