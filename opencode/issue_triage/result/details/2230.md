## Root Cause

test_triton_sampled_addmm (test/test_sparse_csr.py) constructs tensors on multiple XPU devices and
calls torch.sparse.sampled_addmm, which routes through
torch/sparse/_triton_ops.py::_scaled_dot_product_attention and check_device (_triton_ops.py:23-34).
The Triton op path only accepts inputs on one device; XPU has no multi-device sampled_addmm Triton
implementation so check_device rejects the test setup. Earlier bsr_dense_mm / SDPA Triton sub-cases
have been fixed/skipped but sampled_addmm variants still trigger the same ValueError.

## Fix Approach

- Either (a) port the sampled_addmm Triton kernel to support XPU multi-device dispatch similar to the CUDA path, or (b) add XPU skips/xfails in test/xpu/skip_list_common.py for the remaining sampled_addmm_block_size_{16,32,64} bfloat16/float16/float32 cases and register a fallback path.
- Short-term, keep CPU fallback by adding XPU to the skip-list.
- long-term, implement tl.atomic_add based sampled_addmm kernel compatible with triton-xpu.
