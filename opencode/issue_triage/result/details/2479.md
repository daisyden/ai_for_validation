## Root Cause

torch.rand's XPU execution policy is computed from hardware topology in calc_execution_policy() at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/DistributionTemplates.h:42-55 using
syclMaxWorkItemsPerSubSlice() and syclMaxWorkItemsPerTile(). Because PVC and BMG have different
sub-slice/tile counts, num_groups and group_size differ, so the per-element Philox counter index
(computed via idx = global_linear_id and stride group_size*num_groups*UNROLL in
DistributionElementwiseKernelFunctor at line 64-96) maps different random numbers to the same output
element. Even with the same seed, element [2351][403] receives a different Philox sequence position
across devices, producing 0.1885 on PVC vs 1.5199e-05 on BMG.

## Fix Approach

- Make the Philox counter->element mapping independent of device topology: fix the effective grid (e.g. compute num_groups from numel with a fixed UNROLL and a device-independent virtual group_size, as CUDA does in DistributionTemplates.cuh), and always iterate the full numel range with a fixed stride derived from (numel, UNROLL) rather than from hardware-derived num_groups*group_size.
- Alternatively, index the Philox state by the linearized output offset instead of the thread id so outputs are reproducible across devices.
