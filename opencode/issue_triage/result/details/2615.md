## Root Cause

FFT ops fail with `RuntimeError: Unsupported dtype Half` / `torch.float16`. In
third_party/torch-xpu-ops/src/ATen/native/xpu/mkl/SpectralOps.cpp:163-176 the MKL DFT dispatcher
only supports SINGLE/DOUBLE precision and calls `AT_ERROR("MKL FFT doesn't support tensor of type")`
for Half/ComplexHalf. Lines 324-327/381-382/461-462/550-551 attempt a Half→Float promotion wrapper,
but for the remaining failing variants (hfft2/ihfft2/ihfftn and their decomp/python-ref tests) the
promotion path is bypassed (e.g. in `test_decomp` comprehensive path and in certain `_refs` fallback
paths where the ref invokes the aten op directly on a half tensor). oneMKL DFT on XPU has no native
fp16 kernel.

## Fix Approach

- Extend the Half/ComplexHalf promotion wrapper in mkl/SpectralOps.cpp to cover all public fft entry points (_fft_c2c, _fft_r2c, _fft_c2r), including the hfft2/ihfft2/ihfftn variants that currently fall through to the error path.
- Add a common helper that casts input Half→Float / ComplexHalf→ComplexFloat before calling `_exec_fft`, runs DFT, and casts the output back.
- Also register float16 support in op_db through an XPU override so decomp/ref tests exercise the promotion path.
- Alternative short term: mark the remaining failing tests as skipped with `@skipIfNoFP16FFT` referencing oneMKL limitation.
