## Root Cause

test_einsum_random_xpu_float64 fails with `RuntimeError: could not construct a memory descriptor
using strides` in the oneDNN matmul path invoked by einsum's bmm decomposition. Einsum reduces to a
bmm whose inputs have zero-sized dims / non-standard strides; the XPU matmul
(third_party/torch-xpu-ops/src/ATen/native/xpu/Blas.cpp / sycl/BlasKernels.cpp, via
at::native::onednn::matmul) forwards the tensor strides to dnnl::memory::desc without normalizing
empty/degenerate strides, so dnnl rejects the descriptor. Fix is tracked/submitted upstream in
pytorch/pytorch#176875.

## Fix Approach

- Land pytorch/pytorch#176875 (adjusts the stride handling before constructing the dnnl::memory::desc — e.g., falls back to a contiguous descriptor when shape has zero-sized / broadcasted dims) and then remove test_einsum_random_xpu_float64 from torch-xpu-ops test/xpu/skip_list_common.py.
- Additionally, guard the onednn matmul entry point to validate strides and fall back to a generic SYCL bmm when oneDNN cannot build a valid memory descriptor.
- Verify with the provided pytest command for float64 and float32.
