## Root Cause

Release/2.11 shows up to -41.8% throughput regression vs 2.10 on Windows+Arc B580 for 5 HuggingFace
models (DistilBertForQuestionAnswering fp16/fp32, Llama-3.2-1B bf16/fp16, XLNetLMHeadModel
amp-fp16). Reporter explicitly ruled out oneAPI upgrade by rebuilding 2.11 against old oneAPI and
still reproducing the slowdown, so the regression is inside pytorch/torch-xpu-ops code landed
between 2.10 and 2.11 (likely SDPA / layernorm / GEMM wiring changes in the Windows build, e.g.
third_party/torch-xpu-ops/src/ATen/native/xpu/sdp_utils.cpp and the oneDNN matmul primitive cache
paths) rather than a driver/library issue. No stack trace — this is a perf bisect bug, not a
functional one.

## Fix Approach

- Bisect torch and torch-xpu-ops commits between release/2.10 and release/2.11 on Windows using the DistilBertForQuestionAnswering fp16 case (largest delta).
- Profile eager runs with onetrace/unitrace to see whether the extra time is in SDPA kernels, layernorm, matmul primitive creation, or H2D/D2H sync.
- Likely suspects are recent SDPA dispatch changes (sdp_utils.cpp), oneDNN primitive cache invalidation on Windows, or a new blocking copy.
- once the offending commit is identified, restore the fast path or re-enable the previous kernel selection heuristic for Arc B-series.
