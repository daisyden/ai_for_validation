## Root Cause

Under `torch.compile`, DTensor random ops dispatch through OffsetBasedRNGTracker._get_device_state
at torch/distributed/tensor/_random.py:227: `rng_state =
self._device_handle.get_rng_state().to(self._device)`. `get_rng_state()` returns a real CPU uint8
tensor (size 16); when Dynamo traces the compiled region it sees this non-fake tensor flow into
`aten.to.dtype_layout`, tripping FakeTensorMode's `Please convert all Tensors to FakeTensors first`
assertion (_dynamo/utils.py get_fake_value). The tracker is (re)instantiated during dispatch in
torch/distributed/tensor/_dispatch.py:339, so the RNG read happens inside the traced frame. This
affects every DistTensorRandomOpCompileTest test (uniform_, bernoulli, rand_like, etc.) on XPU
because the XPU device handle's get_rng_state returns a CPU tensor that is not registered as a graph
input.

## Fix Approach

- Avoid eagerly reading device RNG state inside traced regions: either (a) move `_compute_rng_offsets` / `_get_device_state` out of the traced dispatch path (compute the offset before torch.compile sees the op and pass via closure/torch.library custom op), or (b) wrap these tensor creations with `torch._dynamo.disable` / `allow_in_graph` + `torch._C._functorch.current_level()` guards so Dynamo treats them as constants, or (c) register rng_state as a side-effectful graph input via `torch._dynamo.graph_break` just around the tracker update.
- Short-term: patch OffsetBasedRNGTracker so that when `torch.compiler.is_compiling()` it uses a cached device-resident state tensor instead of round-tripping through CPU `.to()`.
- Mirrors work already done on CUDA for philox offset tracking.
