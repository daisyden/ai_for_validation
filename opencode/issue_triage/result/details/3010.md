## Root Cause

DTensor random ops (uniform_, bernoulli, normal_, native_dropout, rand_like, etc.) graph-break under
torch.compile with FakeTensorMode. The OffsetBasedRNGTracker path in
torch/distributed/tensor/_random.py invokes aten.to.dtype_layout on a non-fake CPU uint8 RNG state
tensor inside the dispatcher, triggering the FakeTensor assertion 'Please convert all Tensors to
FakeTensors first or instantiate FakeTensorMode with allow_non_fake_inputs'. The bug is in upstream
PyTorch's DTensor RNG tracker not wrapping the RNG state tensor as fake during Dynamo tracing.

## Fix Approach

- Upstream fix in torch/distributed/tensor/_random.py: mark RNG state tensor as fake under FakeTensorMode (use mode.from_tensor) or set allow_non_fake_inputs=True for the tracker's internal aten.to.dtype_layout call.
- Alternatively, register the RNG state operations as torch.library custom ops with proper fake kernels.
- Issue tracks fix landing in PyTorch upstream.
- mark these tests xfail/skip in torch-xpu-ops distributed skip list until the upstream fix lands.
