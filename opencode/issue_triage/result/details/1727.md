## Root Cause

FSDP test
`test/distributed/_composable/fsdp/test_fully_shard_training.py::test_non_root_forward_backward`
calls `torch.get_device_module(device_type)._sleep(...)` (line 521) to stall the device stream for
timing-sensitive validation, but the XPU runtime module does not expose `_sleep` (unlike
`torch.cuda._sleep`). Per maintainer comment, implementing this requires a SYCL-level cycle-delay
primitive available only from oneAPI 2026 (needs a new SYCL/level-zero API for deterministic cycle
wait).

## Fix Approach

- Once oneAPI 2026 is adopted, add a `_sleep(cycles)` binding in `torch/csrc/xpu/Module.cpp` backed by a SYCL kernel that busy-waits the requested cycle count (mirroring `at::cuda::sleep`), and register it on `torch.xpu`.
- Until then, mark the affected FSDP/distributed tests with `@skipIfXpu`/`unittest.skipIf(not hasattr(torch.xpu, '_sleep'))` so they stop blocking the XPU distributed CI.
