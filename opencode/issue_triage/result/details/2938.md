## Root Cause

basic_gnn_gin / basic_gnn_sage torchbench fp32 inference regressed ~25% on PVC 1100 between torch
2.10 and 2.11; also regresses on CUDA 4080s, so the root cause is device-agnostic. jianyizh bisected
to PR #2655 and filed pytorch/pytorch#177117: the guilty change regenerated a pointwise kernel
(signature `in_ptr0:*i64, in_ptr1:*fp32, ...`) with suboptimal YBLOCK/XBLOCK tiling for the
gather/scatter-style kernel used by GNN message-passing, increasing launch count and memory traffic
for small batch (bs=1).

## Fix Approach

- This is an upstream Inductor issue.
- track via pytorch#177117.
- Workaround in torch-xpu-ops CI: skip/baseline these two models until upstream lands a fix to the pointwise autotune configs for (large-y, small-x) shapes, or add XPU-side extra configs in torch/_inductor/runtime/triton_heuristics.pointwise so the tuner recovers the 2.10 config.
- No SYCL kernel change needed.
