## Root Cause

test_pool3d_large_size_int64_xpu (70x32x100x100x100 fp16 input, kernel 5) shows 0.2% mismatched
elements vs CPU fp32 reference with greatest absolute difference 3.75. Since max_pool3d is a pure
selection op, the divergence implies the XPU kernel selects different source elements — almost
certainly an int32 indexing overflow inside
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/DilatedMaxPool3d.cpp at large total element
counts (>2^31), which is exactly the regression scenario pytorch/pytorch#52822 was written to guard.

## Fix Approach

- Audit DilatedMaxPool3d.cpp for any int/int32 index, stride or offset arithmetic and convert to int64 (or use canUse32BitIndexMath() guard with an int64 fallback path), mirroring the CUDA implementation in aten/src/ATen/native/cuda/DilatedMaxPool3d.cu.
- Add a smaller regression test and rerun test_pool3d_large_size_int64_xpu.
- Also verify the backward kernel for the same index-type bug since the test calls y.backward(g).
