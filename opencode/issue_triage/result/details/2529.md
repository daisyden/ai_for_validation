## action_reason

- Method in test/test_nn.py:12508 is named test_ctc_loss_cudnn_tensor_cuda (hardcoded _cuda suffix) and carries @onlyCUDA; test_nn_xpu.py does NOT use XPUPatchForImport, so @onlyCUDA blocks XPU. Expected name test_ctc_loss_cudnn_tensor_xpu does not exist.

## Root Cause

Remaining failures are three upstream tests assuming CUDA/cuDNN-specific behaviour:
test_ctc_loss_cudnn and test_ctc_loss_cudnn_tensor (test/xpu/test_nn_xpu.py:15463,15509) hit
cuDNN-only paths, test_disable_fastpath (test/xpu/test_transformers_xpu.py:1896 calling
test_transformers.py:1479) asserts that MHA fast path is invoked but XPU doesn't register a fastpath
kernel so `fastpath_mock.called` is False, and test_transformerencoderlayer_xpu_float64 expects
float64 parity when XPU MHA/SDPA silently falls back to float32 or skips the fused kernel. The
previously listed ~40 test_to_nn_* module swap tests and linalg addmm float64 cases now pass in
nightly and are struck out.

## Fix Approach

- Skip the three remaining cases via test/xpu/skip_list_common.py (or instantiate_parametrized_tests decorator) as CUDA/cuDNN specific: test_ctc_loss_cudnn[_tensor] has no XPU equivalent and should be `@skipXPU` or `@onlyCUDA`.
- For test_disable_fastpath, either register an XPU MHA fastpath (nested tensor path in aten/src/ATen/native/transformers/) so the mock is hit, or skip on XPU.
- For test_transformerencoderlayer_xpu_float64, add float64 dtype support to the XPU fused transformer encoder or mark dtype float64 skipped on XPU.
