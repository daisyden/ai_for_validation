## Root Cause

test_embedding_bag_device_xpu_int*_int*_float64 and several test_reductions/test_argminmax_multiple
cases produce large numeric diffs (~1.9 abs, 19x rel) only in stock PyTorch CI, not on developer
PVC. Reproduction in the CI docker image ghcr.io/pytorch/ci-image:pytorch-linux-noble-xpu-n-py3
shows wincent8 isolated it to the IGC/SYCL compiler version used to build torch in-CI (lts2) vs. the
newer IGC used for nightly wheels, which pass. The SYCL kernel at
third_party/torch-xpu-ops/src/ATen/native/xpu/sycl/EmbeddingBag.cpp (and reduction kernels for
argmax/argmin/masked_*, e.g. ReduceOpsKernels.cpp) is generating incorrect FP64 code under the older
IGC, yielding accumulator/ordering issues.

## Fix Approach

- Bump the PyTorch CI xpu docker image to use the same IGC/oneAPI version as nightly wheel builds (track via .ci/docker/common/install_xpu.sh and ghcr.io/pytorch/ci-image tag) and re-run.
- in parallel, inspect EmbeddingBag.cpp float64 accumulation path to add explicit fp64 accumulation and avoid relying on IGC fast-math ordering.
- After image bump, remove WAs in pytorch/pytorch PRs #165886 and #169168.
- File an IGC bug with minimal SYCL reproducer if the numerical issue persists with updated compiler.
