## Root Cause

test/distributed/test_device_mesh.py:903 (test_flatten_mesh_3d) asserts
flattened_dp_cp_mesh.get_group().group_desc == 'mesh_dp_cp', but on XPU it is 'default_pg'.
DeviceMesh._flatten creates the flattened sub-PG via ProcessGroup::splitGroup(desc=...)
(torch/distributed/device_mesh.py); NCCL implements splitGroup and propagates the group desc, but
ProcessGroupXCCL (third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp/.hpp) does not override
splitGroup, so the base-class fallback creates a brand-new default PG named 'default_pg' with no
desc propagation.

## Fix Approach

- Implement ProcessGroupXCCL::splitGroup() mirroring ProcessGroupNCCL's splitGroup (and also implement mergeRemoteGroup if needed) so it honors the provided group desc/name and records it on the resulting PG.
- Until that lands, DeviceMesh._flatten for XPU can explicitly call pg._set_group_desc(desc) after creation.
- Add regression coverage via the existing test_flatten_mesh_3d in the XPU distributed UT sweep.
