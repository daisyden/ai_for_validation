## Root Cause

test_flatten_mesh_3d asserts flattened_dp_cp_mesh.get_group().group_desc == 'mesh_dp_cp' but the
XCCL ProcessGroup returns 'default_pg' (test/distributed/test_device_mesh.py:903).
DeviceMesh._flatten() creates a sub-group whose group_desc must be propagated through the backend's
PG construction; ProcessGroupXCCL does not honor the group_desc passed via pg_options/group_name, so
it falls back to the default descriptor, unlike NCCL which threads it through to Work/PG metadata.

## Fix Approach

- In ProcessGroupXCCL (third_party/torch-xpu-ops/src/xccl/ProcessGroupXCCL.cpp), accept and store the group_desc from the Backend::Options/c10d::PGOptions and expose it via getGroupDesc() so DeviceMesh._flatten propagates 'mesh_dp_cp'.
- Mirror the NCCL implementation in torch/csrc/distributed/c10d/ProcessGroupNCCL.cpp.
