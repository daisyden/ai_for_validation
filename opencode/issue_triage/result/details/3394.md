## Root Cause

oneDNN-backed SDPA on XPU segfaults at the GPU/driver layer ('Segmentation fault from GPU at
0xff00000000094000 ... drm_neo.cpp:288') for shape [B=4, S=16413, H=3, D=16] fp32 (and similar
S>16K). The crash reproduces with plain F.scaled_dot_product_attention, no mask, indicating the
oneDNN micro-kernel SDPA primitive mis-handles sequence lengths exceeding an internal tile/block
boundary around 16K, walking off allocated GPU memory. Owner LuFinch reopened JIRA MFDNN-14479
against oneDNN.

## Fix Approach

- Fix in oneDNN SDPA primitive (MFDNN-14479) — bounds/tiling for seq_len>16K fp32.
- As a torch-xpu-ops mitigation until the oneDNN fix lands, add a guard in the XPU SDPA dispatcher (Attention.cpp / sdp_utils) to fall back to the math backend when seq_len exceeds a safe threshold (e.g. 16384) for the oneDNN path.
- add a regression test with the issue's shape.
