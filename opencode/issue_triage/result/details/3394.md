## Root Cause

SDPA on XPU with seq_len=16413 (and seq_len=16611) crashes with a GPU page fault `Segmentation fault
from GPU at 0xff00000000094000 ... drm_neo.cpp:288` originating from the oneDNN-backed SDPA
primitive (label 'dependency component: oneDNN' confirms this). The fault is at PDE level 1 with
NotPresent/Write, indicating the oneDNN-generated kernel writes outside its allocated workspace when
the sequence length exceeds an internal tiling/threshold. Reopened as JIRA MFDNN-14479 by LuFinch.
The torch-xpu-ops SDPA wrapper at src/ATen/native/transformers does not bound seq_len before
dispatching to oneDNN.

## Fix Approach

- Short-term: in the XPU SDPA dispatcher (third_party/torch-xpu-ops/src/ATen/native/transformers), add a seq_len threshold guard that falls back to the math backend when seq_len exceeds the safe oneDNN limit (e.g. >16K), similar to prior workarounds.
- Long-term: track MFDNN-14479 in oneDNN to fix the workspace allocation/tiling overflow.
- Add a regression test with the reported shapes to torch-xpu-ops/test/xpu/test_transformers_xpu.py once the oneDNN fix lands.
